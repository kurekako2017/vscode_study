## S/A/B/C 四级目录管理机制

目录等级用于决定 AI Agent 的操作权限，与表示项目重要程度的“重要等级”不同。所有 AI Agent 在读取、修改、提交或发布前，必须先确认目标目录等级。

| 目录等级 | 定义 | AI 操作规则 |
| --- | --- | --- |
| S | 最高保护目录 | 默认只读；任何修改必须先询问用户；禁止自动提交 GitHub、批量修改、删除和重命名 |
| A | 核心维护目录 | 允许按任务新增文档、更新教程和补充知识库；删除、重命名、大范围重构前必须询问用户 |
| B | 常规项目目录 | 允许完成用户要求范围内的新增和修改；删除、批量改动或结构调整前必须询问用户 |
| C | 实验与练习目录 | 允许进行可恢复的小范围实验；仍须遵守敏感信息检查和 GitHub 用户确认规则 |

目录等级采用父目录保护继承原则：子目录未单独登记时，继承最近父目录的等级；父目录为 S 级时，其全部子目录均按 S 级处理，不得用更宽松的子目录规则绕过。规则冲突时，执行保护更严格的规则。

### `softbs` S 级特别保护规则

`softbs` 是个人知识库与业务资料库，可能包含税务资料、freee 资料、投资资料、API 配置、客户资料、网站维护资料和私人笔记。

- 禁止自动提交 GitHub。
- 禁止批量修改。
- 禁止删除。
- 禁止重命名。
- 任何修改必须先询问用户。

### `web-projects` A 级维护规则

`web-projects` 是 Web 开发知识库。

- 允许新增文档。
- 允许更新教程。
- 允许补充知识库。
- 涉及删除、重命名和重构时，必须先询问用户。

## 维护原则

- “GitHub同步”为“否”的目录默认禁止加入提交。
- “AI允许修改”为“否”的目录只允许读取，不允许 AI 自动编辑。
- 标记为“视上游规则”或“仅经确认”时，必须先阅读该项目自己的 `AGENTS.md`、贡献规范和 `.gitignore`。
- 所有操作先检查目录等级；S 级保护优先于“GitHub同步”和“AI允许修改”等其他字段。
- 同一工作区可能包含多个独立 Git 仓库；执行 Git 命令前必须确认当前仓库根目录。
- 重要等级越高，修改、发布和同步前所需检查越严格。

## 新项目登记规范

新增项目时必须登记：

- 项目名称
- 用途
- 是否同步 GitHub
- 是否允许 AI 修改
- 目录等级
- 重要等级
- 备注

登记时使用真实相对路径。信息不确定时，将“GitHub同步”填写为“否”，将“AI允许修改”填写为“否”，确认后再修改。
