# 评估与运维

## 1. 为什么这一块不能省

很多生成 AI 样例“能跑”，但不代表“能上线”。

现场真正关心的是：

- 命中率如何
- 回答稳不稳
- 成本高不高
- 延迟能不能接受
- 出错时怎么处理

也就是说，前面几章解决的是：

- 能不能做出来

而这一章解决的是：

- 做出来的东西到底靠不靠谱

很多 `PoC` 不是死在“做不出来”，而是死在：

- 没有评估
- 没有失败记录
- 没有改进依据

## 2. 这一阶段要掌握什么

这一章至少要掌握这些：

- 测试问题集
- 命中质量观察
- 失败案例记录
- 成本意识
- 运行日志

学完以后，你应该能做到：

- 不只是展示 demo
- 还能说明 demo 的质量边界和改进方向

## 3. 这一章学完后，应该会什么

学完这一章，至少要能做到：

- 知道为什么“能跑”不等于“可交付”
- 能设计一份最小测试问题集
- 能记录命中与误答情况
- 能观察来源是否正确
- 能说明成本、延迟和运维风险
- 能写一份最小评估记录

## 4. 教程：评估到底在评什么

初学者最容易把评估理解成：

- 看模型答得像不像

但在实际项目里，评估通常更具体。

例如对一个 `RAG` 项目，常见关注点是：

- 有没有命中相关资料
- 有没有命中正确片段
- 回答是否基于资料
- 来源是否正确
- 无命中时是否明确说明

所以评估不是一个抽象概念，而是：

- 把“好不好用”拆成一组能观察的点

## 5. 教程：最小评估流程长什么样

最小评估，一般可以拆成 5 步：

1. 准备测试问题
2. 明确期望资料
3. 执行问答
4. 记录结果
5. 总结问题和改进点

### 步骤 1：准备测试问题

先准备 10 到 20 个问题。

这些问题不要只挑容易的。

最好包含：

- 正常命中问题
- 模糊问题
- 无命中问题
- 容易混淆的问题

### 步骤 2：明确期望资料

在问之前，先写下：

- 这个问题理论上应该命中哪些文档
- 哪些片段最可能相关

这样后面才有“对照标准”。

### 步骤 3：执行问答

可以用：

- `doc_qa_agent`
- `rag_api_demo`

逐条执行问题并记录输出。

### 步骤 4：记录结果

至少记录这些：

- 是否命中
- 是否回答正确
- 是否带来源
- 来源是否合理
- 是否出现幻觉或过度发挥

### 步骤 5：总结问题和改进点

最后不要只停在“通过 / 不通过”。

还要总结：

- 为什么没命中
- 为什么来源不对
- 为什么回答过度发挥
- 下一步改哪里

## 6. 最小示例

这一章的最小示例不是代码，而是一张最小评估表。

因为评估阶段的关键不是先写新功能，而是先建立观察框架。

```text
| 问题 | 期望资料 | 是否命中 | 是否回答正确 | 是否带来源 | 备注 |
|---|---|---|---|---|---|
| 对日项目里的 RAG 和 Agent 哪个优先？ | llm-lab / agent-lab 文档 | 是 | 是 | 是 | 回答基本正确 |
| 数据移行资料主要看什么？ | java-lab 数据库文档 | 否 | 否 | 否 | 检索词不稳定 |
| 一个完全无关的问题 | 无 | 否 | 是 | 否 | 模型过度发挥，需要更强约束 |
```

### 这个最小示例学的是什么

你要从这张表里看懂 5 个点：

1. 评估要有问题集
2. 评估要有期望答案或期望资料
3. 要区分“命中”和“回答正确”不是同一件事
4. 要单独看来源是否合理
5. 评估的目的不是打分好看，而是找到改进方向

## 8. 代码拆解：评估里最容易忽略的点

### 1. “命中”不等于“回答正确”

有时候系统确实检索到了相关资料，但回答还是可能：

- 理解错重点
- 组织得不好
- 混进了不该有的推断

所以评估时至少要分开看：

- 检索质量
- 回答质量

### 2. “回答正确”不等于“来源正确”

模型有时会答对，但来源标签不准确。

在企业场景里，这仍然是问题。

因为现场经常需要：

- 回到原文核对
- 给业务方看依据

### 3. “看起来稳定”不等于“能运维”

如果一个系统：

- 日志没有
- 错误信息不清楚
- 成本不可控
- 无法重复测试

那它即使功能能跑，也很难往前推进。

### 4. 无命中场景特别重要

很多学习者只测“有答案的问题”。

但现场里经常更在意：

- 没资料时会不会乱答

所以无命中测试是必须项，而不是边角项。

## 12. 常见错误与排查

### 1. 只看最终回答，不看检索过程

表现通常是：

- 回答像是对的
- 但不知道为什么对
- 也不知道为什么错

这时候要先补的是：

- 看来源
- 看命中的片段
- 看无命中情况

### 2. 只测容易的问题

表现通常是：

- 结果看起来很好
- 但一换问法就失真

所以问题集一定要覆盖：

- 正常问题
- 模糊问题
- 反问法
- 无关问题

### 3. 不记录失败案例

表现通常是：

- 每次测完只知道“大概还行”
- 但说不出哪里该改

所以至少要保留：

- 误答样例
- 漏答样例
- 无命中样例

### 4. 不关心成本和延迟

表现通常是：

- 功能有了
- 但不清楚一天跑多少次会怎样

当前阶段不需要做复杂计费系统，但至少要建立意识：

- 每次模型调用有成本
- 每次测试也会消耗成本
- 文档越大、片段越多，延迟通常越高

## 13. 练习题

下面这些练习题，建议按顺序做。

### 练习 1：做一份 10 题问题集

围绕你自己的文档目录，整理 10 个问题。

目标：

- 从“随便问”进步到“有计划地测”

### 练习 2：给每个问题写期望资料

在提问前先写：

- 理论上应该命中哪些文档

目标：

- 让评估有对照标准

### 练习 3：记录无命中问题

故意设计 2 到 3 个文档里不存在答案的问题。

目标：

- 训练系统在资料不足时不要乱答

### 练习 4：比较 CLI 和 API 两个版本

用同一批问题分别测：

- `doc_qa_agent`
- `rag_api_demo`

目标：

- 观察从本地脚本到服务化后，有哪些行为差异

### 练习 5：整理一页失败案例

把最典型的 3 个失败样例整理出来，并写明：

- 失败表现
- 可能原因
- 下一步改法

目标：

- 训练改进意识，而不只是记录结果

### 练习 6：做一页最小试验结果报告

内容至少包括：

- 测试对象
- 问题数量
- 命中情况
- 失败情况
- 改进建议

目标：

- 让你的作品集更接近真实 PoC 输出物
