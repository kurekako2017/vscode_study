# 日本现场应用与案件关键词

## 1. 更常见的应用形态

当前日本现场更常见的生成 AI 应用，通常是：

- 社内问答
- 社内搜索
- `RAG` 知识库问答
- 摘要
- 翻译
- 邮件草拟
- 特定流程助手

而不是一开始就做强自治的复杂 Agent。

从很多日本现场相关分享里，也能提炼出一个更现实的判断：

- `AI Agent` 话题热度在上升
- 但很多场景仍停留在 `PoC`、试点或局部试运行
- 先稳定落地的，通常还是 `RAG`、分类、摘要、流程辅助

所以学习时不要只看到“Agent 很火”这一个表面结论。

更稳妥的理解是：

- 日本现场会谈 `AI Agent`
- 但更常见的交付物仍然是可控、可解释、可接系统的应用

## 1.1 现场里说的 Agent，通常是什么意思

现场里说的 `AI Agent`，通常不只是：

- 聊天
- 回答问题

而是更接近这些能力：

- 感知输入或环境变化
- 基于规则或模型做判断
- 调用工具或系统执行动作
- 按目标推进任务

也就是说，`Agent` 和普通聊天模型的主要差别，往往在于：

- 是否能执行
- 是否能调用外部工具
- 是否能根据状态继续推进流程

这也是为什么学习顺序上不建议一开始就跳到复杂 Agent。

因为如果没有：

- 结构化输出
- API 调用
- 文件处理
- `RAG`

这些基础，`Agent` 很容易只剩概念。

## 3. 对应学习优先级

如果按日本案件来学，建议优先顺序是：

1. 模型调用
2. 结构化输出
3. `RAG`
4. `FastAPI`
5. PDF / 文档处理
6. 评估
7. Agent

如果把 `AI Agent` 的现场热度也考虑进去，更现实的顺序是：

1. 模型调用
2. 结构化输出
3. `RAG`
4. `FastAPI`
5. 流程辅助型应用
6. Tool Calling
7. Agent

也就是说，`Agent` 要学，但更适合作为后段进阶。

## 4. 作品集建议

如果你要做作品集，最值得优先准备的是：

- 一个最小 `RAG CLI`
- 一个 `RAG API`
- 一份简化版需求概要
- 一份简化版基本设计
- 一份测试观点或测试用例表

如果后面要补一项 `Agent` 相关作品，建议先做：

- 一个可控的工具调用样例

而不是一开始就做：

- 强自治
- 多 Agent 协作
- 很长链路的自动执行

因为前者更容易讲清楚：

- 输入是什么
- 判断逻辑是什么
- 调了哪些工具
- 输出给谁用

## 4.1 更贴近日本现场的 Agent 作品方向

如果要让 `Agent` 方向也更贴近日企或派遣案件，优先可以考虑：

- 查询社内资料后整理答复
- 从需求文本中提取任务并生成清单
- 根据固定规则调用内部 API
- 做辅助执行的流程助手，而不是完全自治系统

## 5. 和 Agent 的关系

不是不学 Agent，而是：

- 先学 `LLM 应用开发`
- 再学 `Agent 开发`

这更贴近日本现场的实际需求。

一句话概括就是：

- 先把“会调模型、会做结构化输出、会做 `RAG`、会做 API”打牢
- 再把这些能力接成 Agent
