# 云平台与企业环境

## 1. 为什么这一块很重要

如果你的目标是贴近日本 IT 现场和派遣案件，那么只会本地跑 `LLM` 样例还不够。

很多案件真正要求的是：

- 在企业环境里接入模型
- 通过云平台提供能力
- 和内部系统联动
- 处理权限、网络、成本、运维

所以 `云平台与企业环境` 是 `LLM 应用开发` 里很实用的一块。

## 2. 日本案件里常见的平台

### Azure OpenAI

这是日本企业案件里非常常见的一条线。

常见原因：

- 企业更容易走 Microsoft 体系
- 和既有 Azure 环境整合方便
- 权限、网络、监控更容易纳入企业管理

常见关键词：

- `Azure OpenAI`
- `Azure AI Search`
- `Azure Functions`
- `App Service`
- `VNet`

### AWS Bedrock

在 AWS 环境占比较高的公司里，`Bedrock` 也很常见。

常见原因：

- 企业已有 AWS 环境
- 方便和 `S3`、`Lambda`、`ECS`、`RDS` 等服务联动
- 适合和 `RAG`、文档处理、内部 API 集成

常见关键词：

- `AWS Bedrock`
- `Lambda`
- `ECS`
- `S3`
- `API Gateway`

## 3. 企业环境里常见的限制

和个人学习环境相比，企业环境经常会多出这些现实条件：

- 网络出口受限
- 需要代理
- API Key 不能随便放在代码里
- 访问资料目录有权限限制
- 文档和数据来源要可追踪
- 必须控制成本

所以很多时候，难点不只是“能不能调用模型”，而是：

- 能不能在企业网络里跑
- 能不能合规接入
- 能不能上线维护

## 4. 典型集成形态

### 形态 1: RAG API

最常见的形态之一：

- 文档放在文件系统或对象存储
- 后端提供 `RAG API`
- 前端、内部工具或管理后台调用

### 形态 2: 社内検索助手

常见做法：

- 资料来自规程、设计书、FAQ、手顺书
- 搜索结果带出处
- 人工再判断是否采用回答

### 形态 3: 业务系统辅助功能

常见做法：

- 在现有系统里加入摘要、分类、问答、草稿功能
- 不一定做成独立聊天系统

## 5. 云平台相关学习优先级

如果按日本现场案件要求来学，建议优先顺序是：

1. 先理解本地版 `RAG`
2. 再做 `FastAPI`
3. 再理解 `Azure OpenAI` 或 `AWS Bedrock`
4. 再理解文档存储、权限、认证、网络

不要一上来就陷进云平台细节，而忽略主应用链路。

## 8. 成本与注意点

- 云平台接入通常比本地 PoC 更复杂。
- 企业环境里经常需要考虑：
  - 认证
  - 权限
  - 审计
  - 网络
  - 成本
- 如果只会本地 demo，还不足以完全对应企业项目。
- 但如果本地版 `RAG -> FastAPI -> 云平台` 这条链路能讲清楚，已经很贴日本现场。
