# 云平台与企业环境

## 1. 为什么这一块很重要

如果你的目标是贴近日本 IT 现场和派遣案件，那么只会本地跑 `LLM` 样例还不够。

很多案件真正要求的是：

- 在企业环境里接入模型
- 通过云平台提供能力
- 和内部系统联动
- 处理权限、网络、成本、运维

所以 `云平台与企业环境` 是 `LLM 应用开发` 里很实用的一块。

前面几章解决的是：

- 本地怎么把链路跑通

这一章解决的是：

- 如果放到企业环境，会多出哪些现实条件

也就是说，这一章不是要你立刻变成云平台专家，而是要先建立：

- 企业环境意识

## 4. 教程：为什么企业环境和本地环境差很多

在本地学习时，你经常可以这样做：

- 环境变量直接写当前终端
- 文档直接放本地目录
- 只要能访问外网就能调模型
- 自己能看懂输出就算跑通

但企业环境通常不是这样。

企业环境里经常会多出这些现实条件：

- 网络出口受限
- 需要代理或内网接入方式
- API Key 不能直接放在代码或个人机器里
- 访问资料目录要有权限控制
- 日志和审计要能追踪
- 成本不能无限增长

所以很多时候，真正的难点不只是：

- 会不会调模型

而是：

- 能不能在企业网络里稳定跑
- 能不能合规接入
- 能不能让别人也安全地调用

## 5. 教程：日本现场里常见的平台方向

这一步不用一开始学得很深，但要知道常见方向。

### 1. Azure OpenAI

这是日本企业案件里非常常见的一条线。

常见原因：

- 企业原本就在 Microsoft 体系内
- 和既有 Azure 环境整合方便
- 权限、网络、监控更容易纳入企业管理

常见关联关键词：

- `Azure OpenAI`
- `Azure AI Search`
- `App Service`
- `Azure Functions`
- `VNet`

### 2. AWS Bedrock

在 AWS 环境占比较高的公司里，`Bedrock` 也很常见。

常见原因：

- 企业已有 AWS 环境
- 更容易和 `S3`、`Lambda`、`ECS`、`RDS` 等服务联动
- 适合和 `RAG`、文档处理、内部 API 集成配合

常见关联关键词：

- `AWS Bedrock`
- `S3`
- `Lambda`
- `ECS`
- `API Gateway`

### 3. 当前阶段应该怎么理解

对当前学习阶段来说，最重要的不是马上精通某一朵云，而是先理解：

- 云平台是企业接入模型、权限、网络、监控的一层基础设施

也就是说，当前优先级仍然是：

- 先把应用链路做明白
- 再理解它放到云上会怎么变化

## 6. 教程：企业版链路和本地版链路的区别

可以先把本地版链路理解成：

1. 本地文档目录
2. 本地 `RAG`
3. 本地 `FastAPI`
4. 本地终端或本地前端调用

而企业版链路更像：

1. 文档在共享目录、对象存储或企业资料库
2. 后端服务部署在企业云环境
3. 模型通过企业允许的方式接入
4. 前端或内部系统通过认证后调用
5. 日志、权限、成本都要纳入管理

这就是为什么企业版难点通常会出现在：

- 文档来源
- 身份认证
- 权限边界
- 网络连通
- 监控与成本

## 7. 最小示例

这一章的最小示例不是代码，而是一张“本地版和企业版对照表”。

因为这一章最重要的是建立架构理解。

```text
本地版:
- docs_dir = 本地目录
- API Key = 本机环境变量
- 调用方 = 自己
- 日志 = 终端输出

企业版:
- docs_dir = 对象存储 / 共享资料库 / 企业文档目录
- API Key = 受控配置或企业密钥管理
- 调用方 = 前端 / 内部系统 / 管理后台
- 日志 = 可追踪、可审计的服务日志
```

### 这个最小示例学的是什么

你要从这张对照里看懂 5 个点：

1. 企业环境不是把本地脚本直接搬上去就结束
2. 文档来源会变化
3. 密钥管理方式会变化
4. 调用方会变化
5. 日志、权限、成本都会进入主问题

## 9. 企业环境里最常见的限制

这一块很关键，因为很多学习者不会在本地遇到这些问题。

### 1. 网络限制

常见情况：

- 不能直接访问外网
- 需要代理
- 需要走企业允许的出口

### 2. 密钥和配置限制

常见情况：

- API Key 不能放源码里
- 不允许开发者随意持有高权限凭证
- 配置要按环境区分

### 3. 文档和数据权限

常见情况：

- 不是所有文档都能给所有人检索
- 文档来源要可追踪
- 某些资料只能部分开放

### 4. 成本限制

常见情况：

- 调用次数要受控
- 模型选择要看预算
- 文档处理和检索方式也会影响成本

### 5. 运维限制

常见情况：

- 服务要可监控
- 出错要能追踪
- 更新后要能回滚或重载

## 10. 学习优先级怎么排

如果按日本现场案件要求来学，建议优先顺序是：

1. 先理解本地版 `RAG`
2. 再做 `FastAPI`
3. 再理解 `Azure OpenAI` 或 `AWS Bedrock`
4. 再理解文档存储、权限、认证、网络
5. 再补部署、日志、监控

不要一上来就陷进云平台细节，而忽略主应用链路。

如果压缩成一句话，就是：

- 先把应用做明白，再去学它怎么进入企业环境

## 11. 常见错误与排查

### 1. 一上来就学云厂商名词，反而不会做应用

表现通常是：

- 会说很多服务名
- 但不会把 `RAG` 或 API 链路讲清楚

这时候要先回到主线：

- 模型调用
- 结构化输出
- `RAG`
- `FastAPI`

### 2. 只会本地目录，不会解释企业资料来源

表现通常是：

- demo 能跑
- 但别人一问“企业文档放哪、怎么接”，就讲不清楚

所以这一章要至少建立意识：

- 文档可能来自共享目录、对象存储、企业资料库

### 3. 忽略认证和权限

表现通常是：

- 觉得接口能用就够了
- 但没考虑谁能调用、谁能看哪些文档

这在企业环境里通常不够。

### 4. 不关心成本

表现通常是：

- 只追求效果
- 不知道调用量、文档量、模型选择会怎么影响费用

当前阶段不用做复杂成本模型，但至少要能说出：

- 成本会受模型、调用频率、文档规模影响

## 16. 成本与注意点

- 云平台接入通常比本地 PoC 更复杂。
- 企业环境里经常需要考虑：
  - 认证
  - 权限
  - 审计
  - 网络
  - 成本
- 如果只会本地 demo，还不足以完全对应企业项目。
- 但如果本地版 `RAG -> FastAPI -> 云平台` 这条链路能讲清楚，已经很贴日本现场。
