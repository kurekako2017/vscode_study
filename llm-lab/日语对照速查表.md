# LLM 应用日语对照速查表

> 目标：面向日本项目现场，把 LLM 应用开发里常见的中文概念、英文术语、日本語说法、现场含义放在一起。以后看案件、设计书、会议记录、README 时，可以快速对照。

## 1. 使用方法

建议按三种场景使用：

| 场景 | 怎么看 |
| --- | --- |
| 看日本案件 | 先看“案件关键词”和“岗位表达” |
| 看设计书 | 先看“系统分层”和“RAG / API 流程” |
| 面试准备 | 先看“可以怎么说明给现场听” |

日语表达不一定只有一种。日本现场里经常会混用：
- 英文原词：`RAG`, `Prompt`, `Embedding`
- 片假名：`プロンプト`, `エンベディング`
- 日语说明：`検索拡張生成`, `社内文書検索`

## 2. 核心术语对照

| 中文 | English | 日本語 / 现场说法 | 是什么 | 现场作用 |
| --- | --- | --- | --- | --- |
| 大语言模型 | LLM / Large Language Model | 大規模言語モデル / LLM | 能根据上下文生成文本、摘要、JSON、判断的模型 | 生成回答、整理文档、分类、抽取信息 |
| 生成式 AI | Generative AI | 生成AI | 能生成文本、图像、代码等内容的 AI 技术总称 | 日本案件里常用作项目大类关键词 |
| 提示词 | Prompt | プロンプト | 给模型的任务说明、规则、上下文 | 控制模型按什么角色和格式回答 |
| 系统消息 | System message | システムメッセージ | 比用户问题更高层级的行为约束 | 规定模型身份、禁止事项、输出规则 |
| 用户消息 | User message | ユーザーメッセージ | 用户本次输入的问题或请求 | 表示业务用户真正想解决的问题 |
| 上下文 | Context | コンテキスト | 模型本次请求能看到的信息集合 | 放入用户问题、历史、检索资料、规则 |
| 上下文窗口 | Context window | コンテキストウィンドウ | 模型一次最多能处理的文本范围 | 决定长文、RAG、历史记录能放多少 |
| Token | Token | トークン | 模型计量文本长度和费用的单位 | 影响成本、延迟、上下文长度 |
| 温度参数 | Temperature | temperature / 温度パラメータ | 控制输出随机性的模型参数 | 低温更稳定，高温更发散 |
| 结构化输出 | Structured Output | 構造化出力 | 让模型返回固定格式 JSON 或 schema | 让后端程序稳定解析模型结果 |
| Schema | Schema | スキーマ | 字段、类型、必填规则的定义 | 定义 API 或模型输出的数据结构 |
| Pydantic | Pydantic | Pydantic / データ検証ライブラリ | Python 数据校验库 | 校验 LLM 返回的 JSON 是否符合格式 |
| 幻觉 | Hallucination | ハルシネーション / 幻覚 | 模型编造不存在的信息 | 企业上线时必须控制的重要风险 |
| 护栏 | Guardrail | ガードレール | 限制危险输出、敏感信息、越权行为的机制 | 提高安全性和合规性 |

## 3. RAG 相关对照

| 中文 | English | 日本語 / 现场说法 | 是什么 | 现场作用 |
| --- | --- | --- | --- | --- |
| RAG | Retrieval-Augmented Generation | RAG / 検索拡張生成 | 先检索资料，再让模型基于资料回答 | 社内问答、知识库问答、文档检索的常见架构 |
| 社内搜索 | Internal Search | 社内検索 | 在公司内部资料中搜索信息 | 日本案件中非常常见的落地场景 |
| 社内问答 | Internal Q&A | 社内QA / 社内問い合わせ対応 | 基于内部文档回答员工问题 | 减少人工查询和重复咨询 |
| 文档库 | Document Store | ドキュメントストア / 文書管理 | 存放 PDF、Word、Markdown、HTML 等资料 | RAG 的知识来源 |
| 文档切分 | Chunking | チャンク分割 | 把长文档拆成小片段 | 让检索能找到更准确的局部内容 |
| 文档片段 | Chunk | チャンク | 切分后的最小检索单位 | RAG 检索和引用的基本单位 |
| 向量化 | Embedding | エンベディング / ベクトル化 | 把文本转换为向量 | 用相似度查找相关资料 |
| 向量数据库 | Vector DB | ベクトルDB | 存储和搜索向量的数据库 | 支持语义检索 |
| 检索器 | Retriever | リトリーバー / 検索器 | 根据问题找相关 chunk 的模块 | 决定 RAG 找到什么资料 |
| Top-K | Top-K | 上位K件 | 检索返回最相关的 K 个片段 | 控制给模型看的资料数量 |
| 引用来源 | Citation / Source | 引用元 / 出典 | 答案依据的文档、页码或片段 | 增强可信度，方便业务用户确认 |

## 4. API 与后端对照

| 中文 | English | 日本語 / 现场说法 | 是什么 | 现场作用 |
| --- | --- | --- | --- | --- |
| 后端 | Backend | バックエンド | 处理业务逻辑、API、数据库、模型调用的服务 | LLM 应用的主要工程实现层 |
| API | API | API / インターフェース | 系统之间交换数据的接口 | 把 LLM 能力提供给前端或其他系统 |
| FastAPI | FastAPI | FastAPI | Python Web API 框架 | 快速封装模型调用和 RAG 服务 |
| 请求 | Request | リクエスト | 客户端发给服务端的数据 | 包含问题、用户信息、参数 |
| 响应 | Response | レスポンス | 服务端返回给客户端的数据 | 包含答案、引用、错误信息 |
| 鉴权 | Authentication / Authorization | 認証 / 認可 | 确认用户是谁、能访问什么 | 企业系统接入时常见要求 |
| 日志 | Logs | ログ | 系统运行记录 | 排查问题、审计、评估质量 |
| 异常处理 | Error Handling | 例外処理 / エラーハンドリング | 处理失败、超时、格式错误 | 提高系统稳定性 |
| 重试 | Retry | リトライ | 失败后重新执行请求 | 处理网络波动或临时错误 |

## 7. 面试或现场说明模板

### 说明 RAG

中文：

```text
RAG 是检索增强生成。系统先根据用户问题从社内文档中检索相关片段，再把这些片段和问题一起交给 LLM 生成答案，并返回引用来源。
```

日本語：

```text
RAG は検索拡張生成です。ユーザーの質問に対して、まず社内文書から関連するチャンクを検索し、その内容をプロンプトに含めて LLM に回答を生成させます。回答と一緒に引用元も返すことで、信頼性を高めます。
```

### 说明 FastAPI 在 LLM 应用中的作用

中文：

```text
FastAPI 是后端 API 层。它接收前端或其他系统的请求，负责参数校验、调用 RAG 或模型服务、处理异常，并把结果以 JSON 返回。
```

日本語：

```text
FastAPI はバックエンドの API 層です。フロントエンドや他システムからのリクエストを受け取り、パラメータ検証、RAG やモデルサービスの呼び出し、例外処理を行い、結果を JSON として返します。
```

### 说明评估

中文：

```text
LLM 应用的评估不是只看能不能回答，而是要看答案是否准确、是否基于资料、是否有引用、格式是否稳定、成本和延迟是否可接受。
```

日本語：

```text
LLM アプリの評価では、単に回答できるかだけではなく、回答の正確性、根拠資料に基づいているか、引用元があるか、出力形式が安定しているか、コストやレイテンシが許容範囲かを確認します。
```

## 8. 学习建议

学习日语对照时，不需要死背所有词。更有效的方法是：

1. 先记核心现场词：`生成AI`, `RAG`, `社内検索`, `PoC`, `API連携`, `要件定義`, `基本設計`。
2. 再把每个词和一个实际文件或流程绑定起来。
3. 最后练习用日语说明自己的 demo。

例如：

| 你的 demo | 可以用日语怎么说 |
| --- | --- |
| RAG CLI | 社内文書を対象にした簡易 RAG 検証ツール |
| RAG API | FastAPI を利用した RAG バックエンド API |
| 评估表 | RAG 回答品質を確認するための評価観点表 |
| 作品集 | 生成 AI PoC を想定した学習用ポートフォリオ |
