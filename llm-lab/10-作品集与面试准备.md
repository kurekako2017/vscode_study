# 10. 作品集与面试准备

如果学习目标是贴近日本 IT 现场、企业 PoC、派遣案件或转职面试，那么只会写 demo 还不够，还需要把案例整理成能说明问题的作品集。

这一章的重点不是“做更多项目”，而是“把已经做过的项目讲清楚”。

## 10.1 为什么要准备作品集

在日本现场，很多岗位不会只看你会不会调用模型，还会看你是否理解这些内容：

- 这个案例解决了什么业务问题
- 为什么选择这种实现方式
- 输入和输出是什么
- 成本、限制、风险在哪里
- 如果要接入企业环境，下一步要补什么

所以作品集的目标不是展示“我会写代码”，而是展示：

- 我理解场景
- 我能把需求落成 PoC
- 我知道这个 PoC 离正式交付还差什么

## 10.2 当前工作区里最值得展示的案例

按“日本现场可解释性”和“案件匹配度”来看，优先顺序建议如下：

1. `structured_output_demo`
2. `doc_qa_agent`
3. `rag_api_demo`
4. `rag_eval_notes`
5. `tool_agent_demo`

其中最值得优先打磨的是：

- `doc_qa_agent`
- `rag_api_demo`

因为这两个最接近日本现场里常见的：

- 社内検索
- ナレッジ検索
- 文档问答
- RAG API
- 生成 AI 后端 PoC

## 10.3 作品集建议结构

每个案例尽量至少准备下面这些材料：

- `README`
- `需求概要`
- `基本设计`
- `测试观点`
- `简单测试用例表`
- 如果可以，再补 `试验结果报告`

如果项目规模很小，也至少要有：

- 背景
- 目标
- 功能范围
- 技术构成
- 限制与后续课题

## 10.4 面试时最常见的说明顺序

介绍案例时，建议按这个顺序说：

1. 背景
2. 目标用户
3. 解决的问题
4. 技术构成
5. 你负责的部分
6. 当前限制
7. 如果继续做会怎么扩展

例如介绍 `rag_api_demo` 时，可以按这种思路表达：

- 这是一个面向社内文档问答场景的最小 `RAG API` PoC
- 后端使用 `FastAPI`
- 支持 `md / txt / pdf`
- 通过本地切分和关键词检索取得候选上下文
- 再把命中片段交给模型生成回答
- 当前版本重点是验证链路，不是正式企业版
- 如果继续扩展，会增加认证、日志、向量检索和评估

## 10.5 日本案件里更容易被认可的展示点

比起“我用了最先进的框架”，更容易被认可的是这些：

- 能说明为什么这个场景适合 `RAG`
- 能说明为什么先做 PoC，而不是直接做复杂 Agent
- 能说明接口输入输出和错误处理
- 能说明文档来源、检索方式、返回来源片段
- 能说明成本、权限、日志、企业环境限制

也就是说，作品集里最重要的不是“炫”，而是：

- 结构清楚
- 场景真实
- 取舍合理

## 10.7 面试时可能会被问什么

常见问题通常会集中在这些方面：

- 为什么选这个案例
- 为什么不用向量库，而先做关键词检索
- 为什么先做 `RAG`，不先做 Agent
- 当前版本的限制是什么
- 如果接企业系统，认证和权限怎么做
- 如果文档变多，性能和索引怎么优化
- 如果回答错了，怎么评估和改进

准备时建议每个案例至少都能回答：

- 为什么这样设计
- 为什么没有做更复杂
- 下一步会补什么

## 10.8 简历上怎么写更自然

简历或项目说明里，不必写成“我做了一个 AI demo”，而更建议写成这种风格：

- 基于 `Python` 和 `OpenAI API` 实现结构化输出 PoC
- 基于本地文档实现最小 `RAG` 问答系统
- 使用 `FastAPI` 将文档问答能力封装为后端接口
- 支持 `PDF` 文档解析与来源返回
- 结合评估观点整理 `RAG` 改进方向

如果目标是日本岗位，也可以在说明里自然带出这些关键词：

- `Python`
- `FastAPI`
- `RAG`
- `社内検索`
- `構造化出力`
- `PoC`
- `API連携`

## 10.10 成本与注意点

- 当前这些案例多数还是学习版或 PoC，不等于正式企业交付物。
- 只要发生模型调用，就会产生实际 API 成本。
- 面试时不要把简化版说成企业生产版，要明确说清范围和限制。
- 如果后续准备转职或投案件，建议优先把 `doc_qa_agent` 和 `rag_api_demo` 打磨到“有文档、有测试观点、有清楚边界”的程度。
