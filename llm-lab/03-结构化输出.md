# 结构化输出

## 1. 为什么这一步非常重要

在日本现场，很多生成 AI 应用真正需要的不是“自由聊天”，而是：

- 分类结果
- 任务清单
- 标签判断
- 固定 JSON
- 可供后端系统接收的结构化结果

所以结构化输出通常比 Agent 更早用到。

很多企业系统不关心模型“聊得多自然”，更关心的是：

- 字段是不是稳定
- 结果能不能校验
- 后端能不能直接接

这就是为什么结构化输出在实际项目里价值很高。

## 2. 这一阶段要掌握什么

这一章至少要掌握这些：

- 固定 JSON 输出
- schema 约束
- 结果校验
- 失败时的基本处理

学完以后，你应该能做到：

- 不只拿到一段自然语言回答
- 而是拿到代码可直接使用的结构化数据

## 3. 这一章学完后，应该会什么

学完这一章，至少要能做到：

- 知道为什么要定义 schema
- 能看懂一个 `Pydantic` model
- 能让模型按固定结构返回结果
- 能把结果转成字典或 JSON
- 能发现输出结构不符合预期时的问题
- 能修改一个现成的结构化输出示例

## 4. 典型场景

这一章对应的实际场景很多，例如：

- 邮件分类
- 问题分类
- FAQ 路由
- 任务拆解
- 表单内容整理
- 需求摘要
- 风险点提取

这些场景的共同点是：

- 需要稳定字段
- 需要后端继续处理结果

## 5. 教程：为什么不能只靠“让模型输出 JSON”

很多初学者会先写这种提示词：

- 请你输出 JSON

这当然可以作为起点，但在实际项目里不够稳。

原因主要是：

- 字段名可能变
- 字段类型可能变
- 可能多出解释文字
- 可能少字段

所以更好的做法是：

1. 先定义结构
2. 再让模型按结构输出
3. 再由代码做校验

也就是说，结构化输出不只是“提示模型”，而是：

- 提示词 + schema + 校验

## 6. 教程：最小结构化输出长什么样

最小结构化输出，一般可以拆成 4 步：

1. 定义 schema
2. 创建客户端
3. 让模型按 schema 返回结果
4. 读取解析后的对象

### 步骤 1：定义 schema

例如先定义一个简单任务计划结构：

```python
from pydantic import BaseModel


class TaskPlan(BaseModel):
    goal: str
    steps: list[str]
```

这一步的作用是：

- 先把“你到底想拿到什么字段”说清楚

### 步骤 2：创建客户端

```python
client = OpenAI(api_key=api_key)
```

这一步和前一章一样，先把调用入口准备好。

### 步骤 3：让模型按 schema 返回

```python
response = client.responses.parse(
    model="gpt-4o",
    instructions="Return a structured plan.",
    input="帮我制定一个学习 RAG 的三步计划",
    text_format=TaskPlan,
)
```

你现在要重点理解的是：

- `responses.parse(...)`
- `text_format=TaskPlan`

这表示：

- 让 SDK 按定义好的结构去解析和校验结果

### 步骤 4：读取解析后的结果

```python
plan = response.output_parsed
print(plan.goal)
print(plan.steps)
```

这样拿到的就不只是文本，而是：

- 已经过结构校验的对象

## 7. 最小示例

下面给一个适合初学阶段理解的最小示例。

这个示例只做一件事：

- 输入一句自然语言需求
- 输出一个结构化计划

```python
import os
import sys

from openai import OpenAI
from pydantic import BaseModel


class TaskPlan(BaseModel):
    goal: str
    steps: list[str]


def main() -> None:
    api_key = os.getenv("OPENAI_API_KEY")
    if not api_key:
        print("ERROR: OPENAI_API_KEY is not set.", file=sys.stderr)
        sys.exit(1)

    client = OpenAI(api_key=api_key)

    response = client.responses.parse(
        model="gpt-4o",
        instructions="Return a concise structured plan in Chinese.",
        input="帮我制定一个学习 RAG 的三步计划",
        text_format=TaskPlan,
    )

    plan = response.output_parsed
    print(plan.model_dump_json(ensure_ascii=False, indent=2))


if __name__ == "__main__":
    main()
```

### 这个最小示例学的是什么

你要从这段代码里看懂 5 个点：

1. 怎么定义 `Pydantic` schema
2. 怎么让模型按 schema 返回
3. 怎么读取解析后的对象
4. 怎么转成 JSON
5. 为什么结构比自由文本更适合接后端

## 8. 代码拆解

### 1. 为什么先定义 `BaseModel`

因为结构化输出的重点不是“让模型多说一点”，而是：

- 让输出字段稳定下来

定义 `BaseModel` 的过程，本质上是在定义一份数据合同。

### 2. `responses.parse(...)` 在做什么

它的作用可以先简单理解成：

- 请求模型
- 按 schema 解析结果
- 把结果转成更稳定的对象

也就是说，你拿到的不只是文本，而是更适合程序继续处理的数据。

### 3. `output_parsed` 是什么

它表示：

- 已经解析好的结构化结果

你后面可以直接：

- 取字段
- 转字典
- 转 JSON
- 写入文件
- 接到 API 返回值

### 4. 为什么这一步比普通聊天更接近项目

因为企业系统通常不会只接一大段自由文本。

系统更常需要的是：

- `category`
- `priority`
- `summary`
- `tasks`
- `risk_level`

这些结构化字段才能继续：

- 存数据库
- 调工作流
- 交给前端展示
- 接到其他系统

## 9. 进一小步：加字段约束

上面的最小示例还比较简单。

再进一步，你应该学会给字段加约束。

例如：

```python
from typing import Literal

from pydantic import BaseModel


class TaskPlan(BaseModel):
    goal: str
    level: Literal["beginner", "intermediate", "advanced"]
    steps: list[str]
```

这样做的好处是：

- 输出值更稳定
- 结果更容易用于判断分支

比如后端就可以直接根据 `level` 决定后续流程。

## 12. 常见错误与排查

### 1. 只写“输出 JSON”，但结构还是不稳定

表现通常是：

- 字段忽多忽少
- 多出解释文字
- 字段类型不一致

先做的事是：

- 不要只靠提示词
- 先定义 schema
- 再用代码做解析和校验

### 2. schema 定义得太松

表现通常是：

- 虽然能跑
- 但字段值不够稳定

例如如果某个字段本来只有 3 种可能，就不应该完全放开。

这时要考虑：

- 是否可以用 `Literal`
- 是否可以拆成更明确的字段

### 3. schema 定义得太复杂

表现通常是：

- 初学阶段很难读懂
- 调试成本变高

更好的做法是：

- 先从 2 到 4 个字段开始
- 先把核心结构跑通

### 4. 只会打印 JSON，不会继续用

这也是常见问题。

结构化输出的价值不只是“打印得好看”，而是：

- 能写文件
- 能返回 API
- 能驱动后续流程

所以练习时不要只停在打印结果。
