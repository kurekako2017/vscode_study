# 数据库 SQL 与数据访问

## 1. 为什么数据库能力很重要

企业 Java 项目大部分是业务系统，本质上都是数据录入、查询、统计、审批、导入导出。

所以你必须会:

- 看表结构
- 写 SQL
- 调试数据
- 理解事务
- 排查数据问题

## 2. SQL 基础

### DQL

- `SELECT`
- `WHERE`
- `ORDER BY`
- `GROUP BY`
- `HAVING`

### DML

- `INSERT`
- `UPDATE`
- `DELETE`

### DDL

- `CREATE`
- `ALTER`
- `DROP`

## 3. 必须掌握的查询能力

- 单表查询
- 多表关联
- 内连接和左连接
- 分页
- 聚合统计
- 子查询

## 5. ORM 与数据访问框架

适合 SQL 可控性要求较高的场景。

重点:

- Mapper 接口
- XML SQL 映射
- 参数绑定
- 结果映射

### Hibernate / JPA

适合以对象映射驱动的场景。

重点:

- 实体类
- 注解映射
- Repository
- 查询方法

## 7. ETL 与数据集成工具

ETL 也属于数据库操作相关能力的一部分，只是它更偏向:

- 跨系统数据搬运
- 批量导入导出
- 文件和数据库之间的数据转换
- 定时联携

很多对日项目里，数据库工作不只是写 SQL，还会涉及:

- 从 CSV / Excel 导入数据库
- 从数据库导出接口文件
- 多个系统之间做数据同步
- 配合 batch、jobnet、JP1 / Systemwalker 做定时执行

### DataSpider

日本项目里非常常见的数据联携 / ETL 工具。

常见用途:

- DB 到 DB 数据同步
- CSV、Excel、XML、API 和数据库之间的转换
- 定时批量导入导出
- 系统间接口联携

### ASTERIA Warp

也是日本现场常见的可视化数据联携平台。

常见用途:

- 无代码或低代码方式配置数据流
- 文件接口处理
- 数据库更新
- 与企业内部系统做集成

### Talend

国际上更常见的 ETL 工具，日本项目里也会出现。

常见用途:

- 批量数据清洗
- 数据迁移
- 多数据源整合

### HULFT

严格说它更偏文件传输，但在日本项目里经常和 ETL、batch、数据库联携一起出现。

常见用途:

- 系统间传文件
- 夜间批处理交换数据文件
- 配合数据库导入导出使用

## 8. AWS 相关数据库与数据平台

现在不少对日项目已经不只是本地机房数据库，也会用 AWS 上的数据服务。

### RDS

`RDS` 是 AWS 的托管关系型数据库服务。

常见场景:

- 承载 `MySQL`、`PostgreSQL`、`Oracle`、`SQL Server`
- Java Web 系统的主业务数据库
- 测试环境、生产环境数据库统一运维

对日项目里常见关注点:

- 备份
- 权限管理
- 参数组
- 监控
- 性能调优

### Aurora

`Aurora` 是 AWS 的高可用关系型数据库，兼容 `MySQL` 或 `PostgreSQL`。

常见场景:

- 对可用性要求更高的业务系统
- 读写分离
- 希望比普通 `RDS` 有更强扩展性的场景

### DynamoDB

`DynamoDB` 是 AWS 的 NoSQL 数据库。

常见场景:

- 高并发 key-value 查询
- 会话、缓存型数据
- 不适合复杂关系查询的轻量数据存储

对传统日企 Java 项目来说，它没有 `RDS` 那么常见，但在云原生项目里会出现。

### S3 + Athena

这组通常不是 OLTP 业务数据库，而是分析和查询用。

常见场景:

- 把 CSV、日志、导出文件放到 `S3`
- 用 `Athena` 直接查 `S3` 上的数据
- 做件数分析、日志分析、报表辅助查询

这在文件联携、batch 结果分析、审计查询里比较常见。

### DMS

`DMS` 是 AWS 的数据库迁移服务。

常见场景:

- 本地数据库迁移到 AWS
- 老系统数据库迁到新数据库
- 做全量 + 差分同步
- 配合正式切换做数据移行

如果项目里有 `移行`、`切替`、`差分移行`，就很可能会碰到这类服务。

### Glue

`Glue` 更偏数据集成、ETL、元数据管理。

常见场景:

- 数据清洗
- 文件格式转换
- ETL 流程
- 和 `S3`、`Athena`、`Redshift` 联动

如果项目偏数据平台、分析平台，`Glue` 会比传统 ETL 工具更常出现。

### Redshift

`Redshift` 是 AWS 的数据仓库服务。

常见场景:

- 汇总大量业务数据
- BI 报表
- 经营分析
- 历史数据分析

它更适合分析型查询，不是拿来替代日常事务型业务库的。

## 9. 数据库相关 AWS 日文高频术语

下面这些词在 AWS 项目的日文履历、设计书、运维说明里很常见。

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `DB移行` | 数据库迁移 | 老数据库迁到新数据库或云环境 |
| `RDS移行` | RDS 迁移 | 本地 DB 或 EC2 上 DB 迁到 RDS |
| `スナップショット` | 快照 | 备份、回滚前保留、环境复制 |
| `レプリケーション` | 复制、同步 | 主从同步、迁移同步、只读副本 |
| `フェイルオーバー` | 故障切换 | 主实例故障时切换到备用实例 |
| `監視` | 监控 | 监控 CPU、连接数、慢 SQL、存储容量 |
| `バックアップ` | 备份 | 定时备份、保留恢复点 |
| `リストア` | 恢复 | 从备份或快照恢复数据库 |

这些词也经常一起出现:

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `冗長構成` | 冗余结构 | 高可用、双机、Multi-AZ |
| `可用性` | 可用性 | SLA、高可用设计 |
| `自動バックアップ` | 自动备份 | RDS 日常运维 |
| `手動スナップショット` | 手动快照 | 发布前、移行前保留 |
| `復旧` | 恢复、复旧 | 故障恢复、数据恢复 |
| `監視設定` | 监控设置 | CloudWatch 告警、指标监控 |
| `性能監視` | 性能监控 | 监控慢 SQL、负载、I/O |
| `リードレプリカ` | 只读副本 | 读写分离、查询负载分担 |
| `マルチAZ` | 多可用区 | RDS / Aurora 高可用部署 |
| `メンテナンスウィンドウ` | 维护窗口 | 升级、补丁、重启安排 |

看 AWS 数据库相关资料时，可以先这样理解:

- 看到 `DB移行`、`RDS移行`、`レプリケーション`
  多半是在说迁移和同步
- 看到 `スナップショット`、`バックアップ`、`リストア`
  多半是在说备份和恢复
- 看到 `フェイルオーバー`、`冗長構成`、`マルチAZ`
  多半是在说高可用
- 看到 `監視`、`監視設定`、`性能監視`
  多半是在说运维监控

## 10. 对日项目中的数据库工作特点

- 常要根据 Excel 定义书比对字段
- 常要查生产或测试数据定位问题
- 需要严格区分查询、更新、回滚风险
- 不少项目会使用存储过程、批处理、定时任务

另外还经常会碰到这些“不是纯 SQL，但和数据库强相关”的工作:

- ETL 工具配置数据同步

## 11. 对日项目常见数据库定义书与术语

很多日企项目里，数据库相关资料不会只是一份 SQL 文件，而是一整套文档。

### ER図

- 英文一般写作 `ER Diagram`
- 用来表示表和表之间的关系
- 重点看:
  - 主键 `PK`
  - 外键 `FK`
  - `1:1`、`1:N`、`N:M`

### テーブル定義書

- 中文通常叫表定义书
- 英文常写 `Table Definition`
- 一般会写:
  - 表名
  - 字段名
  - 数据类型
  - 长度
  - `NULL` 可否
  - 主键 / 外键
  - 默认值
  - 备注

### カラム定義書 / 項目定義書

- 更细时会单独拆字段定义
- 常写字段业务含义、取值范围、代码值、输入输出规则

### DDL一覧

- `DDL List`
- 用来管理建表、改表、索引、约束相关 SQL
- 常见内容:
  - `CREATE TABLE`
  - `ALTER TABLE`
  - `CREATE INDEX`
  - `DROP INDEX`

### CRUD一覧

- `CRUD List`
- 用来表示哪个功能对哪张表做了 `Create / Read / Update / Delete`
- 对代码阅读、测试设计、影响分析都很有用

### インデックス定義

- 英文一般叫 `Index Definition`
- 用来整理:
  - 主键索引
  - 唯一索引
  - 普通索引
  - 复合索引
- 常用于性能调优和 SQL 调查

### 外部キー定義

- 英文一般叫 `Foreign Key Definition`
- 用来说明表关系和更新删除限制
- 常见关注点:
  - 是否允许孤儿数据
  - 删除父表数据时是否联动
  - 数据导入顺序

## 12. 这些数据库文档通常怎么配合看

在对日项目里，数据库资料通常不是分开看的，而是串起来看的。

- 先看 `ER図`
  明白系统里有哪些表、表之间怎么关联
- 再看 `テーブル定義書`
  明白每张表每个字段是什么
- 再看 `DDL一覧`
  明白数据库最终怎么落地
- 再看 `CRUD一覧`
  明白每个功能实际操作了哪些表
- 如果涉及画面输入
  继续看 `画面項目定義書`、`入力項目・DB項目対応一覧`
- 如果涉及测试
  再看 `単体テスト仕様書`、`結合テスト仕様書`

这样做的好处是:

- 能更快定位问题发生在哪一层
- 能判断改一个字段会影响哪些功能
- 能把设计书、SQL、代码、测试串起来

## 13. 常见 SQL 调优知识点

企业项目里，SQL 能跑和 SQL 跑得稳、跑得快，不是一回事。

### 索引

索引是最常见的调优点。

重点理解:

- 哪些查询条件适合建索引
- 主键索引、唯一索引、普通索引、复合索引的区别
- 索引不是越多越好
- `INSERT`、`UPDATE`、`DELETE` 太多时，索引也会带来维护成本

常见场景:

- 登录时按 `username` 查用户
- 列表按 `status`、`create_time` 查询
- 关联时按外键字段查询

### EXPLAIN

`EXPLAIN` 用来查看 SQL 的执行计划。

至少要会看:

- 是否走索引
- 是否出现全表扫描
- `JOIN` 时先扫哪张表
- 预计扫描行数是否过大

在项目里，查慢 SQL 时，通常都会配合 `EXPLAIN` 一起看。

### 全表扫描

全表扫描不一定一定有问题，但数据量大时风险很高。

常见导致原因:

- 查询条件字段没有索引
- 对索引字段做函数处理
- 模糊查询写法不利于索引命中
- 条件选择性太差

如果线上表数据很大，全表扫描往往会直接影响响应时间。

### JOIN 顺序

多表查询时，不只是 SQL 能查出来就行，还要关注:

- 先过滤小结果集，再去关联大表
- 关联字段是否有索引
- 是否把不必要的表也连进来了

对日项目里常见的慢查询，很多就是多表 `JOIN` 过多、条件不清、索引没跟上。

### N+1

`N+1` 常出现在 ORM 框架里，比如 Hibernate / JPA。

意思是:

- 先查 1 次主数据
- 再对每一条数据各查 1 次明细

这样数据一多，SQL 次数会暴涨。

常见解决方向:

- 适当使用 `JOIN FETCH`
- 一次性批量取出需要的数据
- 避免在循环里反复查数据库

### 分页优化

分页查询在后台列表、管理画面里很常见。

重点关注:

- 大偏移量分页会越来越慢
- 排序字段最好有索引
- 不要先全量查出再在 Java 里分页

如果数据量很大，可能需要考虑:

- 基于主键或时间的游标分页
- 先查主键再回表取详情

### 批量更新注意点

批量更新、批量删除、批量导入在 batch 和 ETL 里很常见。

常见注意点:

- 一次更新太多数据会锁表或长事务
- 要考虑分批提交
- 要考虑失败时怎么回滚或重跑
- 要确认索引、触发器、外键对性能的影响
- 要先确认 `WHERE` 条件，避免误更新全表

对日项目里，这类处理通常还要配合:

- 作业管理工具
- 结果件数确认
- 执行前后数据核对
- 障害时的重跑方案

## 14. 对日项目里常见数据库问题排查套路

### 查不到数据

先确认是不是“真的没有”，还是“条件不对”。

常见排查顺序:

- 确认查的是正确环境
  - 开发、测试、生产库不要搞混
- 确认 `WHERE` 条件是否正确
  - 主键、状态、逻辑删除标记、日期条件
- 确认是否有事务未提交
- 确认是否被权限条件、租户条件、组织条件过滤掉了
- 如果是通过程序查不到
  - 对照日志里的实际 SQL
  - 确认参数绑定值是否和预期一致

对日项目里，这类问题常常最后不是“数据库没数据”，而是:

- 条件写错
- 联携还没跑
- 目标环境查错

### 数据重复

先判断是“显示重复”，还是“DB 里真的重复写进去了”。

常见排查顺序:

- 用 SQL 直接查重复件数
- 确认表上是否有唯一约束
- 确认 batch / 联携是否重复执行
- 确认画面按钮是否被重复点击
- 确认程序是否缺少幂等控制
- 确认 `JOIN` 是否把 1:N 关系查成了重复显示

常见根因:

- 没有唯一索引
- 接口重试导致重复插入
- ETL 或 batch 重跑时没有去重
- SQL `JOIN` 写法导致画面重复

### 更新失败

更新失败时，不要只看“报错了”，要先分清是程序问题、SQL 问题，还是数据约束问题。

常见排查顺序:

- 看异常日志
  - 主键冲突
  - 非空约束
  - 外键约束
  - 长度超限
- 确认 `WHERE` 条件是否正确
- 确认更新对象是否真的存在
- 确认事务是否回滚了
- 确认是否被别的事务锁住
- 如果是 ORM
  - 确认实体状态、提交时机、脏数据更新是否正常

### 联携件数不一致

这是日本项目里非常常见的问题，尤其是 batch、ETL、文件接口场景。

常见排查顺序:

- 确认来源侧件数
- 确认目标侧件数
- 确认中间过滤条件
- 确认失败件数、跳过件数、重复件数
- 确认文件是否缺行、空行、编码异常
- 确认是否有重跑、部分重跑、断点续跑

经常要对的不是只有“总件数”，还包括:

- 新增件数
- 更新件数
- 删除件数
- 错误件数
- 跳过件数

### 慢 SQL

慢 SQL 的排查不能只看 SQL 文本，要一起看数据量、索引、执行计划。

常见排查顺序:

- 先确认慢的是哪条 SQL
- 用 `EXPLAIN` 看执行计划
- 看是否走索引
- 看扫描行数是否过大
- 看是否有不必要的 `JOIN`
- 看排序、分页、聚合是否代价过高
- 看是不是应用侧触发了 `N+1`

常见改进方向:

- 补索引
- 改写查询条件
- 拆分复杂 SQL
- 减少查询列
- 调整分页方式

### 锁等待 / 死锁

这类问题通常出现在高并发更新、batch 批量更新、多个作业同时跑的时候。

常见排查顺序:

- 确认是锁等待还是死锁
- 查数据库锁信息、事务信息
- 看是哪张表、哪几条记录冲突
- 看是否有长事务未提交
- 看更新顺序是否不一致
- 看 batch 是否一次提交太多数据

常见解决方向:

- 缩短事务时间
- 分批提交
- 统一更新顺序
- 避免高峰时段跑重型 batch
- 给高频条件加合适索引，减少锁范围

## 15. 数据移行相关文档

对日项目里，`数据移行` 往往是数据库工作里单独的一大块，不只是“写一段导入 SQL”。

常见场景:

- 老系统切新系统
- 多套系统数据合并
- Excel / CSV 主数据导入
- 测试环境、正式环境初始化

### 移行計画書

- 中文一般叫移行计划书
- 英文可理解为 `Migration Plan`
- 主要写:
  - 移行对象
  - 移行范围
  - 移行日程
  - 停机窗口
  - 角色分工
  - 风险与回退方式

### 移行方針書

- 说明这次移行是全量还是差分
- 说明是一次性切换还是分阶段切换
- 说明主数据、交易数据、历史数据分别怎么处理

### 移行対象一覧

- 列出哪些表、哪些文件、哪些主数据需要移行
- 常见列:
  - 对象名
  - 来源
  - 目标
  - 件数预估
  - 移行方式
  - 负责人

### 移行マッピング一覧

- 中文可以理解成移行映射表
- 是数据移行里很重要的一份资料
- 常写:
  - 旧字段
  - 新字段
  - 类型转换规则
  - 代码值转换规则
  - 默认值
  - 空值处理

### 移行手順書

- 中文一般叫移行步骤书
- 常写:
  - 执行顺序
  - 使用脚本
  - 输入文件
  - 输出结果
  - 检查点
  - 异常时处理方法

### 移行結果報告書

- 用来记录本次移行执行结果
- 常见内容:
  - 计划件数
  - 实际件数
  - 成功件数
  - 错误件数
  - 差异说明
  - 障害与处理结果

### 移行リハーサル資料

- 正式切换前，经常会先做 rehearsal
- 常用来确认:
  - 移行耗时
  - 手顺是否有问题
  - 件数是否对得上
  - 回退是否可行

## 16. 数据移行时常见的技术点

移行时不只是“把数据搬过去”，还会遇到很多具体技术问题。

- 字段长度不一致
- 数据类型不一致
- 代码值转换
- 主键重新採番
- 主外键关系重建
- 重复数据清理
- 空值补默认值
- 逻辑删除数据是否移行
- 历史数据是否保留

如果是通过 ETL 或 batch 做移行，还要特别注意:

- 是否支持断点续跑
- 是否能输出错误明细
- 是否能按表分批执行
- 是否能保留移行日志和件数结果

## 17. 数据移行验证时常见的确认点

移行最怕的不是“脚本没跑”，而是“看起来跑完了，但数据不对”。

所以通常会确认:

- 来源件数和目标件数是否一致
- 主表和明细表件数是否合理
- 主键、外键关系是否正确
- 必填项是否为空
- 金额、数量、状态值是否异常
- 抽样比对画面显示是否正常
- 接口、batch 是否还能基于移行后数据正常运行

常见验证方式:

- 总件数确认
- 条件别件数确认
- 抽样突合
- 业务侧确认
- 画面确认

## 18. 数据移行相关日文高频术语

这些词在移行计划书、移行手顺书、移行结果报告里出现频率很高。

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `移行` | 移行、迁移 | 老系统切新系统、数据搬迁 |
| `切替` | 切换 | 正式从旧系统切到新系统 |
| `本番移行` | 正式环境移行 | 生产环境正式执行数据移行 |
| `リハーサル` | 演练、预演 | 正式移行前的 rehearsal |
| `移行元` | 移行源 | 原系统、原数据库、旧文件 |
| `移行先` | 移行目标 | 新系统、新数据库、目标文件 |
| `マッピング` | 映射 | 旧字段到新字段的对应关系 |
| `採番` | 编号生成 | 主键、流水号重新生成 |
| `洗替` | 全量替换 | 先删后灌、整批刷新 |
| `差分移行` | 差分移行、增量迁移 | 只迁移新增或变更数据 |

这些词也经常一起出现:

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `移行方針` | 移行方针 | 全量还是差分、一次还是分阶段 |
| `移行対象` | 移行对象 | 哪些表、哪些文件、哪些主数据要迁移 |
| `移行手順` | 移行步骤 | 先后顺序、执行命令、检查点 |
| `移行結果` | 移行结果 | 成功件数、失败件数、差异说明 |
| `切戻し` | 回退、切回 | 移行失败时恢复到旧系统 |
| `移行判定` | 移行判定 | 是否达到切换条件 |
| `差分抽出` | 差分抽取 | 抽取新增、修改、删除对象 |
| `事前確認` | 事前确认 | 正式移行前确认环境、数据、手顺 |
| `事後確認` | 事后确认 | 移行后确认件数、画面、接口 |
| `移行漏れ` | 漏移行 | 部分数据没有成功迁过去 |

看移行资料时，可以先这样理解:

- 看到 `移行元`、`移行先`、`マッピング`
  多半是在讲数据从哪里来、到哪里去、怎么对应
- 看到 `切替`、`本番移行`、`切戻し`
  多半是在讲正式上线切换和失败回退
- 看到 `リハーサル`、`事前確認`、`事後確認`
  多半是在讲移行前后的验证流程
- 看到 `洗替`、`差分移行`、`差分抽出`
  多半是在讲移行方式

## 19. 数据库相关日文高频术语对照表

下面这些词在日文设计书、障害票、联携说明、测试报告里很常见。

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `件数` | 件数、记录数 | 对账、联携结果确认、测试结果确认 |
| `取込` | 导入、取入 | `CSV` 导入数据库、文件取入处理 |
| `連携` | 联携、系统对接 | 系统间数据同步、接口传输 |
| `洗替` | 全量替换、整批刷新 | 先清后灌、整表重建、日次全量更新 |
| `差分` | 差分、增量 | 只处理新增、变更、删除部分 |
| `突合` | 核对、比对 | 来源数据和目标数据件数或内容比对 |
| `排他` | 排他控制、锁控制 | 防止并发更新、重复执行、数据冲突 |
| `ロールバック` | 回滚 | 更新失败后撤销事务 |

| `論理削除` | 逻辑删除 | 设置删除标记，不真正删数据 |
| `物理削除` | 物理删除 | 真正从表里删除记录 |

这些词也经常和上面那几个一起出现:

| 日文 | 中文意思 | 常见场景 |
|---|---|---|
| `取込件数` | 导入件数 | ETL、batch、接口文件处理结果 |
| `処理件数` | 处理件数 | batch 执行结果汇总 |
| `対象件数` | 目标件数 | 本次处理对象总数 |
| `正常件数` | 正常件数 | 成功处理条数 |
| `異常件数` | 异常件数 | 失败条数 |
| `エラー件数` | 错误件数 | 异常、校验失败、SQL 失败条数 |
| `未処理` | 未处理 | 还未联携、未执行、未更新 |
| `再実行` | 重跑、再执行 | batch 或 job 失败后重新执行 |
| `再取込` | 重新导入 | 导入失败后重新取入数据 |
| `整合性` | 一致性、整合性 | 主外键一致、件数一致、数据正确性 |
| `整合チェック` | 整合性检查 | 批处理、数据巡检、结果核对 |
| `重複` | 重复 | 重复数据、重复联携、重复插入 |
| `欠損` | 缺失 | 文件缺行、字段缺值、数据缺失 |
| `採番` | 编号生成 | 主键编号、流水号生成 |
| `採番不正` | 编号异常 | 重号、缺号、编号规则错误 |

读日文数据库相关文档时，可以先这样理解:

- 看到 `件数`、`突合`、`整合性`
  多半是在做结果核对
- 看到 `取込`、`連携`、`再取込`
  多半是在说文件接口或 ETL
- 看到 `排他`、`ロールバック`、`コミット`
  多半是在说事务、并发、锁
- 看到 `論理削除`、`物理削除`
  多半是在说删除方式和数据保留策略

## 20. 数据库场景常用日语表达例句

下面这些表达，在设计书、移行资料、障害票、会议说明里都很常见。

| 日语表达 | 中文意思 |
|---|---|
| `取込件数と出力件数を突合する` | 核对导入件数和输出件数 |
| `差分データのみ連携する` | 只联携差分数据 |
| `論理削除フラグを立てる` | 设置逻辑删除标记 |
| `排他制御により二重更新を防止する` | 通过排他控制防止重复更新 |
| `移行元データと移行先データを比較する` | 比较移行源数据和移行目标数据 |
| `対象データを洗替する` | 对目标数据做全量替换 |
| `取込対象外データをスキップする` | 跳过不属于导入对象的数据 |
| `エラー件数を集計する` | 汇总错误件数 |
| `再取込を実施する` | 执行重新导入 |
| `更新対象データを抽出する` | 抽取更新对象数据 |
| `主キーの採番ルールを確認する` | 确认主键编号规则 |
| `整合性チェックを実施する` | 进行整合性检查 |
| `コミット前に件数を確認する` | 在提交前确认件数 |
| `ロールバック後に再実行する` | 回滚后重新执行 |
| `論理削除データは移行対象外とする` | 逻辑删除数据不作为移行对象 |

开会或写文档时，也很常出现这种说法:

- `取込結果を明日までに共有してください`
  请在明天之前共享导入结果
- `件数差異の原因を調査します`
  我们来调查件数差异的原因
- `差分連携の条件を再確認します`
  重新确认差分联携条件
- `排他制御の影響で更新が失敗した可能性があります`
  可能因为排他控制影响导致更新失败
- `移行リハーサルで手順と所要時間を確認します`
  在移行 rehearsal 中确认步骤和所需时间

## 21. 学习建议

- 自己建表并写 20 到 30 条 SQL
- 练习用户表、订单表、部门表、权限表的多表查询
- 用 JDBC 或 MyBatis 完成一次完整 CRUD
- 学会从日志和 SQL 执行结果定位问题
- 了解一个 ETL 工具的基本概念，比如 `DataSpider` 或 `Talend`
- 练习 `CSV -> 数据库`、`数据库 -> CSV` 这样的简单数据联携场景
- 尝试自己做一份简化版 `ER图`、`表定义书`、`CRUD 一览`
- 用 `EXPLAIN` 观察几条慢 SQL 和快 SQL 的差别
- 练习给查询条件、排序字段、关联字段设计基础索引
