# 对日项目常用工具总览

本页按工具类别整理日本开发项目中经常出现的软件、工具和技术。  
除了 IDE、数据库工具，也补充了日本现场很常见的中间件、ジョブ管理、帳票工具、ETL 和低代码平台。

## 1. 开发工具

### JDK

- 作用: Java 编译与运行环境
- 常见版本: 8、11、17
- 重点: 环境变量、版本切换、兼容性

### Eclipse

- 传统项目常用
- 常搭配 Tomcat、SVN、Checkstyle 插件
- 常见插件名称与用途:
- Eclipse Web Tools Platform: 用于 Dynamic Web Project、Servlet、JSP、Tomcat Server 管理
- Subclipse / Subversive: 用于 SVN 检出、更新、提交
- EGit: 用于 Git 操作
- Checkstyle Plugin: 用于代码规范检查
- Spring Tools: 用于 Spring 配置分析、Bean 跳转和 Spring 项目辅助开发

### IntelliJ IDEA

- 现代项目常用
- 对 Spring Boot、Maven、Git 支持更好
- 常见插件名称与用途:
- Lombok Plugin: 识别 Lombok 注解，避免编辑器报红
- MyBatisX: 用于 Mapper 接口和 XML 快速跳转
- CheckStyle-IDEA: 用于本地代码规范检查
- SonarLint: 用于实时静态代码问题提示
- Database Tools: 用于连接数据库并执行 SQL
- Maven Integration: 用于依赖刷新、生命周期执行和打包

### STS

- Spring Tool Suite
- 用于 Spring 项目开发
- 常见功能与用途:
- Spring 项目向导: 快速创建 Spring 项目
- Boot Dashboard: 管理和启动 Spring Boot 应用
- Server 视图: 用于部署和调试 Tomcat
- Maven / Git 集成: 沿用 Eclipse 生态完成构建和版本管理

## 2. Web 应用服务器与部署平台

### Tomcat

- 最常见
- 适合 Servlet、JSP、Spring MVC、Spring Boot 外置部署
- 用途:
- 本地部署 WAR 包
- 联调 JSP/Servlet 项目
- 验证 Session、Filter、登录流程等 Web 功能

### JBoss / WildFly

- Java EE 项目常见
- 简历里有时会写成 `JBoss`，口语里偶尔会误写成 `jobss`
- 用途:
- 部署 Java EE 应用
- 支撑较重型的企业中间件项目

### WebLogic

- Oracle 企业级应用服务器
- 金融、政企老系统中可能出现
- 用途:
- 部署大型企业系统
- 管理数据源、事务、集群和企业级服务

### WebSphere

- IBM 企业级应用服务器
- 日本大型企业老系统可能仍在使用
- 用途:
- 运行大型传统企业系统
- 管理复杂中间件集成和事务环境

### WebOTX

- NEC 系企业应用服务器
- 日本现场偶尔会遇到
- 用途:
- 部署 Java EE / 企业级 Web 系统

### IIS

- Windows 系服务器环境常见
- 多用于 .NET 或反向代理场景
- 用途:
- 承载 Web 应用
- 反向代理或静态资源发布

## 3. Batch / 作业管理平台

### JP1

- 日本项目里非常常见的ジョブ管理平台
- 完整名称常见为 `JP1/AJS`
- 用途:
- 管理 batch 执行顺序
- 设置日次、月次、定时作业
- 处理返回码、告警、重跑、依赖关系

### Systemwalker

- 富士通系运维 / 作业管理平台
- 日本现场也较常见
- 用途:
- ジョブ调度
- 监控批处理与系统运行状态

### Tivoli Workload Scheduler

- IBM 系作业调度平台
- 旧项目和大企业系统里可能出现
- 用途:
- 管理作业网络
- 控制跨服务器批处理流程

### Windows Task Scheduler / cron

- 学习或小规模项目常用替代方案
- 用途:
- 定时执行脚本、jar、bat、sh
- 模拟 JP1 / Systemwalker 的基础调度功能

### 免费替代方案推荐

| 工具 | 类型 | 适合场景 | 说明 |
|---|---|---|---|
| `Quartz` | Java 内嵌调度器 | 单体 Java / Spring 项目里的 batch 定时执行 | 最适合 Java 开发者入门，直接集成到应用里 |
| `JobRunr` | Java 后台任务框架 | Spring Boot 中做持久化后台任务、定时任务 | 比 Quartz 更现代，适合做 Java 内部 batch |
| `Jenkins` | CI / 定时执行平台 | 用定时构建方式执行 shell、bat、jar、maven batch | 学习成本低，很多现场本来就有 Jenkins |
| `Rundeck` | 作业调度 / 运维自动化 | 想做接近 JP1 的可视化作业管理 | 有 UI、日志、权限、调度，比较像轻量ジョブ管理 |
| `Apache Airflow` | 工作流调度平台 | 多步骤 batch、数据处理、依赖关系复杂的流程 | 更偏数据流程和工作流，不是纯 Java 专用 |
| `Windows Task Scheduler / cron` | OS 调度 | 学习、个人练习、小规模环境 | 最轻量，但缺少企业级可视化和依赖控制 |

推荐顺序:

- Java 初学和项目内 batch: `Quartz`
- Spring Boot 现代项目: `JobRunr`
- 想模仿 JP1 / Systemwalker 的“作业管理感”: `Rundeck`
- 已有 Jenkins 环境: `Jenkins`
- 流程复杂、跨系统多步骤: `Apache Airflow`

### 免费工具对比表

| 工具 | 是否免费 | 是否有 Web UI | 是否适合 Java | 是否支持依赖关系 | 是否支持重跑 | 是否接近 JP1 / Systemwalker 风格 |
|---|---|---|---|---|---|---|
| `Quartz` | 是 | 否 | 很适合 | 基础支持，需代码实现 | 可实现 | 低 |
| `JobRunr` | 是 | 有 | 很适合 | 基础支持 | 支持 | 低到中 |
| `Jenkins` | 是 | 有 | 适合 | 支持，通过 Job / Pipeline | 支持 | 中 |
| `Rundeck` | 是 | 有 | 适合 | 支持 | 支持 | 高 |
| `Apache Airflow` | 是 | 有 | 一般，更偏 Python / 工作流 | 强 | 支持 | 中到高 |
| `Windows Task Scheduler / cron` | 是 | 有限，OS 自带界面或无 | 一般 | 很弱 | 有限 | 低 |

简要理解:

- `Rundeck` 最像轻量版作业管理平台，最接近 `JP1` / `Systemwalker` 的感觉。
- `Jenkins` 虽然本职是 CI，但很多现场会拿来做 batch 调度，也比较实用。
- `Quartz` 和 `JobRunr` 更像“Java 内部任务框架”，不是企业级作业管理平台。
- `Airflow` 很强，但更偏数据流程、工作流平台，不是传统日企 batch 调度的直觉替代。

## 4. AWS 云平台相关

### EC2

- AWS 上最基础、最常见的服务器服务
- 日本项目里很多“上云”其实仍然是传统服务器搬到 EC2
- 用途:
- 部署 Java Web 应用
- 部署 batch 程序
- 搭配 Tomcat、JBoss、WebLogic 等中间件运行

### RDS

- 托管数据库服务
- 用途:
- 承载 Oracle、MySQL、PostgreSQL、SQL Server 等数据库
- 减少自管数据库运维成本

### S3

- 对象存储
- 用途:
- 保存上传文件、导出文件、CSV、日志归档、备份文件
- 存放 batch 输出结果和临时交换文件

### CloudWatch

- AWS 监控与日志服务
- 用途:
- 查看应用日志
- 监控 CPU、内存、错误率、作业状态
- 设置批处理或应用异常告警

### IAM

- 权限控制基础服务
- 用途:
- 管理用户、角色、策略
- 控制应用、batch、部署脚本访问 AWS 资源的权限

### Systems Manager (SSM)

- AWS 运维管理服务
- 用途:
- 远程登录和执行命令
- 参数管理
- 替代部分跳板机和手工 SSH 登录场景

### ELB / ALB

- 负载均衡服务
- 用途:
- 将流量分发到多个 Java 应用实例
- 提供 Web 服务入口

### Route 53

- AWS DNS 服务
- 用途:
- 域名解析
- 应用入口切换

### AWS Batch

- AWS 原生 batch 执行平台
- 适合新项目或云原生 batch
- 用途:
- 提交批处理 Job
- 管理队列、计算环境、重试策略
- 执行 Java batch、ETL、数据处理任务

### EventBridge

- AWS 事件与调度服务
- 用途:
- 定时触发 batch
- 触发 Lambda、ECS Task、Step Functions

### Lambda

- 无服务器计算
- 用途:
- 执行轻量 batch
- 执行文件处理、通知、定时任务

### ECS / Fargate

- 容器运行平台
- 用途:
- 部署 Java API、Spring Boot 服务
- 运行定时 batch 容器
- `Fargate` 可减少服务器管理

### ECR

- 容器镜像仓库
- 用途:
- 保存 Java 应用和 batch 的 Docker 镜像

### CodePipeline / CodeBuild / CodeDeploy

- AWS 持续交付相关服务
- 用途:
- 自动构建、自动部署
- 连接 Git、S3、EC2、ECS

### CloudFormation / Terraform

- 基础设施即代码
- 用途:
- 管理 AWS 环境
- 自动创建服务器、网络、数据库、权限

### Step Functions

- 工作流编排服务
- 用途:
- 串联多个 batch 或 Lambda
- 实现类似作业流、状态控制、重试逻辑

- 新项目、现代项目常见

- 常见配套工具:

## 16. AWS 上 Java / Batch 常见部署知识点

### Java Web 部署常见模式

- `EC2 + Tomcat`
  最传统，和本地 Linux 服务器部署思路接近
- `EC2 + WebLogic / WebSphere`
  老系统上云时常见
- `ECS / Fargate + Spring Boot`
  现代项目较常见
- `ALB + ECS/EC2`
  对外提供负载均衡入口

### Batch 部署常见模式

- `EC2 + cron`
  最简单，适合传统 batch 搬到云上
- `EC2 + JP1 / Systemwalker`
  旧项目上云但保留原有作业管理思路
- `AWS Batch`
  云原生 batch 平台
- `EventBridge + Lambda`
  轻量定时任务
- `EventBridge + ECS Task`
  容器化 batch 很常见
- `Step Functions + Lambda / ECS`
  适合多步骤 batch 流程

### 对 Java 开发者要掌握的 AWS 相关知识点

- EC2 上如何部署 `jar` / `war`
- Tomcat、Nginx、ALB 的基本关系
- RDS 连接、连接串、权限与安全组
- S3 文件上传下载
- CloudWatch 日志查看和告警
- IAM Role 与最小权限
- Batch / Lambda / ECS 的触发方式
- CodeBuild / CodeDeploy / CodePipeline 的基本流程

### AWS 服务速查表

| AWS 服务名 | 中文解释 | 对日项目常见使用场景 |
|---|---|---|
| `EC2` | 云服务器 | 部署 `Tomcat`、`JBoss`、`WebLogic`、Java Web、Java batch |
| `RDS` | 托管数据库服务 | 承载 `Oracle`、`MySQL`、`PostgreSQL`、`SQL Server` 数据库 |
| `S3` | 对象存储 | 保存上传文件、日志归档、CSV 输出、批处理结果文件、备份文件 |
| `CloudWatch` | 监控与日志服务 | 查看应用日志、监控 batch 状态、设置 CPU / 错误率告警 |
| `IAM` | 权限管理服务 | 管理用户、角色、策略，控制应用和 batch 访问 AWS 资源 |
| `Systems Manager` / `SSM` | 运维管理服务 | 远程执行命令、参数管理、免 SSH 维护服务器 |
| `ELB` / `ALB` | 负载均衡服务 | 给 Java Web 系统提供访问入口，把流量分发到多个实例 |
| `Route 53` | DNS 服务 | 域名解析、环境切换、系统入口配置 |

## 19. 对日项目中经常遇到的现实情况

- 技术栈新旧并存，不能只会 Spring Boot
- 维护项目很多，老框架和老中间件必须看得懂
- Excel 设计书、测试项、帳票 Layout 非常重要
- 工具熟练度会直接影响工作效率
- 不少项目更重视规范、稳定性和可追溯性，而不是追求最新技术
- 履历里经常会同时出现“开发语言 + 中间件 + DB + 作业管理 + 帳票工具 + ETL 平台”
- 新项目越来越可能要求同时理解本地中间件和 AWS 云服务

## 22. 推荐学习路线

这一套更适合想进入对日 Java / batch / 运维类项目的人，按“先能跑起来，再逐步接近企业现场”的顺序学习。

### 第 1 阶段: Task Scheduler / cron

- 学习目标:
- 先理解“定时执行”最基本的概念
- 学会定时执行 `bat`、`sh`、`jar`
- 推荐掌握:
- Windows `Task Scheduler`
- Linux `cron`
- 你会得到:
- 知道 batch 是怎么被定时拉起的
- 能做最基础的本地 batch 练习

### 第 2 阶段: Quartz

- 学习目标:
- 在 Java / Spring 项目中真正写定时任务
- 理解 `Job`、`Trigger`、`Cron`、调度规则
- 推荐场景:
- Java 项目内部 batch
- 小型定时任务
- 你会得到:
- 知道 Java 代码层怎么实现 batch
- 能把调度逻辑写进 Spring 项目

### 第 3 阶段: Jenkins

- 学习目标:
- 学会用 Web UI 管理任务
- 学会用 Job / Pipeline 去跑脚本、Maven、batch
- 推荐场景:
- 公司已有 Jenkins
- 想快速实现“可视化调度 + 日志查看”
- 你会得到:
- 能从“开发者本地定时”进入“团队可管理执行”
- 对企业里的自动化执行流程有直观感受

### 第 4 阶段: Rundeck

- 学习目标:
- 学习更接近 `JP1` / `Systemwalker` 的作业管理思路
- 理解依赖关系、重跑、权限、日志、作业编排
- 推荐场景:
- 想学习日企现场常见的作业管理思维
- 想模拟ジョブネット管理
- 你会得到:
- 对“企业级 batch 管理平台”的感觉更接近实战
- 比 Jenkins 更像专门的作业调度工具

### 第 5 阶段: AWS Batch / EventBridge

- 学习目标:
- 学会 batch 上云后的部署和调度方式
- 理解 `AWS Batch`、`EventBridge`、`Lambda`、`ECS Task`
- 推荐场景:
- 新项目、云原生项目
- AWS 上的 Java / batch 系统
- 你会得到:
- 能看懂 AWS batch 项目履历
- 能把本地 batch 思路迁移到云平台

### 推荐顺序总结

| 阶段 | 工具 | 重点 |
|---|---|---|
| 1 | `Task Scheduler / cron` | 学会最基本的定时执行 |
| 2 | `Quartz` | 学会 Java 内部 batch 实现 |

| 5 | `AWS Batch / EventBridge` | 学会上云后的 batch 调度 |

### 如果你的目标是“对日项目就职 / 转职”

- 偏传统日企项目:
- `Task Scheduler / cron` -> `Quartz` -> `Jenkins` -> `Rundeck` -> 了解 `JP1`
- 偏云上新项目:
- `Task Scheduler / cron` -> `Quartz` -> `Jenkins` -> `AWS Batch / EventBridge`
- 偏 Java 开发:
- 先把 `Quartz`、`Jenkins`、`AWS Batch` 理清
- 偏运维 / batch 管理:
- 先把 `Task Scheduler`、`Rundeck`、`JP1` 思想理清
