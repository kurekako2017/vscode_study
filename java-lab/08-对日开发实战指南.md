# 对日开发实战指南

## 1. 对日项目不只是写代码

对日开发通常有几个明显特点:

- 重文档
- 重流程
- 重测试证据
- 重沟通准确性
- 技术栈可能偏保守

所以你学习 Java 时，必须同时学习项目协作方式。

## 2. 常见日语术语

| 日语术语 | 中文含义 | 英文常见说法 | 常见缩写/说明 |
| --- | --- | --- | --- |
| `設計書` | 设计书、设计文档 | `Design Document` / `Specification` | 泛指项目中的各种设计文档 |
| `基本設計` | 基本设计、概要设计、外部设计 | `Basic Design` / `External Design` | `BD`，也常叫 `ED`，常涉及 UI、画面、接口、输入输出、业务流程 |
| `詳細設計` | 详细设计、内部设计 | `Detailed Design` / `Internal Design` / `Low-Level Design` | `DD`，也常叫 `ID`，更偏程序内部逻辑、模块、表结构、处理细节 |
| `製造` | 编码实现 | `Implementation` / `Coding` | 日企项目里常把写代码称为制造 |
| `単体テスト` | 单体测试 | `Unit Test` | `UT` |
| `結合テスト` | 结合测试、集成测试 | `Integration Test` | `IT` |
| `総合テスト` | 综合测试、系统测试 | `System Test` | `ST` |
| `受入テスト` | 验收测试 | `Acceptance Test` / `User Acceptance Test` | `AT` / `UAT` |
| `障害票` | 缺陷单、故障单 | `Defect Ticket` / `Bug Ticket` / `Issue Ticket` | 用于记录和跟踪问题 |
| `改修` | 修改、修复、功能调整 | `Modification` / `Fix` / `Enhancement` | 既可能是修 Bug，也可能是小功能变更 |
| `調査` | 调查、问题分析 | `Investigation` / `Analysis` | 常见于问题定位、原因分析 |
| `保守運用` | 运维维护、保守运用 | `Operation and Maintenance` | 也可写作 `Maintenance and Operations` |

### 术语补充说明

- `基本設計` 通常更接近外部设计，关注用户看得见的部分和系统对外表现。
- `詳細設計` 通常更接近内部设计，关注程序如何实现、模块如何协作、数据如何处理。
- `単体テスト (UT)`、`結合テスト (IT)`、`総合テスト (ST)` 是对日项目中非常常见的测试阶段简称。

## 3. 常见职位简称

| 略称 | 日语/常见写法 | 中文含义 | 英文常见说法 | 常见职责 |
| --- | --- | --- | --- | --- |
| `PG` | `プログラマー` | 程序员 | `Programmer` | 负责详细设计后的编码、单体测试、缺陷修复 |
| `SE` | `システムエンジニア` | 系统工程师 | `System Engineer` | 负责设计、需求整理、评审、推进开发 |
| `BrSE` / `BSE` | `ブリッジSE` | 桥梁工程师、赴日桥梁 SE | `Bridge System Engineer` | 负责中日沟通、需求传达、说明设计和进度协调 |
| `PL` | `プロジェクトリーダー` | 项目组长 | `Project Leader` | 负责小组任务分配、进度跟进、成员管理 |
| `PM` | `プロジェクトマネージャー` | 项目经理 | `Project Manager` | 负责整体项目计划、资源、风险、对客沟通 |
| `TL` | `テックリード` | 技术负责人 | `Tech Lead` | 负责技术方案、代码质量、疑难问题处理 |
| `TLD` | `チームリーダー` | 团队负责人 | `Team Leader` | 和 `PL` 接近，部分项目会混用 |
| `LG` | `リーダー` / `リーダー格` | 组长、领导岗位 | `Leader` | 常指带人或带任务的人，不一定是正式职级 |
| `QA` | `品質保証` | 质量保证 | `Quality Assurance` | 负责测试质量、检查结果、质量管控 |
| `SA` | `システムアナリスト` | 系统分析师 | `System Analyst` | 偏上游分析、需求梳理、业务建模 |
| `BA` | `ビジネスアナリスト` | 业务分析师 | `Business Analyst` | 负责业务需求整理和用户沟通 |
| `OP` | `オペレーター` | 运维操作员 | `Operator` | 负责日常作业执行、监控、定时任务处理 |

### 职位简称补充说明

- `PG` 和 `SE` 是对日项目里最常见的两个基础角色。很多招聘里会直接写 `PG募集`、`SE募集`。
- `BrSE` 也常写成 `BSE`，实际就是桥梁 SE，通常要求更强的日语和沟通能力。
- `LG` 不是特别严格的全球统一缩写，很多现场里就是 `Leader` 或 `Leader格` 的简称，表示带队或准带队角色。
- `PL` 更偏执行层面的项目带队，`PM` 更偏整体管理和对外负责。

## 4. 常见设计书与资料名称

### 设计阶段常见文档

| 日语名称 | 中文名称 | 英文常见说法 | 常见略称/说明 |
| --- | --- | --- | --- |
| `設計書` | 设计书 | `Design Document` | 泛指设计资料总称 |
| `概要設計書` | 概要设计书 | `High-Level Design Document` | 有时会和 `BD` 放在同一层理解，但项目间叫法不完全一致 |
| `基本設計書` | 基本设计书 | `Basic Design Document` / `External Design Document` | `BD` / `ED`，偏外部设计 |
| `詳細設計書` | 详细设计书 | `Detailed Design Document` / `Internal Design Document` | `DD` / `ID`，偏内部设计 |
| `製造説明書` | 制造说明书、实现说明书 | `Implementation Guide` / `Coding Instruction` | 说明具体开发实现方式 |
| `機能設計書` | 功能设计书 | `Functional Design Document` | 描述功能需求与处理逻辑 |
| `非機能要件一覧` | 非功能需求一览 | `Non-Functional Requirements List` | 包括性能、安全、可用性等 |
| `処理フロー図一覧` | 处理流程图一览 | `Process Flow List` | 汇总各处理流程图 |
| `クラス図・処理フロー図` | 类图、处理流程图 | `Class Diagram / Process Flow Diagram` | 常用于详细设计阶段 |
| `シーケンス図` | 时序图 | `Sequence Diagram` | 表示对象或服务调用顺序 |
| `状態遷移一覧` | 状态迁移一览 | `State Transition List` | 描述状态变化规则 |
| `例外設計一覧` | 异常设计一览 | `Exception Design List` | 汇总异常处理方式 |

## 5. 常见工作内容

- 根据设计书实现画面或批处理
- 按测试项执行单体测试
- 修复缺陷
- 调查生产问题
- 修改 SQL 或配置
- 输出作业报告

## 7. 初级工程师最容易踩的坑

- 只会写语法，不会看项目结构
- 不会看日志，不会查数据库
- 改动没有考虑影响范围
- 没有按设计书和测试项验证
- 不理解日文字段名和业务术语
- 导入项目后编码错乱

## 8. 建议补充的非代码能力

- Excel 熟练度
- 日语 IT 词汇积累
- Bug 汇报表达
- 作业进度汇报
- 设计书阅读能力

## 9. 推荐的学习实践方式

### 练习 1

手工搭建一个 Tomcat + JSP + Servlet + JDBC 登录项目。

### 练习 2

使用 Spring Boot + MyBatis 完成用户管理和分页查询。

### 练习 3

模拟对日项目:

- 写一份简单画面设计书
- 按设计书编码
- 写单体测试项
- 修复自己制造的一个缺陷

## 10. 最终能力目标

完成本资料库后，你应该能做到:

- 独立搭建 Java 开发环境
- 看懂传统 Java Web 项目
- 开发基础 Spring Boot 项目
- 使用数据库工具查数和改数
- 理解对日项目常见流程与工具
- 能作为初中级工程师进入 Java 对日项目并持续成长
