## 1. 案件理解

### 1.1 案件概要

> 各サービスごとのデータ連携開発を共通化するための共通IFシステムです。  
> 各サービスに対して、他システムへの連携、および連携状況を管理する共通機能を提供する位置づけです。  
> 具体的には、Kafka受信機能、IF仕様に従ったAPI / ファイル連携、ファイル取込などが主な機能になります。

### 1.2 技术要素

- バッチ開発
- Java / Spring Boot / MySQL
- Kafka
- Azure Service Bus
- Azure Red Hat OpenShift
- BLOB Storage
- JSON操作
- JQコマンド
- JSONPath
- B-Shell

### 1.3 面试中要强调的方向

- Java / Spring Bootでのバッチ・API開発経験
- IF仕様書に基づくデータマッピング、データ変換、チェック処理
- Kafka / Service Bus など非同期連携の基本理解
- BLOB Storage / ファイル取込 / ファイル連携の経験に近い説明
- JSON、JQ、JSONPathを使ったデータ確認・調査
- 障害時のログ確認、原因切り分け、再実行・リトライの考え方
- FWを設計・使用するためのレイヤー分割、共通化、保守性の考え方

## 4. 想定面试问题：案件概要 / 担当内容

### Q1. この案件をどのように理解していますか

> 各サービスごとのデータ連携を共通化するための共通IFシステムだと理解しております。  
> 各サービスから共通機能を利用して、他システムへの連携、連携状況の管理、Kafka受信、API連携、ファイル取込などを行う仕組みだと認識しております。  
> そのため、Java / Spring Bootでのバッチ開発だけではなく、IF仕様、データマッピング、エラーハンドリング、ログ確認が重要になる案件だと考えております。

### Q2. 共通IFシステムとはどのようなシステムですか

> 個別サービスごとに重複して実装されがちなデータ連携処理を共通化するシステムです。  
> 例えば、ファイル取込、API連携、メッセージ受信、連携結果の管理、エラー時のログ出力や再処理などを共通機能として提供します。  
> 共通化することで、各サービス側の実装負荷を下げ、保守性や運用性を高めることができると理解しております。

### Q3. この案件で活かせる経験は何ですか

> Java / Spring Bootを用いたWebアプリケーション開発、バッチ処理、SQL調査、API連携、ファイル連携の経験が活かせると考えております。  
> また、SAP移行案件やJava / AWS移行案件で、データマッピング、ファイル取込、ログ確認、不整合調査、障害切り分けを行った経験がございます。  
> 今回の案件でも、IF仕様を確認しながら、データ連携処理を安全に実装・テストしていきたいと考えております。

### Q4. FWを設計・使用するための考え方とは何だと思いますか

> 共通処理と個別処理を適切に分けることが重要だと考えております。  
> 例えば、ログ出力、エラーハンドリング、入力チェック、リトライ、ファイル操作、DBアクセスなどは共通化しやすい部分です。  
> 一方で、サービスごとに異なる変換ルールや業務チェックは個別処理として分離します。  
> 保守性、拡張性、テストしやすさを意識して設計することが大切だと考えております。

## 5. 想定面试问题：Java / Spring Boot / MySQL

### Q1. Spring Bootでは何を担当しましたか

> Spring Bootでは主にREST API開発、Webアプリケーション改修、バッチ処理を担当しておりました。  
> Controller、Service、Repository構成で実装し、JPAやMyBatisを利用した経験がございます。  
> また、ファイル取込、ログ調査、不具合対応、SQL確認なども行っておりました。

### Q2. バッチ開発では何を意識しますか

> 処理件数、実行時間、エラー時の再実行、ログ出力、トランザクション管理を意識します。  
> 大量データを扱う場合は、メモリ使用量やコミット単位、リトライ方法、途中で失敗した場合の再処理方法も重要です。  
> 運用時に原因調査しやすいように、開始・終了ログ、件数ログ、エラー内容を分かりやすく出力することを大切にしております。

### Q3. MySQLでは何を確認しますか

> テーブル定義、インデックス、JOIN条件、検索条件、更新対象件数を確認します。  
> 不具合調査では、画面や連携データ、アプリケーションログ、DBデータを突き合わせて原因を切り分けます。  
> 性能面では、WHERE条件、インデックス利用、不要な全件検索がないかを確認します。

### Q4. Repository層では何をしますか

> Repository層は、主にDBアクセスを担当する層です。  
> Service層から呼び出され、SQLやO/Rマッパーを利用してデータの検索、登録、更新、削除を行います。  
> 業務ロジックはService側に置き、RepositoryではDB操作を分離するように意識しておりました。

### Q5. テストでは何を確認しますか

> 正常系、異常系、境界値、必須チェック、データ変換結果、DB更新結果、ログ出力、エラー時の挙動を確認します。  
> IF連携では、入力データ、出力データ、連携先への送信内容、リトライやエラー時の状態管理も確認します。

## 6. 想定面试问题：Kafka / Azure Service Bus / BLOB Storage

### Q1. KafkaやAzure Service Busについて説明できますか

> KafkaやAzure Service Busは、システム間で非同期にメッセージを連携するための仕組みだと理解しております。  
> 送信側と受信側を疎結合にできるため、リアルタイム連携や大量データ連携、障害時のリトライなどに利用されます。  
> Kafkaは大量イベント処理やストリーム処理に強く、Azure Service BusはAzure上のメッセージングサービスとしてキューやトピック連携に利用されるものと認識しております。

### Q2. Kafka受信機能では何を意識しますか

> 受信したメッセージの形式、必須項目、重複受信、処理順序、エラー時の扱い、再処理方法を意識します。  
> また、どこまで処理が完了したかをログやステータスで確認できることが重要です。  
> 障害時には、メッセージ内容、受信時刻、処理結果、エラー内容を確認して原因を切り分けます。

### Q3. メッセージ連携でエラーが出た場合、どう調査しますか

> まず対象メッセージ、発生時刻、エラー内容、処理ステータスを確認します。  
> 次に、メッセージ形式、JSON項目、必須値、コード値、連携先の状態、ネットワークや認証情報を確認します。  
> アプリケーションログと連携データを突き合わせ、原因を切り分けた上で、再処理やデータ修正の要否を判断します。

### Q4. BLOB Storageはどのように使われますか

> BLOB Storageは、ファイルやJSONデータなどを保存するオブジェクトストレージとして利用されます。  
> 連携ファイルを一時的に配置したり、バッチ処理やデータ取込の入力ファイルとして利用するイメージです。  
> ファイル名、配置パス、権限、処理済みファイルの扱い、エラー時の退避先などを設計時に確認する必要があります。

### Q5. Azure Red Hat OpenShiftについて知っていますか

> Azure Red Hat OpenShiftは、Azure上で利用できるOpenShiftのマネージドサービスだと理解しております。  
> Kubernetesをベースにしたコンテナ実行基盤で、アプリケーションのデプロイ、スケーリング、ログ確認などに利用されます。  
> 深い基盤設計はこれからキャッチアップする部分もありますが、開発・保守の立場でログ確認や動作確認を行うことは対応可能です。

## 7. 想定面试问题：JSON / JQ / JSONPath / B-Shell

### Q1. JSON処理では何を意識しますか

> JSONの項目名、必須項目、型、階層構造を確認し、IF仕様書と実データの差異がないかを意識します。  
> 配列構造やネストが深い項目では、どの階層の値を利用するのかを明確にします。  
> 不正データが入った場合に、エラーメッセージやログで原因を追跡できることも重要です。

### Q2. JQコマンドやJSONPathは何に使いますか

> JQコマンドやJSONPathは、JSONデータから必要な項目を抽出したり、特定条件のデータを確認したりするために利用します。  
> 例えば、連携データの中から特定項目だけを確認する、配列内の値を確認する、ログに出力されたJSONの内容を調査するといった場面で利用できます。

### Q3. シェル作成経験はありますか

> B-Shellを利用した簡単なシェル作成や、既存シェルの確認・修正を行った経験がございます。  
> 主にファイル操作、コマンド実行、ログ出力、異常時の確認など、運用やデータ連携に関わる処理で利用しておりました。  
> 深いシェル開発は案件に合わせてキャッチアップしながら対応可能です。

### Q4. シェルで注意することは何ですか

> 引数チェック、戻り値確認、ログ出力、異常終了時の処理、ファイル存在チェック、権限、文字コードを意識します。  
> また、本番環境で実行する場合は、対象ファイルや対象ディレクトリを間違えないように、事前確認と実行後確認を行うことが重要です。

## 8. 数据移行 / マッピング / 障害対応

### Q1. データ移行やデータ連携の流れを説明してください

> まず連携対象データと連携先の項目を確認し、IF仕様書をもとにマッピングと変換ルールを整理します。  
> 次に、入力データの取得、必須チェック、形式チェック、データ変換、連携処理、結果確認、エラー時のログ出力や再処理を行います。  
> テストでは、正常系だけでなく、項目不足、型不一致、コード値不正、連携先エラーなどの異常系も確認します。

### Q2. マッピング作成では何を確認しますか

> 現行項目と連携先項目の対応関係、データ型、桁数、必須項目、コード値、初期値、変換ルールを確認します。  
> また、連携しない項目、固定値を設定する項目、条件によって変換が変わる項目も整理します。  
> 不明点がある場合は、設計書、現行データ、業務仕様を確認し、必要に応じて業務側に確認します。

### Q3. 連携エラーが発生した場合、どう対応しますか

> まずエラー内容、対象データ、発生件数、発生タイミングを確認します。  
> 次に、必須項目不足、型不一致、桁数超過、コード値不正、変換ルール誤り、連携先エラーなどの観点で原因を切り分けます。  
> 原因を特定した後、データ修正、マッピング修正、ロジック修正、再実行の手順を整理して対応します。

### Q4. 本番障害や事故が発生した場合、どう対応しますか

> まず影響範囲、発生時刻、対象サービス、業務影響を確認し、関係者へ共有します。  
> 次に、ログ、監視情報、直近のリリース内容、DB状態、連携データを確認し、原因を切り分けます。  
> 必要に応じて暫定対応で業務影響を抑え、その後、恒久対応と再発防止策を整理します。

### Q5. 課題管理はどのように進めますか

> 課題内容、担当者、期限、優先度、影響範囲、対応状況を一覧で管理します。  
> 定例や進捗確認の場で未解決課題を確認し、ブロッカーになっているものは早めにエスカレーションします。  
> 解決後は、対応内容と決定事項を残し、同じ問題が再発しないようにします。
