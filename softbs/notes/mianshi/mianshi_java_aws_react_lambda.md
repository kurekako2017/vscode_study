## 3. 開発環境の説明

### 3.1 基本の構成

> 開発環境は、WSL 上の Linux 環境をベースに、VS Code と Eclipse を使い分ける構成を想定しておりました。  
> Java 側は WSL 上の JDK と Maven を使い、Python 側は WSL 上の venv を使って Lambda の実装や検証を行うイメージです。  
> フロント側は Node.js を WSL で動かし、React / TypeScript の開発やビルドを行います。  
> 日常の編集は VS Code、Java の確認やデバッグは Eclipse を使うと効率がよいと考えております。

### 3.2 WSL を使う理由

> WSL を使う理由は、Linux 本番環境に近い形で Java、Python、Node.js、Docker の動作確認ができるためです。  
> Windows 側と Linux 側で依存関係がずれにくく、npm や Python の仮想環境、シェルスクリプトの扱いも安定します。  
> 私自身も、開発時は WSL 上で作業し、必要に応じて VS Code Remote - WSL や Eclipse から接続して確認する形が分かりやすいと考えております。

### 3.3 Java / Python / Node.js の役割分担

> Java はメインの業務ロジックや API、バッチ処理を担当し、Python は Lambda の軽量処理や補助的なスクリプトで使うイメージです。  
> React / TypeScript はブラウザ側の画面表示とユーザー操作を担当し、API 呼び出しはフロントからバックエンドへ送る構成です。  
> それぞれを WSL ベースで揃えることで、ローカルと本番の差を小さくできます。

## 4. AWS / Docker / Lambda の説明

### 4.1 Docker をどう使うか

> Docker は、開発環境をコンテナ化して、Java、Python、Node.js、DB などのバージョン差異を吸収するために使います。  
> ローカルでは Docker Compose で必要なサービスをまとめて起動し、アプリ本体と依存サービスを近い構成で確認できます。  
> 私は、Docker は本番と同じ動作確認をしやすくするための基盤だと理解しており、開発環境の統一や再現性の確保に有効だと考えております。

### 4.2 AWS への配置イメージ

> AWS では、画面や API、バッチ、Lambda をそれぞれの役割に応じて配置し、S3 や Secrets Manager、CloudWatch を組み合わせて運用するイメージです。  
> 例えば、設定値や接続情報は Secrets Manager で管理し、ファイル連携は S3、ログ確認は CloudWatch、常時動かす処理は ECS、イベント駆動の短い処理は Lambda という使い分けになります。  
> EC2 は必要に応じて補助的に使う構成と理解しております。

### 4.3 Lambda のデプロイ説明

> Lambda は、関数単位でコードをアップロードし、トリガーとハンドラを設定して動かします。  
> Python であれば依存ライブラリを含めてパッケージ化し、Java であればビルドした成果物をまとめてデプロイする形になります。  
> 実務では、S3 連携や EventBridge、API 経由で起動することが多いと理解しております。  
> デプロイ後は CloudWatch Logs を見て、入力イベント、処理結果、例外内容を確認します。

### 4.4 どう説明すると面接で通りやすいか

> AWS は「何をどこに置くか」と「どう監視するか」をセットで説明すると分かりやすいと考えております。  
> 例えば、フロントは React、バックエンドは Java、軽量なイベント処理は Lambda、ファイル保管は S3、設定情報は Secrets Manager、ログ確認は CloudWatch、コンテナ実行は Docker / ECS という整理です。  
> その上で、障害時にはログとイベント内容を見て、どの層で止まったかを切り分ける、という説明がしやすいです。

### 4.5 Vite の frontend を AWS でどう動かすか

> Vite はローカル開発用の開発サーバーなので、本番の AWS では通常そのまま Vite サーバーを常駐させるのではなく、`npm run build` で静的ファイルを作って配信します。  
> 必要なのは、AWS 上でビルドするなら Node.js / npm、配信するなら Nginx、Apache、S3 + CloudFront、または ECS 上の Web サーバーです。  
> つまり、本番で必要なのは Vite 自体ではなく、Vite で生成した `dist` をどう配信するかです。

> 代表的には、AWS 上で `npm ci && npm run build` を実行して静的ファイルを作り、Nginx で配信するか、S3 に配置して CloudFront 経由で公開する形です。  
> もし EC2 や ECS 上で起動する場合でも、Vite の開発サーバーを立ち上げるというよりは、ビルド済み成果物を静的配信する構成が一般的だと理解しております。

## 5. React / TypeScript の説明

### 5.1 React の役割

> React は画面をコンポーネント単位で組み立てるフロントエンド技術で、画面表示、入力、API 呼び出し結果の反映を担当します。  
> バックエンドが API を提供し、React がそのデータを受け取って画面に描画する、という分担になります。  
> 私は一部画面改修の経験があり、コンポーネント分割、状態管理、API 呼び出しの流れを意識して対応しておりました。

### 5.2 TypeScript と組み合わせる意味

> TypeScript を使うと、props、state、API レスポンスの型を明示できるため、画面と API の受け渡しを安全にしやすくなります。  
> React では、コンポーネントの引数や戻り値に型を付けることで、実装時に誤りを早めに見つけられます。  
> そのため、React と TypeScript は相性がよく、チーム開発でも保守しやすい構成だと理解しております。

### 5.3 面接での説明例

> React 側では、画面を部品化して、一覧、詳細、入力フォームなどをコンポーネントとして分けます。  
> TypeScript では、API の戻り値や画面状態に型を付けることで、実装時の確認をしやすくします。  
> その結果、バックエンドとの JSON 形式のやり取りを安定させやすいと考えております。

### 5.4 もし深い経験を問われたら

> React / TypeScript は一部実務経験がございますが、全体設計や複雑な状態管理はまだ学習を継続している部分もございます。  
> ただし、GitHub 上のサンプルや公式ドキュメントで継続的に確認しており、案件のコードベースに合わせてキャッチアップ可能です。

### 5.5 React の分層と役割分担

> React 側は、上から順に UI コンポーネント層、Page / View 層、State Hook 層、Service 層、API Client 層のように分けて考えると整理しやすいです。  
> UI コンポーネント層はボタン、フォーム、商品カード、カート一覧などの見た目とイベント発火を担当します。  
> View 層は複数の UI コンポーネントを組み合わせて1画面を作り、画面全体のレイアウトや子コンポーネントへの値の受け渡しを行います。  
> State Hook 層はログイン状態、商品一覧、カート情報のような画面共通の状態を持ち、更新処理をまとめます。  
> Service 層は「初期データ取得」「ログイン」「保存」などの業務操作をまとめ、API 呼び出しの流れを整理します。  
> API Client 層は fetch や axios などで API base url、JSON header、cookie、エラー処理を統一します。

> その下で、Spring Boot 側は Controller、Service、DAO に分かれます。  
> React は表示と操作、Spring Boot は認証、業務ロジック、DB 更新を担当する、と分けて説明すると分かりやすいです。  
> つまり、React は「画面をどう見せるか」と「どう操作させるか」を担当し、Spring Boot は「何を処理するか」と「何を保存するか」を担当する、という整理です。

## 6. 前後端の連携説明

### 6.1 どう連携するか

> 前後端は、基本的にバックエンドが API を提供し、React がその API を呼び出して画面に反映する形です。  
> 画面の入力値はフロントで基本チェックを行い、最終的な業務チェックや DB 更新はバックエンド側で行います。  
> そのため、API の項目名、必須項目、エラー時の返却形式を事前に揃えることが重要です。

### 6.2 どこを意識するか

> フロント側では、ユーザーにとって分かりやすい表示、ボタン制御、入力チェック、ローディング表示を意識します。  
> バックエンド側では、入力値の検証、業務ロジック、DB 更新、例外処理、ログ出力を意識します。  
> 両者で API 仕様を合わせ、項目の追加や変更があれば、影響範囲を確認しながら調整することが大切です。

### 6.3 実務での言い方

> 私は、画面と API のつなぎ込みでは、仕様書だけでなく実データやログも見ながら、どの項目がどこで作られているかを確認していました。  
> そのため、フロントとバックエンドのどちらか一方だけではなく、両方の流れを見ながら調整することを意識しております。
