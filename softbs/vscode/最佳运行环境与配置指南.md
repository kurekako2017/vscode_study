# 最佳运行环境与配置指南

这份文档用于说明当前工作区 `vscode_study` 的推荐开发方式。

你的主要使用方式是：

```text
Windows 11 + WSL2 Ubuntu + VS Code Remote - WSL
```

辅助使用方式是：

```text
Windows 11 + WSL2 Ubuntu + IntelliJ IDEA / IDEA 连接 WSL 工具链
```

核心原则：

- 长期开发主目录放在 WSL Linux 文件系统：`/home/victorkure/workspace/vscode_study`
- Windows 侧 `D:\dev\source_code\vscode_study` 只作为备份、查看、迁移或临时编辑入口。
- 需要安装依赖、运行服务、启动前端、跑测试时，优先在 WSL 里执行。
- VS Code 是主力编辑器，IDEA 主要用于 Java / Spring Boot 项目。

## 1. 推荐环境结论

| 使用场景 | 推荐方式 | 原因 |
| --- | --- | --- |
| 日常学习和文档整理 | VS Code + Remote - WSL | 打开快，适合整个工作区、多语言、多文档 |
| TypeScript / React / Vue / Next.js | VS Code + WSL 终端 | Node、npm、pnpm 在 Linux 下更稳定 |
| Python / LLM / Agent 示例 | VS Code + WSL 终端 | venv、pip、命令行工具更接近 Linux 生产环境 |
| Java / Spring Boot | VS Code 或 IDEA + WSL JDK/Maven | IDEA 调试体验好，WSL 工具链更统一 |
| Docker / LocalStack / DevOps | WSL 终端 + Docker Desktop WSL2 backend | Docker 与 Linux 工具链配合更自然 |
| 只看文档或轻量修改 | Windows VS Code 直接打开 D 盘也可以 | 不运行依赖时问题较少 |

一句话：

```text
能在 WSL 里跑，就不要在 D 盘挂载路径里跑。
```

## 3. 最佳日常启动流程

### 方式 A：WSL 终端启动 VS Code

这是最推荐的方式。

```bash
cd /home/victorkure/workspace/vscode_study
code .
```

打开后确认：

- VS Code 左下角显示 `WSL: Ubuntu`
- 终端默认是 WSL shell
- `pwd` 显示 `/home/victorkure/workspace/vscode_study`

### 方式 B：VS Code 图形界面进入 WSL

1. 在 Windows 打开 VS Code。
2. 按 `Ctrl+Shift+P`。
3. 执行 `Remote-WSL: New Window`。
4. 选择 `File -> Open Folder...`。
5. 输入：

```text
/home/victorkure/workspace/vscode_study
```

### 方式 C：脚本启动

如果仓库里已有启动脚本，可以在 Windows 侧执行：

```powershell
scripts\wsl\start-wsl-vscode-dev.bat
```

用途：

- 自动进入 WSL
- 自动切到工作区目录
- 自动用 VS Code Remote - WSL 打开

## 4. VS Code + WSL 配置

### 必装扩展

建议在 Remote - WSL 环境中安装或启用：

| 扩展 | 用途 |
| --- | --- |
| Remote - WSL | 连接 WSL 工作区 |
| Python | Python 运行和调试 |
| Pylance | Python 类型提示 |
| ESLint | JavaScript / TypeScript 代码检查 |
| Prettier | 前端格式化 |
| Docker | Dockerfile / compose 辅助 |
| Extension Pack for Java | Java 基础支持 |
| Spring Boot Extension Pack | Spring Boot 项目辅助 |

### 推荐 VS Code 使用习惯

- 只打开一个主工作区：`/home/victorkure/workspace/vscode_study`
- 在 VS Code 内置终端执行项目命令
- 前端项目依赖安装在 WSL 中的项目目录
- Python 虚拟环境放在项目内 `.venv`
- Java 项目优先使用 WSL 中的 JDK / Maven

### 终端确认命令

```bash
pwd
which node
which npm
which python3
which java
which mvn
```

如果路径都在 Linux 环境，例如 `/usr/bin`、`/home/...`，说明当前工具链在 WSL 中。

## 5. IDEA + WSL 使用方式

IDEA 适合作为 Java / Spring Boot 的辅助主力工具，尤其是：

- Spring Boot 调试
- Maven 项目管理
- 断点调试
- 数据库插件
- Java 重构和代码导航

### 推荐打开方式

优先在 IDEA 中打开 WSL 路径：

```text
\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study
```

如果只开发某个 Java 项目，也可以打开具体项目：

```text
\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study\java-projects\JtProject
```

### IDEA 中建议配置

| 配置项 | 推荐值 |
| --- | --- |
| Project SDK | WSL 中的 JDK |
| Maven | WSL 中的 Maven 或项目自带 `mvnw` |
| Terminal | WSL bash |
| Git | WSL Git 或保持同一环境的 Git |
| Line Separator | LF |

### Java / Spring Boot 建议

如果项目带有 `mvnw`，优先使用：

```bash
./mvnw test
./mvnw spring-boot:run
```

如果需要指定 JDK：

```bash
JAVA_HOME=/home/victorkure/.jdks/ms-25.0.3 ./mvnw test
```

注意：

- 不要一会儿用 Windows Maven，一会儿用 WSL Maven。
- 不要让 IDEA 在 Windows D 盘和 WSL home 两边同时改同一个项目。
- 如果 IDEA 打开的是 WSL 路径，终端和构建工具也尽量使用 WSL。

## 8. Windows 与 WSL 同步策略

### 推荐原则

只设一个主目录：

```text
主目录：/home/victorkure/workspace/vscode_study
镜像目录：D:\dev\source_code\vscode_study
```

日常开发只改主目录。

### 从 WSL 同步到 Windows

在 Windows PowerShell 中执行：

```powershell
robocopy "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" "D:\dev\source_code\vscode_study" /MIR /XD .git node_modules .venv target .next dist build
```

用途：

- 把 WSL 主目录镜像到 D 盘
- 给 Windows 工具查看
- 做本地备份

### 从 Windows 回灌到 WSL

只有确认 D 盘才是最新时才执行：

```powershell
robocopy "D:\dev\source_code\vscode_study" "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" /MIR /XD .git node_modules .venv target .next dist build
```

注意：

- `/MIR` 会删除目标目录中源目录没有的文件。
- 不确定时先把 `/MIR` 改成 `/E`。
- 不要在两边同时改同一批文件。

### 更推荐的同步方式

长期来说，优先用 Git：

```bash
git status
git add .
git commit -m "update docs"
git push
```

然后另一边用：

```bash
git pull
```

## 9. 什么时候可以用 Windows 原生

可以用 Windows 原生 VS Code / PowerShell / Git Bash 的场景：

- 只看文档
- 只改 README
- 只做 Git 查看
- 运行 Windows 专用脚本
- 临时查看 D 盘镜像目录

不建议用 Windows 原生的场景：

- `npm install`
- 大量前端构建
- Python venv + native package
- Docker / LocalStack 相关学习
- Java 项目需要和 Linux 部署环境保持一致时

## 10. Git 与行尾设置

推荐在 WSL 主目录内做 Git 操作。

检查：

```bash
git config --get core.autocrlf
git status
```

建议：

- WSL 中保持 LF 行尾。
- 不要在 Windows 和 WSL 用不同 Git 设置反复改同一文件。
- 如果某些文件反复出现行尾变化，检查 `.gitattributes`。

## 11. 常见问题排查

### VS Code 左下角没有显示 WSL

可能原因：

- 打开的是 Windows 本地文件夹
- 没有通过 Remote - WSL 进入

处理：

```bash
cd /home/victorkure/workspace/vscode_study
code .
```

### `code .` 不可用

处理：

1. 在 Windows 打开 VS Code。
2. 安装 `Remote - WSL`。
3. 执行 `Remote-WSL: New Window`。
4. 回到 WSL 终端重试 `code .`。

### `npm install` 很慢或出错

先确认是不是在 `/mnt/d` 下运行。

```bash
pwd
```

如果路径是 `/mnt/d/...`，建议迁移到：

```bash
/home/victorkure/workspace/vscode_study
```

### IDEA 找不到 JDK / Maven

处理方向：

- 确认 IDEA 打开的是 WSL 路径。
- Project SDK 选择 WSL 中的 JDK。
- Maven 优先使用项目自带 `mvnw`。
- Terminal 设置为 WSL bash。

### Docker 命令不可用

检查：

```bash
docker version
docker ps
```

如果不可用：

- 确认 Docker Desktop 已启动。
- 确认 Docker Desktop 启用了 WSL2 backend。
- 确认 Docker Desktop 的 WSL Integration 开启了 Ubuntu。
