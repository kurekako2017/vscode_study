## 工作区统一环境方案

这个工作区里的非 Java 示例，建议统一采用同一套运行方式：长期开发放在 /home/victorkure/workspace/vscode_study，实际执行时始终通过 WSL 访问这份 Linux 本地文件系统里的代码。

这套方案的核心是把代码和运行环境一起切到 Linux 里。Windows 侧保留原始仓库或备份视图，WSL 侧负责执行 Python、Node、React、Docker 相关命令，VS Code 通过 Remote - WSL 连接到同一个工作区。

具体配置原则如下：

- 代码常驻在 WSL：长期开发副本放在 /home/victorkure/workspace/vscode_study，减少挂载盘 I/O 和路径差异问题。
- 迁移可回看：Windows 上的 D 盘仓库可作为初始源或临时对照，但不作为长期运行主目录。
- 运行在 WSL：Python、Node.js、npm/pnpm、测试命令、前端开发服务器都在 WSL 终端里执行。
- 编辑在 VS Code：用 Remote - WSL 打开 Linux 文件系统中的目录，直接编辑和运行，避免 Windows 与 Linux 两套工具链混用。
- 依赖按项目安装：Python 示例用项目内虚拟环境，Node/React 示例用项目内 node_modules，Docker 只在需要时启用。
- 同步靠 Git：如果把仓库放到 /home/...，Windows 和 WSL 不再共享同一份底层文件，改动通过 Git push/pull 或定期同步完成。

适合这套方案的场景：

- 需要运行 Python 示例、Node 示例、React 示例、前端构建、脚本工具。
- 需要保证路径、换行符、权限、包管理器行为尽量接近 Linux 生产环境。
- 希望长期稳定开发，不想每次都在挂载盘和本地 Linux 文件系统之间切换。

不适合继续坚持 Windows 原生环境的场景：

- 只有少量纯文档修改，或者只运行 Windows 专用工具。
- 你所在环境完全不能启用 WSL。
- 某个示例明确依赖 Windows API 或 Windows 原生软件。

为什么选择 WSL2（简要理由）

- 与生产 Linux 环境一致：Node / Python / 包管理和原生工具在 Linux 下行为更稳定。
- 兼容性最好：多数示例（特别是 Node/React、npm install、native 模块编译、Docker/DevContainer）在 WSL 环境更少问题。
- 开发体验佳：VS Code Remote - WSL 允许在 Linux 环境中直接打开工作区、安装扩展并使用本地终端。
- 与 Docker 集成顺畅：Docker Desktop 的 WSL2 后端可直接在 WSL 与 Windows 之间无缝工作。

何时考虑替代方案

- 使用 Windows 原生（VS Code + Git Bash）适合：仅运行简单脚本、工具必须是 Windows 原生程序，或你无法启用 WSL（如公司策略）。
- 纯 Windows Git Bash：方便做 Git 操作与简单命令行任务，但对 node 原生模块或需要 Linux 工具链的项目不稳定。

快速配置步骤（推荐顺序）

1) 启用并安装 WSL2（Windows 11）

```powershell
# 以管理员 PowerShell 运行（Windows 11）
wsl --install -d Ubuntu
# 或手动启用（若需要）：
# dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
# dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
# 重启后在 PowerShell 中运行：
wsl --set-default-version 2
```

2) 打开 WSL（第一次启动配置 Ubuntu），更新并安装常用工具

```bash
# 在 WSL Ubuntu 中运行
sudo apt update && sudo apt upgrade -y
sudo apt install -y build-essential git curl ca-certificates
```

补充说明：如何启动 WSL / Ubuntu

- Windows 11 上最常见的方式是按 `Win` 键，搜索并打开 `Ubuntu`。
- 也可以在 PowerShell 或 Windows Terminal 里执行 `wsl`，如果想指定发行版则用 `wsl -d Ubuntu`。
- 第一次启动 Ubuntu 时，会让你创建 Linux 用户名和密码；完成后就可以直接进入 WSL 终端。
- 如果你安装了多个发行版，`wsl -l -v` 可以先看当前有哪些发行版，再决定进入哪个。

3) 安装 Python 与虚拟环境

```bash
sudo apt install -y python3 python3-venv python3-pip
python3 -m pip install --user pipx
# 可选：设置 pip 使用用户目录或全局 venv
```

4) 安装 Node.js（建议通过 nvm 管理）

```bash
# 安装 nvm（按官方说明）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.5/install.sh | bash
# 重新加载 shell 后安装 Node
nvm install --lts
nvm use --lts
# 安装常用包管理器（可选）
npm install -g pnpm
```

5) 安装并配置 Docker（可选但推荐用于容器化示例）

- 在 Windows 上安装 Docker Desktop，打开 WSL2 后端并启用对 WSL 发行版的整合。
- 在 WSL 中确认 docker 可用：

```bash
wsl -d Ubuntu
docker version
```

6) 在 VS Code 中安装并使用 Remote - WSL

- 在 Windows 上安装 VS Code（Stable）。
- 安装扩展：`Remote - WSL`、`Python`、`Pylance`、`Pyright`、`ESLint`、`Prettier`、`ES7 React/Redux/GraphQL`、`Debugger for Chrome`、`Docker`。
- 在 VS Code 里：`Remote-WSL: New Window`，然后用 `File > Open Folder` 打开 WSL 中的长期开发目录 `/home/victorkure/workspace/vscode_study`。

7) 这个工作区里各类示例怎么执行

- Python 示例：在对应目录创建 `.venv`，安装依赖后运行脚本或启动调试。
- Node / React 示例：在对应目录用 npm、pnpm 或 yarn 安装依赖，然后启动开发服务器或构建任务。
- 文档与说明页：可直接在 VS Code 中编辑，但仍建议保持 Remote - WSL 打开的同一工作区，避免切换环境。
- Docker 示例：通过 Docker Desktop 的 WSL2 后端运行，命令在 WSL 终端里执行。
- 这些示例都不要求移动目录，只要求在正确的终端环境中运行。

8) 推荐的项目运行示例（在 WSL 终端）

- Python 导入虚拟环境并运行：

```bash
cd /home/victorkure/workspace/vscode_study/python-projects/ai-lab
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python example.py
```

- React / Node 示例：

```bash
cd /home/victorkure/workspace/vscode_study/web-projects/my-react-app
pnpm install
pnpm start
# 或 npm install && npm start
```

8) Git 与 Git Bash

- 推荐在 WSL 内使用 `git` 做代码操作以保证文件权限与 LF 行尾一致性；Windows 上可保留 Git Bash 用于快速本地命令或在 Windows 主机执行脚本。

9) VS Code 设置建议（settings.json 片段）

```json
{
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "python.defaultInterpreterPath": "/usr/bin/python3",
  "editor.formatOnSave": true

```

安全与注意事项

- WSL 与 Windows 文件系统互访问，但长期在 Windows 文件系统（/mnt/c /mnt/d）上运行大量 node_modules 可能性能较差，建议在 WSL 的家目录下克隆或 `rsync` 项目以获得最佳 I/O 性能，或将项目挂载到 WSL 的 Linux 文件系统中。
- 权限与行尾：在跨系统开发时注意 `.gitattributes` 与 `core.autocrlf` 的设置以避免不必要的差异。
- 若需要 GPU 加速（例如某些 ML 本地测试），Windows 原生或 WSLg 的 GPU 支持设置复杂，视硬件与驱动而定，可能需要额外配置（可另行提供指南）。

总结建议（一句话）

- 最佳方案：使用 WSL2 + VS Code Remote - WSL，在 WSL 内安装 Python / Node / Docker，平衡兼容性与开发体验；把 Windows Git Bash 留作主机端快捷命令与 Git 操作备用。

## Terminal 与 Git Bash 使用指南

1) 什么时候用哪个终端

- 在 Remote‑WSL 会话中：优先使用 WSL 的 shell（`bash` / `zsh`），所有开发、安装依赖、运行 dev server、构建等操作都在 WSL 里进行。  
- 在 Windows 主机上：用于轻量文件编辑、查看或运行仅限 Windows 的工具时使用 `PowerShell` 或 `Command Prompt`。  
- 当你只想执行 Git 操作或快速运行脚本（不需 Linux 环境）时，可在 Windows 使用 `Git Bash`，但若项目代码常与 Linux 工具链交互（node 原生模块、make、python 扩展），请在 WSL 中运行 `git`。

2) 在 VS Code 中使用终端（快捷操作）

- 打开终端：`Ctrl+` `（反引号）` 或 `Terminal -> New Terminal`。在 Remote‑WSL 窗口里默认会打开 WSL shell。  
- 在 Windows VS Code 中设置默认终端为 Git Bash（settings.json 示例）：

```json
{
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "terminal.integrated.profiles.windows": {
    "Git Bash": {
      "path": "C:\\Program Files\\Git\\bin\\bash.exe"
    }
  }
}
```

3) 常用命令示例

- 在 WSL 内打开仓库并启动前端：

```bash
cd /home/victorkure/workspace/vscode_study/web-projects/my-react-app
npm install
npm start
```

- 在 WSL 内对 Python 项目操作：

```bash
cd /home/victorkure/workspace/vscode_study/python-projects/ai-lab
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python example.py
```

- 在 Windows 上用 Git Bash 打开仓库（仅查看或轻量编辑）：

```bash
cd /d/dev/source_code/vscode_study
code .
```

说明：`code .` 在 Git Bash 中会打开 Windows 版本的 VS Code（非 Remote‑WSL 会话），如果想在 WSL 中打开，请在 WSL shell 内运行 `code .`。

4) 推荐实践与注意事项

- 在同一项目里尽量保持 Git 操作在同一环境（推荐 WSL）以避免行尾与权限差异。  
- 若使用 Git Bash 做 Git 操作，请确保 `core.autocrlf` 与 `.gitattributes` 已正确配置。  
- 当出现权限或构建问题时，先在 WSL 中重试，以排除跨平台差异。

## Windows 与 WSL 文件访问与同步（如何“同步” D:\ 与 /mnt/d）

关键结论：长期开发的最佳落点是 `/home/victorkure/workspace/vscode_study`；如果你还保留 Windows 侧仓库，`D:\dev\source_code\vscode_study` 仍可作为初始源或备份视图，但不建议作为长期运行目录。根据性能和开发习惯，有两种常用做法：

1) 直接使用挂载路径（最简单，零复制）

- 说明：在 WSL 中访问 `/mnt/d/...` 即在操作 Windows 磁盘上的原始文件。Windows VS Code 与 WSL 都能看到同一套文件。  
- 优点：无需额外操作，适合查看/编辑/运行轻量脚本。  
- 缺点：大量 I/O（如 `npm install`、构建、测试）性能较差，可能导致文件系统延迟和 node_modules 问题。  
- 快速命令（在 WSL）：

```bash
cd /mnt/d/dev/source_code/vscode_study
code .    # 在 WSL 中打开（Remote-WSL），或直接在 Git Bash/Windows 打开路径
```

2) 把仓库克隆到 WSL 本地文件系统（推荐用于长期开发）

- 说明：在 WSL 的 home 下（例如 `/home/victorkure/workspace`）维护开发副本，I/O 性能更好，兼容 Linux 工具链。
- 推荐做法：使用 Git 做单一来源（commit/push/pull）同步不同环境的改动。  
- 操作示例（把 Windows 上的仓库镜像到 WSL）：

```bash
# 在 WSL 中创建目录并克隆（从 Windows 挂载路径克隆本地副本）
mkdir -p /home/victorkure/workspace
cd /home/victorkure/workspace
git clone /mnt/d/dev/source_code/vscode_study vscode_study
cd vscode_study
# 安装依赖并开发
```

- 若已在 WSL 本地修改并想把改动回传到 Windows 上的原始仓库，请使用标准 Git workflow（commit -> push 到远程，或在两端通过 pull/push 同步）。避免使用文件复制做为常规同步方式。  

3) Windows 上拉取 WSL 里的最新文件（你现在最需要的操作）

- 适用场景：你把 WSL 里的 `/home/victorkure/workspace/vscode_study` 作为主目录开发，同时希望 Windows 侧的 `D:\dev\source_code\vscode_study` 保持最新镜像，用来查看、备份或偶尔在 Windows 工具里打开。
- 推荐方式：在 Windows PowerShell 里从 WSL 共享路径把文件拉回 D 盘。

Windows 里快速打开这个 WSL 目录的方式：

1. 在文件资源管理器地址栏直接输入 `\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study`。
2. 打开后右键这个文件夹，选择“固定到快速访问”。
3. 如果你更喜欢桌面入口，右键桌面新建一个快捷方式，目标同样填这个路径。
4. 以后你就可以从“快速访问”直接进入 WSL 工作区，不必每次手动输入路径。

```powershell
robocopy "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" "D:\dev\source_code\vscode_study" /MIR /XD .git node_modules .venv
```

- 说明：`/MIR` 会把目标目录镜像成源目录，包含删除目标里多余的文件；第一次用可以先把 `/MIR` 改成 `/E`，确认无误后再切回镜像模式。
- 如果你的 WSL 发行版不是 `Ubuntu`，先用 `wsl -l -v` 查发行版名称，再替换上面的 `\\wsl$\Ubuntu`。
- 如果你不想每次手敲，可以把命令保存成 `sync-from-wsl.ps1`，以后在 Windows 里右键运行或在 PowerShell 里执行。

```powershell
robocopy "D:\dev\source_code\vscode_study" "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" /MIR /XD .git node_modules .venv
```

4) 更稳的日常同步原则

- 让 WSL 的 `/home/victorkure/workspace/vscode_study` 做唯一主目录。

- 如果你确实需要双向协作，优先让 Git 负责版本同步，不要靠人工复制目录。

## WSL + VS Code 具体操作流程（从打开到开发）

这一段按“第一次打开 WSL -> 用 VS Code 进入 -> 开始改代码 -> 进入 /home/victorkure/workspace/vscode_study”来写，适合直接照着做。

1) 打开 WSL / Ubuntu

- 最简单的方式：按 `Win` 键，直接搜索并打开 `Ubuntu`；这就是在 WSL 里启动 Ubuntu 发行版。
- 也可以在 Windows Terminal / PowerShell 里执行：

```powershell
wsl
# 或指定发行版
wsl -d Ubuntu
```

- 第一次进入时，按提示完成 Linux 用户名和密码配置。
- 进入后建议先更新环境：

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y git curl build-essential
```

2) 进入当前仓库

- 长期开发建议直接进入 `/home/victorkure/workspace/vscode_study`。
- 在 WSL 终端里执行：

```bash
cd /home/victorkure/workspace/vscode_study
pwd
```

- 这个目录需要你在 WSL 里主动创建或通过 `git clone` 拉出来；它不是 Windows 磁盘的挂载视图，所以不会自动出现。
- 如果 `cd` 失败，先检查仓库是否已经 clone 到 `/home/victorkure/workspace/vscode_study`，再确认 Linux 用户名是否就是 `victorkure`。

- 如果你想少记一步，也可以直接运行仓库里的启动脚本：

```powershell
scripts\wsl\start-wsl-vscode-dev.bat
```

- 这个脚本会把你带到 WSL、切到仓库目录，并打开 VS Code 的 Remote-WSL 会话。

3) 用 VS Code 打开 WSL 工作区

- 在 Windows 上安装 VS Code Stable，并安装扩展 `Remote - WSL`。
- 在已经打开的 WSL 终端里执行：

```bash
code .
```

- 如果这是第一次用 `code`，先在 Windows 侧打开 VS Code，再按 `Ctrl+Shift+P` 运行 `Remote-WSL: New Window`，确认 Remote-WSL 插件已正常工作。
- 打开成功后，VS Code 左下角通常会显示 `WSL: Ubuntu`，这表示当前窗口已经连到 WSL。

4) 在 VS Code 里怎么用

- 终端：`Ctrl+` `` 打开终端，默认就是 WSL shell，不需要再切回 PowerShell。
- 文件：用 VS Code 直接编辑 `/home/victorkure/workspace/vscode_study` 下的文件，保存后就是 Linux 文件系统中的同一份内容。
- 扩展：Python、Pylance、ESLint、Prettier、Docker 这类扩展应在 Remote-WSL 环境里安装或启用，这样工具链和终端环境保持一致。
- 调试：前端和 Python 示例优先在 WSL 终端中启动，再用 VS Code 的调试配置附加或运行。

5) 如何理解“同步” D:\dev\source_code\vscode_study

- 如果你直接在 `/mnt/d/dev/source_code/vscode_study` 里开发，就不需要额外同步，Windows 和 WSL 看到的是同一份文件。
- 如果你把仓库放在 `/home/victorkure/workspace/vscode_study`，Windows 和 WSL 就不再共享同一份底层文件，此时同步方式应改成 Git。
- 这类场景里，“同步”不是复制目录，而是通过 commit / push / pull 保持改动一致。

6) Windows 上怎么拉取 WSL 最新文件

- 打开 Windows PowerShell。
- 运行下面命令，把 WSL 里的最新内容拉回 D 盘：

```powershell
robocopy "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" "D:\dev\source_code\vscode_study" /MIR /XD .git node_modules .venv
```

- 如果你只想先试运行，不想删除目标目录多余文件，把 `/MIR` 改成 `/E`。
- 如果 `Ubuntu` 不是你的发行版名称，先执行 `wsl -l -v` 查看后再替换。
- 如果 Windows 端已经开着这个项目的 VS Code 窗口，建议先关闭相关文件，避免复制时出现占用冲突。

7) Windows 上从 D 盘回灌到 WSL home

- 如果你已经确认 Windows 侧 `D:\dev\source_code\vscode_study` 才是最新版本，可以反向把它同步回 WSL 主目录。
- 打开 Windows PowerShell，执行：

```powershell
robocopy "D:\dev\source_code\vscode_study" "\\wsl$\Ubuntu\home\victorkure\workspace\vscode_study" /MIR /XD .git node_modules .venv
```

- 这个命令会把 D 盘内容同步到 WSL 的 home 目录。
- 如果你只想先试跑，改用 `/E`。
- 如果 WSL 目录里有你不想保留的旧文件，`/MIR` 才会把它们一起清掉。

8) 如果你想自动化

- 可以把上面的 `robocopy` 命令写进一个 `sync-from-wsl.ps1`。
- 以后只需要在 Windows 侧运行这个脚本，就能把 WSL 里的最新文件同步到 D 盘。

9) 推荐的日常开发顺序

- 先打开 WSL。
- 再 `cd /home/victorkure/workspace/vscode_study`。
- 然后 `code .` 进入 Remote-WSL。
- 最后在 VS Code 终端里执行安装依赖、启动开发服务器、跑测试。

10) 什么时候不要直接用 /mnt/d

- 如果项目会频繁执行 `npm install`、大量构建或测试，建议把仓库克隆到 WSL 的 home 目录，例如 `/home/victorkure/workspace/vscode_study`。
- 原因是 Windows 盘挂载路径在大量 I/O 下通常比 WSL 本地文件系统慢，尤其是 `node_modules` 较大的前端项目。
