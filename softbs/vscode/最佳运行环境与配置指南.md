# 最佳运行环境与配置指南（针对本仓库非 Java 示例：Python / Node / React 等）

推荐首选：在 Windows 上使用 WSL2（Ubuntu）配合 VS Code 的 Remote - WSL 插件。

为什么选择 WSL2（简要理由）

- 与生产 Linux 环境一致：Node / Python / 包管理和原生工具在 Linux 下行为更稳定。 
- 兼容性最好：多数示例（特别是 Node/React、npm install、native 模块编译、Docker/DevContainer）在 WSL 环境更少问题。
- 开发体验佳：VS Code Remote - WSL 允许在 Linux 环境中直接打开工作区、安装扩展并使用本地终端。 
- 与 Docker 集成顺畅：Docker Desktop 的 WSL2 后端可直接在 WSL 与 Windows 之间无缝工作。

何时考虑替代方案

- 使用 Windows 原生（VSCode + Git Bash）适合：仅运行简单脚本、工具必须是 Windows 原生程序，或你无法启用 WSL（如公司策略）。
- 纯 Windows Git Bash：方便做 Git 操作与简单命令行任务，但对 node 原生模块或需要 Linux 工具链的项目不稳定。

快速配置步骤（推荐顺序）

1) 启用并安装 WSL2（Windows 11）

```powershell
# 以管理员 PowerShell 运行（Windows 11）
wsl --install -d Ubuntu
# 或手动启用（若需要）：
# dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
# dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
# 重启后在 PowerShell 中运行：
wsl --set-default-version 2
```

2) 打开 WSL（第一次启动配置 Ubuntu），更新并安装常用工具

```bash
# 在 WSL Ubuntu 中运行
sudo apt update && sudo apt upgrade -y
sudo apt install -y build-essential git curl ca-certificates
```

3) 安装 Python 与虚拟环境

```bash
sudo apt install -y python3 python3-venv python3-pip
python3 -m pip install --user pipx
# 可选：设置 pip 使用用户目录或全局 venv
```

4) 安装 Node.js（建议通过 nvm 管理）

```bash
# 安装 nvm（按官方说明）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.5/install.sh | bash
# 重新加载 shell 后安装 Node
nvm install --lts
nvm use --lts
# 安装常用包管理器（可选）
npm install -g pnpm
```

5) 安装并配置 Docker（可选但推荐用于容器化示例）

- 在 Windows 上安装 Docker Desktop，打开 WSL2 后端并启用对 WSL 发行版的整合。
- 在 WSL 中确认 docker 可用：

```bash
wsl -d Ubuntu
docker version
```

6) 在 VS Code 中安装并使用 Remote - WSL

- 在 Windows 上安装 VS Code（Stable）。
- 安装扩展：`Remote - WSL`、`Python`、`Pylance`、`Pyright`、`ESLint`、`Prettier`、`ES7 React/Redux/GraphQL`、`Debugger for Chrome`、`Docker`。
- 在 VS Code 里：`Remote-WSL: New Window`，然后用 `File > Open Folder` 打开 WSL 中挂载的仓库路径（例如 `/mnt/d/dev/source_code/vscode_study`）或直接在 Windows 资源管理器中右键 “在 WSL 中打开” 的方式。

7) 推荐的项目运行示例（在 WSL 终端）

- Python 导入虚拟环境并运行：

```bash
cd /mnt/d/dev/source_code/vscode_study/python-projects/ai-lab
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python example.py
```

- React / Node 示例：

```bash
cd /mnt/d/dev/source_code/vscode_study/web-projects/my-react-app
pnpm install
pnpm start
# 或 npm install && npm start
```

8) Git 与 Git Bash

- 推荐在 WSL 内使用 `git` 做代码操作以保证文件权限与 LF 行尾一致性；Windows 上可保留 Git Bash 用于快速本地命令或在 Windows 主机执行脚本。

9) VS Code 设置建议（settings.json 片段）

```json
{
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "python.defaultInterpreterPath": "/usr/bin/python3",
  "editor.formatOnSave": true
}
```

安全与注意事项

- WSL 与 Windows 文件系统互访问，但长期在 Windows 文件系统（/mnt/c /mnt/d）上运行大量 node_modules 可能性能较差，建议在 WSL 的家目录下克隆或 `rsync` 项目以获得最佳 I/O 性能，或将项目挂载到 WSL 的 Linux 文件系统中。
- 权限与行尾：在跨系统开发时注意 `.gitattributes` 与 `core.autocrlf` 的设置以避免不必要的差异。
- 若需要 GPU 加速（例如某些 ML 本地测试），Windows 原生或 WSLg 的 GPU 支持设置复杂，视硬件与驱动而定，可能需要额外配置（可另行提供指南）。

总结建议（一句话）

- 最佳方案：使用 WSL2 + VS Code Remote - WSL，在 WSL 内安装 Python / Node / Docker，平衡兼容性与开发体验；把 Windows Git Bash 留作主机端快捷命令与 Git 操作备用。

## WSL2 下如何用 VS Code 打开工作区（快速指南）

推荐的开发目录

- 优先（直接打开已有仓库）：`/mnt/d/dev/source_code/vscode_study` — 这是你当前仓库在 WSL 中的挂载路径，适合直接打开并运行示例。
- 高 I/O 性能（可选）：`~/projects/vscode_study` — 若需要频繁进行 `npm install` / 大量文件读写，建议将仓库克隆到 WSL 的 Linux 文件系统以获得更好性能。

快速打开（前提：已安装 WSL2、VS Code 与 Remote - WSL 扩展）

- 在 WSL（Ubuntu）终端中运行：

```bash
cd /mnt/d/dev/source_code/vscode_study
code .
```

说明：`code .` 会通过 Remote‑WSL 在 WSL 环境中打开该文件夹。如果提示找不到 `code`，请在 Windows 上安装 VS Code 并启用 Remote‑WSL 扩展，然后在 VS Code 的命令面板运行 “Remote‑WSL: New Window” 再重试。

从 Windows 的 VS Code 打开并切换到 WSL（图形界面）

1. 在 Windows 上打开 VS Code。  
2. 按 `Ctrl+Shift+P`，执行 `Remote-WSL: New Window`。  
3. 在新窗口里选择 `File -> Open Folder...`，输入或浏览到 `/mnt/d/dev/source_code/vscode_study` 并打开。  
4. 若在 Windows 窗口已打开该文件夹，使用命令面板选择 `Remote-WSL: Reopen Folder in WSL` 切换到 WSL 会话。

仅浏览或编辑（不运行代码）的快速打开方式

- 若只是查看文档或做轻量编辑，可直接在 Windows 中打开 `D:\dev\source_code\vscode_study`，右键选择 “Open with Code” 用 Windows 的 VS Code 浏览和编辑；若将来需要运行示例或安装依赖，建议通过 Remote‑WSL 重新打开为 WSL 会话以保证一致性。

额外简要建议

- 长期开发并运行示例（安装依赖、运行开发服务器、构建前端）：始终在 Remote‑WSL 会话下操作以避免兼容性与性能问题。  
- 偶尔查看文档或编辑 README：Windows VS Code 更便捷。

## Terminal 与 Git Bash 使用指南

1) 什么时候用哪个终端

- 在 Remote‑WSL 会话中：优先使用 WSL 的 shell（`bash` / `zsh`），所有开发、安装依赖、运行 dev server、构建等操作都在 WSL 里进行。  
- 在 Windows 主机上：用于轻量文件编辑、查看或运行仅限 Windows 的工具时使用 `PowerShell` 或 `Command Prompt`。  
- 当你只想执行 Git 操作或快速运行脚本（不需 Linux 环境）时，可在 Windows 使用 `Git Bash`，但若项目代码常与 Linux 工具链交互（node 原生模块、make、python 扩展），请在 WSL 中运行 `git`。

2) 在 VS Code 中使用终端（快捷操作）

- 打开终端：`Ctrl+` `（反引号）` 或 `Terminal -> New Terminal`。在 Remote‑WSL 窗口里默认会打开 WSL shell。  
- 在 Windows VS Code 中设置默认终端为 Git Bash（settings.json 示例）：

```json
{
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "terminal.integrated.profiles.windows": {
    "Git Bash": {
      "path": "C:\\Program Files\\Git\\bin\\bash.exe"
    }
  }
}
```

3) 常用命令示例

- 在 WSL 内打开仓库并启动前端：

```bash
cd /mnt/d/dev/source_code/vscode_study/web-projects/my-react-app
npm install
npm start
```

- 在 WSL 内对 Python 项目操作：

```bash
cd /mnt/d/dev/source_code/vscode_study/python-projects/ai-lab
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python example.py
```

- 在 Windows 上用 Git Bash 打开仓库（仅查看或轻量编辑）：

```bash
cd /d/dev/source_code/vscode_study
code .
```

说明：`code .` 在 Git Bash 中会打开 Windows 版本的 VS Code（非 Remote‑WSL 会话），如果想在 WSL 中打开，请在 WSL shell 内运行 `code .`。

4) 推荐实践与注意事项

- 在同一项目里尽量保持 Git 操作在同一环境（推荐 WSL）以避免行尾与权限差异。  
- 若使用 Git Bash 做 Git 操作，请确保 `core.autocrlf` 与 `.gitattributes` 已正确配置。  
- 当出现权限或构建问题时，先在 WSL 中重试，以排除跨平台差异。

## Windows 与 WSL 文件访问与同步（如何“同步” D:\ 与 /mnt/d）

关键结论：Windows 的 `D:\dev\source_code\vscode_study` 已被 WSL 挂载为 `/mnt/d/dev/source_code/vscode_study`，两者是同一份文件的不同视图——通常不需要复制或实时同步。但根据性能和开发习惯，有两种常用做法：

1) 直接使用挂载路径（最简单，零复制）

- 说明：在 WSL 中访问 `/mnt/d/...` 即在操作 Windows 磁盘上的原始文件。Windows VS Code 与 WSL 都能看到同一套文件。  
- 优点：无需额外操作，适合查看/编辑/运行轻量脚本。  
- 缺点：大量 I/O（如 `npm install`、构建、测试）性能较差，可能导致文件系统延迟和 node_modules 问题。  
- 快速命令（在 WSL）：

```bash
cd /mnt/d/dev/source_code/vscode_study
code .    # 在 WSL 中打开（Remote-WSL），或直接在 Git Bash/Windows 打开路径
```

2) 把仓库克隆到 WSL 本地文件系统（推荐用于长期开发）

- 说明：在 WSL 的 home 下（例如 `~/projects`）维护开发副本，I/O 性能更好，兼容 Linux 工具链。  
- 推荐做法：使用 Git 做单一来源（commit/push/pull）同步不同环境的改动。  
- 操作示例（把 Windows 上的仓库镜像到 WSL）：

```bash
# 在 WSL 中创建目录并克隆（从 Windows 挂载路径克隆本地副本）
mkdir -p ~/projects
cd ~/projects
git clone /mnt/d/dev/source_code/vscode_study vscode_study
cd vscode_study
# 安装依赖并开发
```

- 若已在 WSL 本地修改并想把改动回传到 Windows 上的原始仓库，请使用标准 Git workflow（commit -> push 到远程，或在两端通过 pull/push 同步）。避免使用文件复制做为常规同步方式。  

3) 一次性镜像或排除目录同步（使用 `rsync`）

- 当你需要把 Windows 上的完整工作区快速复制到 WSL（或反向）时，可用 `rsync`：

```bash
# 从 Windows 挂载路径同步到 WSL 本地（排除 node_modules）
rsync -av --exclude='node_modules' /mnt/d/dev/source_code/vscode_study/ ~/projects/vscode_study/

# 反向同步（慎用 --delete）
rsync -av --exclude='node_modules' ~/projects/vscode_study/ /mnt/d/dev/source_code/vscode_study/
```

- 注意：`rsync` 是复制操作，不会保留 Git 历史在目标上（除非包含 `.git`）。通常只用于一次性镜像或备份。

4) 跨环境 Git 最佳实践

- 在仓库中添加或维护 `.gitattributes`，并配置 `core.autocrlf` 以避免行尾差异。  
- 建议把 Git 操作统一在 WSL 环境执行，或明确约定哪个环境为主要提交端。  

选择建议

- 想要最快捷、零配置：直接在 `/mnt/d/...` 使用（方式1）。  
- 追求性能与稳定开发体验：把仓库克隆到 WSL 本地（方式2），并使用 Git 同步到远程仓库作为单一事实来源。
