# OpenClaw 使用教程

## 步骤 1：确认默认模型和认证

- **为什么做**：模型引用和认证是两个独立状态；“列表中可见”不等于“实际可调用”。
- **做什么**：查看当前模型、认证概览和实时探测。
- **执行命令**：

```bash
openclaw models status
openclaw models status --probe
```

- **预期结果**：显示唯一的 primary model，对应 provider 认证可用；`--probe` 会产生真实 API 请求，可能消耗额度。

## 步骤 2：按优先级配置模型

- **为什么做**：明确 primary 和 fallback 顺序，才能预测成本和故障切换行为。
- **做什么**：优先 OpenRouter；当 OpenRouter 不可用时，通过交互认证助手添加 NVIDIA provider；最后添加本地 Ollama。
- **执行命令**：

```bash
openclaw models auth login --provider openrouter --method oauth
openclaw models set openrouter/auto
openclaw models auth add
openclaw models list --all --provider nvidia
ollama pull qwen2.5-coder:1.5b
openclaw models list --local
```

- **预期结果**：OpenRouter 为 primary，NVIDIA 认证与当前 Model ID 由交互助手保存，Ollama 模型可被本地发现。

> OpenClaw 连接 Ollama 使用原生 API：WSL 本地是 `http://127.0.0.1:11434`，Docker Gateway 是 `http://host.docker.internal:11434`，不加 `/v1`。

## 步骤 3：先限制 workspace 和工具

- **为什么做**：OpenClaw 是长期运行的助手，过宽的文件、Shell 和浏览器权限会扩大提示注入和误操作风险。
- **做什么**：初学阶段只使用专用测试 workspace，默认只读，禁止写入、Shell 和浏览器，然后逐项放行。
- **执行命令**：

```bash
mkdir -p ~/.openclaw/workspace-learning
openclaw doctor
code ~/.openclaw
```

- **预期结果**：配置目录权限受限，测试 Agent 只能读取指定 workspace，高影响工具被拒绝或需要确认。

## 步骤 4：运行只读任务

- **为什么做**：在接入真实消息渠道前，应先验证本地 Agent 和权限边界。
- **做什么**：把一份无敏感信息的 Markdown 放入 workspace，要求 OpenClaw 摘要，不写文件。
- **执行命令**：

```bash
cp <github目录>/02_工具知识库/GitHub_AI_工具学习体系.md ~/.openclaw/workspace-learning/
openclaw gateway status
```

在 Control UI/Chat 中输入：

```text
只读取 workspace-learning/GitHub_AI_工具学习体系.md，列出五个重点工具和它们的通关成果。不修改文件，不访问网络。
```

- **预期结果**：输出与文档一致，无文件变更和网络工具调用。

## 步骤 5：与 VSCode/Codex 分工

- **为什么做**：OpenClaw 适合长期助手和渠道入口，Codex 适合仓库中可审查的代码变更；不要让两者同时写同一工作树。
- **做什么**：OpenClaw 只输出任务说明/研究结果，Codex 在 Git 分支中实施和验证。
- **执行命令**：

```bash
cd ~/code/<your-test-repo>
git switch -c feat/from-openclaw-spec
codex
```

- **预期结果**：仓库变更只由当前 Codex 会话产生，OpenClaw 不拥有该工作树的写权限。
