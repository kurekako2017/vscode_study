# RAGFlow 实战案例：GitHub AI 工具学习问答库

## 问题定义

- 语料：本目录中的学习体系和五个工具教程。
- 查询：安装前置、模型切换、实战步骤、故障排查。
- 新鲜度：文档更改后手动重新解析，后续再自动化。
- 安全：不上传 `.env`、日志中的 Key、私有代码或个人资料。
- 评估：Recall@5、引用正确率、答案忠实度。

## 步骤 1：准备干净语料

- **为什么做**：重复文档和密钥会降低检索质量并造成安全风险。
- **做什么**：只复制 Markdown 教程，检查密钥样式和重复文件。
- **执行命令**：

```bash
mkdir -p ~/ragflow-lab-data/github-ai-system
find <github目录> -path '*/工具实战/*' -name '*.md' -exec cp --parents {} ~/ragflow-lab-data/github-ai-system/ \;
rg -n 'sk-[A-Za-z0-9_-]{12,}|nvapi-[A-Za-z0-9_-]+' ~/ragflow-lab-data/github-ai-system || true
find ~/ragflow-lab-data/github-ai-system -type f -name '*.md' -exec sha256sum {} \; | sort
```

- **预期结果**：密钥扫描无命中，语料中只有准备上传的 Markdown。

## 步骤 2：建立基线索引

- **为什么做**：先用默认切块与检索建立基线，才能判断后续优化是否有效。
- **做什么**：在 RAGFlow 创建数据集，上传语料，使用默认 General/Markdown 配置解析。
- **执行命令**：

```bash
explorer.exe "$(wslpath -w ~/ragflow-lab-data/github-ai-system)"
```

- **预期结果**：所有文档解析完成，切块预览保留标题与步骤边界。

## 步骤 3：建立评估集

- **为什么做**：只凭主观感觉无法判断 RAG 修改是否改善。
- **做什么**：编写至少 10 个问题，每个问题记录期望文档和关键事实。
- **执行命令**：

```bash
code ~/ragflow-lab-data/evaluation.md
```

建议字段：`ID | 问题 | 期望文档 | 关键事实 | Top5命中 | 引用正确 | 答案忠实`。

- **预期结果**：10 个问题覆盖五个工具，每个期望答案均可在语料中找到。

## 步骤 4：评估并针对失败选择技术

- **为什么做**：Query Rewrite、Hybrid Search 和 Rerank 会增加延迟与成本，应只解决已观察到的失败。
- **做什么**：先跑基线；术语精确匹配失败时提高混合搜索中的关键词权重，top-k 顺序不佳时再加 reranker，口语问题不清时再启用 rewrite。
- **执行命令**：

```bash
code ~/ragflow-lab-data/evaluation.md
```

- **预期结果**：记录基线和修改后的 Recall@5、引用正确率、平均延迟和模型成本，只保留有可测改善的复杂度。

## 步骤 5：制定刷新与权限策略

- **为什么做**：长期维护的 RAG 必须处理文档变更、删除、重复和访问控制。
- **做什么**：为每份文档记录路径、Git commit、最后索引时间和可见范围；先手动执行，稳定后再自动化。
- **执行命令**：

```bash
git -C <github目录> rev-parse HEAD
date --iso-8601=seconds
```

- **预期结果**：能回答“这个答案来自哪个文档版本、何时索引、谁可访问”。
