# RAGFlow 常见问题

## 问题 1：Elasticsearch 启动失败

- **为什么做**：最常见根因是 `vm.max_map_count` 过低或内存不足。
- **做什么**：检查内核参数、容器状态和 Elasticsearch 日志。
- **执行命令**：

```bash
sysctl vm.max_map_count
sudo sysctl -w vm.max_map_count=262144
cd ~/code/ragflow/docker
docker compose ps
docker compose logs --tail=200 es01
```

- **预期结果**：参数达到 262144，Elasticsearch 不再循环重启。

## 问题 2：文档解析长时间卡住

- **为什么做**：DeepDoc、OCR 和大 PDF 会消耗大量内存与 CPU。
- **做什么**：先用一份小 Markdown 和轻量解析器验证流程，同时观察资源与 task executor 日志。
- **执行命令**：

```bash
docker stats --no-stream
cd ~/code/ragflow/docker
docker compose logs --tail=200 ragflow-server
```

- **预期结果**：小 Markdown 可完成解析；若 PDF 单独失败，则聚焦文档解析器和资源，而非重装整个 RAGFlow。

## 问题 3：RAGFlow 容器连不上 Ollama

- **为什么做**：容器里的 `localhost:11434` 不是 Windows/WSL 宿主。
- **做什么**：用 Docker Desktop 的宿主别名测试连通性，并在 UI 中使用该地址。
- **执行命令**：

```bash
docker run --rm curlimages/curl:latest http://host.docker.internal:11434/api/tags
```

- **预期结果**：返回包含 `qwen2.5-coder:1.5b` 的模型列表。

## 问题 4：答案流畅但引用错误

- **为什么做**：这通常是检索、切块或提示约束问题，不是只换更大模型就能解决。
- **做什么**：在 Retrieval Test 中查看 top-k；若正确块未召回，优化切块/检索；若已召回但答案未使用，再优化提示和模型。
- **执行命令**：

```bash
code ~/ragflow-lab-data/evaluation.md
```

- **预期结果**：每个失败被归类为“未召回、排序低、上下文丢失、生成不忠实”之一。

## 问题 5：升级后数据丢失

- **为什么做**：`docker compose down -v` 会删除卷，不是普通停止命令。
- **做什么**：升级前备份卷和配置，停止时不使用 `-v`。
- **执行命令**：

```bash
cd ~/code/ragflow/docker
docker compose down
docker volume ls
git diff -- .env docker-compose.yml
```

- **预期结果**：容器停止但数据卷仍存在；升级前可看到配置差异。
