# browser-use 实战案例：采集 GitHub 项目学习信息

## 目标与安全边界

从一个公开 GitHub 仓库页读取项目描述、主要语言、License 和 README 安装入口，输出 `output/project-note.md`。只允许 `github.com` 和 `raw.githubusercontent.com`，不登录、不点 Star、不创建 Issue。

## 步骤 1：固定输入和验收标准

- **为什么做**：浏览器 Agent 的自然语言任务必须有明确边界。
- **做什么**：创建输出目录，选择一个公开仓库 URL。
- **执行命令**：

```bash
cd ~/code/browser-use-lab
mkdir -p output
export TARGET_REPO='https://github.com/browser-use/browser-use'
```

- **预期结果**：`output/` 存在，`echo "$TARGET_REPO"` 输出公开 GitHub URL。

## 步骤 2：限制 Agent 权限

- **为什么做**：只读任务不应拥有写网站或跨域操作的自由。
- **做什么**：在 `agent.py` 设置域名白名单，将任务文本改为明确的只读采集说明。
- **执行命令**：

```bash
code agent.py
```

任务必须包含：

```text
只读取公开页面；不登录、不提交表单、不执行写操作。
返回项目描述、主要语言、License、安装入口和每项的来源 URL。
```

- **预期结果**：代码中可见域名白名单和禁止写操作的任务约束。

## 步骤 3：运行并保存结果

- **为什么做**：保存原始结果，才能评估不同模型的准确率。
- **做什么**：先用 OpenRouter 执行，将输出同时保存为日志。
- **执行命令**：

```bash
LLM_PROVIDER=openrouter LLM_MODEL='<模型-ID>' uv run python agent.py 2>&1 | tee output/run.log
```

- **预期结果**：任务正常结束，`output/run.log` 存在，结果包含四类信息和来源 URL。

## 步骤 4：用 Codex 整理与验证

- **为什么做**：浏览器 Agent 的最终结果仍需要结构检查和人工抽查。
- **做什么**：要求 Codex 只根据日志生成 Markdown，对缺少证据的字段标记“待验证”。
- **执行命令**：

```bash
codex -C ~/code/browser-use-lab
```

在 Codex 中输入：

```text
读取 output/run.log，生成 output/project-note.md。不添加日志中没有的事实；每项保留来源 URL；待验证项明确标注。
```

- **预期结果**：`project-note.md` 结构清晰，手动打开仓库页能核对主要语言、License 和安装入口。
