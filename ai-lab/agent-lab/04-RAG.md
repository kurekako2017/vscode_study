# Agent 中的 RAG

> 归位说明：RAG 的基础概念、文档读取、切分、检索、引用和最小代码教程，统一看 [llm-lab/04-RAG.md](../llm-lab/04-RAG.md)。本文件只讲 `Agent` 视角下的 RAG：Agent 为什么需要 RAG、RAG 在 Agent 里扮演什么角色、怎么和 Tool Calling / Workflow 组合。

下面这张图把 `RAG` 的两条主线拆开画清楚了：

- 上半部分是 `知识更新`：从原始文档到向量库
- 下半部分是 `知识检索`：从问题到召回、重排、增强、生成

![Agent RAG 流程图](./assets/agent_rag_flow.svg)

这版图故意把每一步压成短词，先方便你抓住流程，不需要一开始就记完整术语。

如果先按“流程顺序”读这张图，可以这样理解：

1. `原始文档`
   - 先把公司文档、手册、FAQ、规范等资料准备好。
   - 这些内容还只是普通文本，不能直接高效检索。
2. `切分成 chunks`
   - 长文档会被切成多个小片段，也就是 `chunks`。
   - 这样做是为了避免一次塞入太长文本，也方便后面只取相关片段。
   - 每个 chunk 通常会保留来源信息，比如文件名、段落号、chunk 编号。
3. `Embedding`
   - 每个 chunk 会被转成向量。
   - 向量不是答案，而是“语义表示”，方便系统比较“哪段内容和问题更像”。
4. `Vector DB / 向量库`
   - 向量和 chunk 的来源信息一起存进去。
   - 这样后面检索时，可以先找出相似向量，再拿回对应的原文片段。
5. `用户提问`
   - 用户输入问题后，系统不会直接猜答案，而是先把问题也做同样的表示转换。
6. `问题 Embedding`
   - 问题会被转成向量。
   - 这样它才能和文档 chunk 的向量做相似度比较。
7. `Retriever / 检索器`
   - 检索器会在向量库里找和问题最接近的几个 chunk。
   - 这一步通常叫 `Top-K 检索`，意思是返回最相关的前几个结果。
8. `RAG 增强`
   - 系统把检索到的 chunks 拼成上下文，提供给模型。
   - 这一步的目标是让模型“基于资料回答”，而不是凭空生成。
9. `LLM 生成回答`
   - 模型读取问题和上下文后生成最终答案。
   - 如果资料足够，回答会更稳；如果资料不足，应该明确说明。
10. `输出来源`
   - 最好把命中的 chunk 来源一起返回。
   - 这样用户能知道答案来自哪里，也方便人工核对。

如果再把图里的元素对照一下，可以这样记：

| 图中元素 | 含义 | 作用 |
| --- | --- | --- |
| `原始文档` | 还没处理的资料 | 输入到 RAG 的知识来源 |
| `chunks` | 切分后的文档片段 | 把长文档拆成可检索的小块 |
| `Embedding` | 文本转向量 | 让系统比较语义相似度 |
| `向量数据库` | 存向量和来源信息的地方 | 负责保存和检索 |
| `用户提问` | 用户输入的问题 | 触发检索流程 |
| `Top-K` | 返回最相关的前几个结果 | 控制召回范围 |
| `相关 chunks` | 命中的片段 | 提供给模型作为依据 |
| `拼接上下文` | 把命中的片段整理成提示词 | 让模型基于资料回答 |
| `LLM 生成回答` | 最终输出答案的模型 | 生成自然语言回答 |
| `答案` | 返回给用户的结果 | 最终交付内容 |

这里图里特意把 `source` 一起标出来，是因为 RAG 不只是“找片段”，还要“带着来源返回”。

- `chunks` 负责承载正文内容
- `source` 负责告诉你这段内容来自哪里
- 两者一起返回，模型才能给出可追溯的回答，用户也才能核对依据

## 1. Agent 中的 RAG 是什么

`Agent 中的 RAG` 不是重新学习一遍 RAG 基础，而是把 RAG 当成 Agent 的一个 `知识工具 / Knowledge Tool`。

可以先这样理解：

```text
普通 RAG：用户提问 -> 系统检索资料 -> 模型基于资料回答
Agent 中的 RAG：Agent 判断需要资料 -> 调用 RAG 工具检索 -> 根据检索结果决定下一步
```

在 Agent 系统里，RAG 通常属于：

| 分类 | 说明 |
| --- | --- |
| 技术类型 | Agent 知识工具、社内资料检索能力 |
| 系统层次 | Tool 层 / Knowledge Retrieval 层 |
| 连接对象 | Tool Calling、Agent Workflow、Planner、Executor |
| 日本现场说法 | `RAG ツール`, `社内検索ツール`, `ナレッジ検索`, `情報取得処理` |

一句话先记住：

- `llm-lab/04-RAG.md` 学“怎么做 RAG”，本文件学“Agent 怎么使用 RAG”。

## 2. 为什么 Agent 需要 RAG

很多 Agent 失败，不是因为模型不会推理，而是因为它没有可靠资料。

Agent 如果只靠模型已有知识，容易出现：

- 不知道公司内部规则
- 不知道项目最新资料
- 不知道 API 规格和限制
- 看起来回答流畅，但没有依据

RAG 给 Agent 补的是 `可查询的外部知识`。

在日本现场，最常见的关系通常是：

```text
社内文書 / 規程 / 手順書
  -> RAG / 社内検索
  -> 業務補助 Agent
  -> 人が確認して実行
```

所以学习顺序上，先会做 RAG，再把 RAG 接成 Agent 工具，会比一开始做复杂多 Agent 更稳。

## 2.1 Embedding 是什么

图里的 `Embedding`，你可以先把它理解成“把文字变成向量”的步骤。

```text
把文字变成模型能计算相似度的向量表示。
```

更直接一点说，`Embedding` 做的是“表示转换”：

- 文本本身对程序来说只是字符串
- `Embedding` 会把字符串变成一串数字
- 这串数字可以表达“语义接近不接近”

举个简单例子：

- `报销流程怎么走`
- `差旅报销的申请步骤`

这两句话文字不一样，但语义很接近。`Embedding` 的目标就是让它们在向量空间里靠得更近，方便后面检索。

在 RAG 里，`Embedding` 主要有两次用途：

| 阶段 | 做什么 | 目的 |
| --- | --- | --- |
| 文档入库 | 把文档 chunk 转成向量 | 方便存到向量数据库里 |
| 用户提问 | 把问题转成向量 | 方便和文档向量做相似度比较 |

所以图中的 `Embedding` 不是“答案生成器”，而是 `表示转换器`：

- 输入是文字
- 输出是向量
- 作用是让系统能算相似度、做召回、找相关片段

如果没有 `Embedding`，系统就只能靠关键词匹配，效果通常会差很多，尤其是同义表达、近义表达、上下文表达比较多的时候。

## 2.2 为什么 RAG 要先做 Embedding

RAG 里先做 `Embedding`，是为了把“人能理解的文本”变成“机器好比较的数字表示”。

这样做的好处是：

- 可以把文档预先索引起来
- 可以快速比较问题和文档的语义接近程度
- 可以支持“没完全同词，但意思相近”的检索

你可以把它理解成一个两步过程：

```text
文档切块 -> 转向量 -> 存入向量库
用户提问 -> 转向量 -> 找相近向量 -> 取回相关 chunk
```

一句话记忆：

- `Embedding` 负责把文字变成向量
- `向量数据库` 负责存和查这些向量
- `检索算法` 负责算谁更像谁

可以直接记成：

```text
Embedding = 语义表示
Vector DB = 语义存储
Retriever = 语义匹配
```

## 4. Agent + RAG 的角色分工

先看“是什么”，再看“作用”：

| 角色 / 名词 | 日本語 / 现场说法 | 是什么 | 在 Agent 中的作用 | 对应位置 |
| --- | --- | --- | --- | --- |
| User | 利用者 / 依頼者 | 提出任务的人 | 给 Agent 一个目标或问题 | 命令行 prompt / Web 请求 |
| Agent Controller | エージェント制御 | 管理任务推进的控制层 | 判断下一步是否需要查资料、调用工具或输出答案 | `run_agent()` / workflow controller |
| Planner | 計画担当 | 任务拆解角色 | 判断“先查资料再回答”还是“直接回答” | `plan_task()` |
| RAG Tool | RAG ツール / 社内検索ツール | 被 Agent 调用的知识检索工具 | 根据 query 返回相关片段和来源 | `search_internal_docs()` |
| Retriever | 検索器 | RAG 内部检索模块 | 从文档库找 Top-K 片段 | `retrieve()` |
| Observation | 観測結果 / ツール結果 | 工具返回给 Agent 的结果 | 让 Agent 看见真实资料后继续判断 | `function_call_output` |
| Final Answer | 最終回答 | Agent 给用户的最终输出 | 基于资料和任务目标输出回答 | 控制台 / API response |

核心区别：

- 普通 RAG 的主角是 `检索 + 回答`。
- Agent + RAG 的主角是 `判断是否需要检索 + 调用检索工具 + 基于结果继续行动`。

## 6. 和 Tool Calling 的关系

RAG 在 Agent 里通常会包装成一个 Tool。

例如工具可以设计成：

```python
def search_internal_docs(query: str, top_k: int = 5) -> dict:
    """Search internal documents and return relevant chunks with sources."""
```

它的输入输出要尽量稳定：

| 项目 | 说明 |
| --- | --- |
| 输入 | 用户问题、检索关键词、Top-K |
| 输出 | 命中的文档片段、来源文件、相似度或得分 |
| 失败时 | 返回空结果或错误原因，不要让 Agent 猜 |
| 安全边界 | 限制只能查允许访问的文档目录或索引 |

Tool Calling 负责：

- 让模型提出“我要查资料”
- 把 query 参数传给 RAG 工具
- 把 RAG 工具结果回填给模型

RAG 负责：

- 真实检索资料
- 返回相关片段
- 提供来源依据

## 7. 和 Agent Workflow 的关系

在可控 Agent 里，不建议让模型随便无限调用 RAG。

更稳的方式是固定工作流：

```text
分析任务 -> 判断需要哪些资料 -> 调用 RAG -> 整理证据 -> 生成回答 -> 人工确认
```

对应到工作流角色：

| 工作流阶段 | RAG 的位置 | 说明 |
| --- | --- | --- |
| Analyzer | 判断是否需要资料 | 如果问题涉及社内规则、设计书、FAQ，就需要 RAG |
| Planner | 生成检索计划 | 决定查哪些关键词、查几次、Top-K 多少 |
| Executor | 调用 RAG 工具 | 执行检索并拿到 source |
| Reviewer | 检查依据 | 判断答案是否真的基于检索结果 |
| Finalizer | 输出最终结果 | 给用户回答，同时标注来源和不足 |

## 8. 推荐练习

不要在这里重新做 RAG 基础，而是把已有 RAG 能力接进 Agent。

练习顺序：

1. 跑通 [ai-lab/agent-lab/projects/doc_qa_agent](./projects/doc_qa_agent/README.md)。
2. 跑通 [ai-lab/agent-lab/projects/tool_agent_demo](./projects/tool_agent_demo/README.md)。
3. 把 `retrieve()` 包装成一个只读工具，例如 `search_docs`。
4. 让 Agent 在回答前先判断是否需要调用 `search_docs`。
5. 最终回答必须包含 `Sources`，没有资料时明确说资料不足。

最小目标：

```text
用户问一个项目资料问题
  -> Agent 判断需要查文档
  -> 调用 search_docs
  -> 得到 chunk + source
  -> 基于资料回答
  -> 输出来源
```


- 为什么最终回答必须保留来源

## 11. 常见坑

- 在 Agent 文档里重复学习 RAG 基础，却没有理解“RAG 作为工具”的位置。
- 让 Agent 每一步都查资料，导致成本和延迟上升。
- RAG 工具只返回文本，不返回来源。
- 检索结果不足时，Agent 仍然强行回答。
- 没有权限边界，Agent 可以查不该查的资料。
- 没有人工确认，把检索结果直接用于高风险业务动作。
