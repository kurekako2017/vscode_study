# 需求概要

## 1. 背景

在 `RAG` 或文档问答之外，现场还会出现另一类需求：让模型在有限范围内自己查看目录、读取文件、搜索文本，再基于结果整理答案。

这类需求比纯问答更接近 Agent，但在学习阶段不适合一开始就做复杂自治流程，因此先做一个只读、可控、最小化的工具调用 PoC。

## 2. 目的

实现一个最小可运行的 `Tool Calling` 示例，让模型能够在指定工作目录内：

- 列出文件
- 读取文本文件
- 搜索关键字
- 基于工具结果输出最终回答

## 3. 目标用户

- 学习 `Agent` 基础能力的开发者
- 已掌握模型调用与 `RAG`，准备继续学习工具调用的人
- 想理解“日本现场中有限工具调用助手”基本形态的人

## 4. 使用场景

典型场景如下：

- 指定一个本地目录作为工作范围
- 输入“帮我找 README 重点”或“帮我搜索数据库相关文档”
- 模型根据任务自动决定是否调用工具
- 工具返回结果后，模型再给出整理后的文本回答

## 5. 功能要求

### 5.1 文件列表获取

- 可以列出指定相对路径下的文件与子目录

### 5.2 文件读取

- 可以读取工作目录内的 UTF-8 文本文件内容

### 5.3 文本搜索

- 可以在工作目录内按关键字进行不区分大小写搜索
- 返回匹配文件、行号、文本内容

### 5.4 回答生成

- 模型必须基于工具返回结果整理最终回答
- 不应虚构不存在的文件内容

## 6. 非功能要求

- 运行方式为本地命令行
- 工具访问范围必须限制在 `--workdir`
- 默认只提供只读工具
- 实现保持最小化，便于学习和扩展

## 7. 范围外事项

当前版本不包含以下内容：

- 写文件
- 执行系统命令
- 调用数据库
- 调用外部 API
- 多 Agent 协作
- 长期记忆和任务编排

## 8. 成功标准

满足以下条件则认为 PoC 达成目标：

- 模型可以在需要时自动调用工具
- 工具调用结果能正确回填给模型
- 模型能基于工具结果返回可理解的最终答案
- 路径访问不会越出指定工作目录

## 9. 预期价值

这个案例的价值不在于“做成完整企业系统”，而在于帮助学习者理解：

- 工具 schema 如何定义
- 模型如何决定调用哪个工具
- 本地程序如何执行工具并把结果回传给模型
- 为什么安全边界在工具型 Agent 中很重要

## 10. 成本与注意点

- 这是学习版 PoC，不是正式企业版 Agent。
- 每次模型调用都会产生 API 成本。
- 当前工具只有只读能力，适合学习，不适合复杂业务自动化。
- 如果后续要贴近企业现场，还需要补日志、认证、权限、审计和失败重试。
