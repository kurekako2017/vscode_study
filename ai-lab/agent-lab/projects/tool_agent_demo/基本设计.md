# 基本设计

## 1. 系统概要

`tool_agent_demo` 是一个最小化的命令行 Tool Calling 示例。

系统由本地 Python 程序和模型调用两部分组成：

- 本地程序负责定义工具、执行工具、控制访问边界
- 模型负责理解任务、选择工具、基于工具结果输出回答

## 2. 系统构成

| 模块 | 作用 |
|---|---|
| 参数解析 | 接收 `prompt`、`model`、`workdir` |
| OpenAI Client 初始化 | 读取 `OPENAI_API_KEY` 并创建客户端 |
| 路径控制 | 保证所有文件访问限制在 `--workdir` 内 |
| 工具定义 | 定义 `list_files`、`read_file`、`search_text` schema |
| 工具执行 | 根据模型返回的函数名和参数执行本地工具 |
| Tool Loop | 循环处理函数调用，直到得到最终文本回答 |

## 3. 输入输出设计

### 3.1 输入

命令行输入参数如下：

- `prompt`
  - 用户任务说明
- `--model`
  - 使用的模型名
- `--workdir`
  - 工具允许访问的工作目录

### 3.2 输出

- 成功时输出最终文本回答
- 失败时输出错误信息并以非 0 状态结束

## 4. 工具设计

### 4.1 `list_files`

- 用途：列出指定相对路径下的文件和目录
- 输入：`path`
- 输出：文件名、类型

### 4.2 `read_file`

- 用途：读取 UTF-8 文本文件内容
- 输入：`path`
- 输出：文件内容

### 4.3 `search_text`

- 用途：在文件或目录下搜索关键字
- 输入：`query`、`path`
- 输出：命中的文件、行号、文本

## 5. 处理流程

整体处理流程如下：

1. 解析命令行参数
2. 校验 `OPENAI_API_KEY`
3. 校验 `--workdir`
4. 向模型发送用户任务和工具定义
5. 如果模型返回函数调用，则解析参数并执行本地工具
6. 将工具执行结果作为 `function_call_output` 回填给模型
7. 重复执行，直到得到最终文本回答
8. 输出回答并结束

## 6. 安全设计

本样例最重要的设计点之一是路径边界控制。

通过路径解析逻辑，所有工具访问都必须满足：

- 目标路径位于 `--workdir` 内
- 不允许通过相对路径跳出工作目录

这能帮助学习者理解：工具型 Agent 的关键不是“能做很多事”，而是“能力范围必须可控”。

## 7. 错误处理

当前版本对以下情况进行基础处理：

- 未设置 `OPENAI_API_KEY`
- `--workdir` 不存在或不是目录
- 路径越界
- 文件不存在
- 读取的文件不是 UTF-8 文本
- 工具名不存在
- 请求过程中出现异常

## 8. 设计取舍

为了保持最小可运行，当前版本有意简化了很多内容：

- 不做写操作
- 不做命令执行
- 不做数据库或网络工具
- 不做多轮计划拆解
- 不做任务状态持久化

这些取舍是为了先把“最小 Tool Calling 闭环”跑通。

## 9. 扩展方向

如果继续往更贴近现场的方向扩展，可以考虑：

- 增加只读 API 工具
- 增加数据库查询工具
- 增加日志和审计记录
- 增加步骤展示和执行记录
- 增加权限控制
- 增加任务计划和失败重试

## 10. 成本与注意点

- 当前版本是学习版和 PoC，不是正式企业交付物。
- 模型调用会产生 API 成本。
- 当前工具访问范围有限，适合学习，不适合直接上复杂生产场景。
- 如果要进入企业环境，还需要补认证、权限、日志、审计和运维设计。
