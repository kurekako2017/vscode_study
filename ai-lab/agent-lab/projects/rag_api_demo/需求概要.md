# rag_api_demo 需求概要

## 1. 背景

在日本 IT 现场，`RAG` 类应用不只需要命令行 PoC，很多时候还需要进一步变成可被其他系统调用的后端接口。

典型需求包括：

- 社内検索 API
- ナレッジ検索 API
- 生成 AI 功能的后端 PoC
- 面向前端、业务系统或内部工具的问答服务

## 2. 目的

本项目的目的是在 `doc_qa_agent` 的基础上，提供一个最小可运行的 `FastAPI` 版后端服务，实现：

- 文档问答能力的 API 化
- 健康检查
- 文档重载
- 适合后续扩展为企业型 `RAG` 服务的基础结构

这里的服务对象不是“程序自己”，而是外部客户端，例如 `curl`、浏览器、前端页面或其他后端系统。

## 3. 目标用户

- 想贴近案件要求学习 `FastAPI + RAG` 的工程师
- 需要构建生成 AI PoC 后端的开发者
- 需要做社内搜索接口试作的学习者

## 4. 使用场景

- 指定一个本地资料目录
- 服务在第一次访问 `/health`、`/ask` 或 `/reload` 时按需加载 `md / txt / pdf`
- 调用方通过 HTTP 接口提交问题
- 系统检索相关片段并生成回答
- 接口返回答案和来源信息

## 5. 功能要求

### 5.1 文档加载

- 支持读取 `md`
- 支持读取 `txt`
- 支持读取 `pdf`
- 启动后按需加载文档
- 可通过接口重新加载

### 5.2 问答接口

- 提供 `POST /ask`
- 接收问题文本
- 支持指定模型名
- 返回回答与来源

### 5.3 健康检查

- 提供 `GET /health`
- 返回运行状态
- 返回当前文档目录和片段数

### 5.4 文档重载

- 提供 `POST /reload`
- 重新加载文档并重建片段

## 6. 非功能要求

- 本地可运行
- 结构尽量简单
- 便于理解和后续扩展
- 不依赖外部向量库

## 7. 范围外事项

本样例暂不包含：

- 向量检索
- 用户认证
- 权限控制
- 检索质量评估
- 日志持久化
- 数据库存储

## 8. 前提条件

- Python 环境可用
- 如果要使用 Real 模式，`OPENAI_API_KEY` 已配置
- `RAG_API_DOCS_DIR` 指向有效目录
- 目录中至少存在可读资料文件

## 9. 预期成果

- 一个可启动的 `FastAPI` 服务
- 一个可通过 HTTP 调用的最小 `RAG API`
- 一个更贴近日本案件要求的后端 PoC 样例

## 10. 成功判定

满足以下条件可视为完成：

- `/health` 正常返回
- `/ask` 能返回回答
- `/reload` 能正常重载文档
- 能输出来源列表
- 支持 `pdf` 资料读取
