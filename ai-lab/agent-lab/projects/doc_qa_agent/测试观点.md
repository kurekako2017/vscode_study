# doc_qa_agent 测试观点

## 1. 目的

本文件用于整理 `doc_qa_agent` 的最小测试观点，确保该命令行版 `RAG` 样例至少能覆盖：

- 文档读取
- 文档切分
- 关键词检索
- 问答生成
- 来源返回
- 错误处理

## 2. 测试范围

- 命令行参数
- 文档目录读取
- `md / txt` 文件加载
- 关键词检索
- 回答生成
- 来源输出

## 3. 主要测试观点

### 3.1 参数相关

- `question` 必填
- `--docs` 指定有效目录时可执行
- `--model` 指定模型时可正常传递

### 3.2 文档读取

- 能读取 `md`
- 能读取 `txt`
- 能递归扫描子目录
- 非 UTF-8 文本可跳过

### 3.3 切分相关

- 文档可被切分为多个片段
- 每个片段带有来源标签

### 3.4 检索相关

- 问题和文档相关时可命中片段
- 命中结果按分数排序
- 无命中时能返回依据不足信息

### 3.5 回答相关

- 能基于命中片段生成回答
- 回答不应脱离上下文随意发挥
- 能输出来源片段列表

### 3.6 错误处理

- 文档目录不存在时报错
- 无可读文件时报错
- `OPENAI_API_KEY` 未配置时报错
- 模型调用失败时报错

## 4. 优先级建议

### 高优先级

- 文档能读取
- 能生成回答
- 能返回来源

### 中优先级

- 多文件命中排序
- 子目录读取
- 无命中时合理说明

### 低优先级

- 超长文档
- 特殊符号场景

## 5. 代价与注意点

- 这是学习用最小 `RAG`，不是正式企业版检索系统。
- 当前不支持 `PDF`，所以很多日本现场常见资料还无法直接纳入。
- 检索方式是关键词重叠，准确率和召回率都有限。
- 每次问答都会触发模型调用，会产生实际 API 成本。
- 当前没有日志持久化和评估机制，后续如果要贴近现场还需要补。
