# 功能使用与测试通用模板

> 这份模板适合直接复制到任何新项目里。  
> 它不绑定 `shopkeeper-agent-main`，专门用来写“某个功能怎么用、怎么测”。

## 1. 单个功能的标准写法

```markdown
# 功能名称

## 1. 功能定位
- 这个功能是做什么的
- 它解决什么问题
- 不用它会有什么影响

## 2. 适用场景
- 场景 1
- 场景 2
- 场景 3

## 3. 前置条件
- 需要哪些环境变量
- 需要启动哪些服务
- 需要准备哪些账号、密钥或数据

## 4. 使用方式
### 4.1 命令行
```bash
...
```

### 4.2 前端操作
1. 打开哪个页面
2. 点击什么按钮
3. 输入什么内容

### 4.3 API 调用
```bash
curl ...
```

## 5. 输入示例
- 示例 1
- 示例 2

## 6. 预期结果
- 正常结果应该长什么样
- 结果文件、页面或接口响应应该是什么

## 7. 测试方法
### 7.1 正常流程测试
1. ...
2. ...
3. ...

### 7.2 异常流程测试
1. 故意不配某个变量
2. 故意传错参数
3. 故意断开依赖服务

## 8. 验收标准
- 能否正常执行
- 能否返回正确结果
- 能否输出可追踪的日志或文件

## 9. 常见问题
- 问题 1
- 问题 2

## 10. 关联文件
- 接口文件
- 工具文件
- 前端文件
- 文档链接
```

## 2. 多功能文档写法

如果一个模块里包含多个功能，就按下面方式拆分：

```markdown
# 模块名称

## 1. 功能 A
...

## 2. 功能 B
...

## 3. 功能 C
...
```

每个功能都建议至少写清楚这些内容：

- 功能定位
- 适用场景
- 前置条件
- 使用方式
- 测试方法
- 验收标准

## 3. 适合直接复用的最小模板

如果你只想写最短版本，可以只保留这 6 段：

1. 功能定位
2. 适用场景
3. 前置条件
4. 使用方式
5. 测试方法
6. 验收标准

## 4. 建议

- 新建项目时，先写通用版。
- 当项目功能已经明确后，再补项目定制版。
- 如果功能和外部服务绑定很紧，建议把“异常流程测试”写得更细一些。
