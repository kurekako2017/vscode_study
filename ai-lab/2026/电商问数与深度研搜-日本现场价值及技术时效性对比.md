# 电商问数与深度研搜：日本现场价值及技术时效性对比

> 说明：本文是案例对比分析稿，聚焦“电商问数”与“深度研搜”的业务价值、技术路线和适用场景。当前 `ai-lab` 主线完成度与可运行实现，请以 [2026补齐实施状态.md](./2026补齐实施状态.md) 为准；本文中的版本与实现描述保留为 2026-06-21 左右的分析快照。

> 分析日期：2026-06-21  
> 分析对象：`ai-agents-from-zero/实战项目-电商问数/`、`ai-agents-from-zero/实战项目-深度研搜/`，以及对应的 `shopkeeper-agent-main/`、`deepsearch-agents-main/` 完整工程。  
> 目的：判断两个案例在日本现场项目中的实用性、业务价值、可迁移性、知识覆盖、工程成熟度和技术版本时效性，并给出学习顺序。


  核心结论：

  - 日本现场优先学习“电商问数”，业务落地更直接。
  - “深度研搜”适合作为第二个进阶项目，Agent 技术覆盖更广。
  - 两者都是 2026 新架构，但依赖版本并非当前最新。
  - 两个项目都明显缺少自动化测试、评估、认证、权限和生产治理。
  - 最佳作品集方向是将两者组合成“日本小売经营分析 Agent”。

## 一、结论先行

两个案例都值得学，但定位不同，不能简单判断谁全面就只学谁。

| 结论项 | 电商问数 | 深度研搜 |
| --- | --- | --- |
| 核心类型 | 受控的 Text-to-SQL / 数据分析工作流 | 开放式 Deep Research / 多智能体任务系统 |
| 日本现场直接实用性 | 高 | 中高 |
| 企业 PoC 成交与说明难度 | 较低，业务方容易理解 | 中等，需要先限定场景与治理边界 |
| 技术广度 | 中高 | 很高 |
| 业务确定性 | 较高 | 较低，受模型和外部信息影响更大 |
| 跨行业通用性 | 高，适合有结构化数据的行业 | 很高，适合调查、知识整合和报告工作 |
| 生产化距离 | 中等 | 中等偏远 |
| 推荐学习顺序 | 第一项目 | 第二项目 |

建议按以下顺序学习：

```text
电商问数
  先掌握可控图编排、企业数据、检索、SQL、API 和交付边界
      ↓
深度研搜
  再掌握多智能体、长任务、记忆、HITL、多来源工具和文件交付
```

如果目标是日本企业的 AI / DX / 数据活用项目，建议投入比例为：

- 电商问数：60%
- 深度研搜：40%

如果目标是 Agent 框架研发、AI 平台或技术预研岗位，可以反过来投入 40% / 60%，但仍建议先理解电商问数的确定性工作流。

## 二、为什么这两个案例在日本现场都有价值

IPA 的《DX 动向 2025》显示，日本企业的数据活用目的更偏向销售与服务水平提升、后台业务效率和生产率提升；同时，日本企业在数据管理系统、全社数据活用方针、标准化和人才方面仍有明显问题。大型企业的生成 AI 导入较快，但生成 AI 真正嵌入部门业务流程的比例仍偏低，治理和误答风险也是突出问题。[IPA《DX 动向 2025》](https://www.ipa.go.jp/digital/chousa/dx-trend/tbl5kb0000001mn2-att/dx-trend-data-collection-2025.pdf)

这意味着日本现场真正需要的不是只会调用模型，而是能够：

- 接入既存数据库、文件、知识库和业务 API。
- 把自然语言需求转换成受控流程。
- 说明数据来源、执行步骤、权限和错误处理。
- 先做小范围 PoC，再逐步嵌入部门业务。
- 用测试、日志、人工确认降低误答与误操作风险。

电商问数更直接对应“结构化数据活用”；深度研搜更直接对应“非结构化资料、多来源调查和报告自动化”。经济产业省的调查也强调，特定用途的生成 AI 更依赖场景相关的高质量数据，而不只是模型本身。[METI《生成AIに関する実態調査報告書》](https://www.meti.go.jp/policy/kyoso_seisaku/20250625_ai.pdf)

## 三、案例本质与架构对比

### 1. 电商问数

核心问题是：用户用自然语言提出经营问题，系统如何找到正确的表、字段、业务指标和字段值，生成 SQL，校验后查询数仓。

```text
自然语言问题
  → 关键词抽取
  → 字段 / 字段值 / 指标三路并行召回
  → 合并上下文
  → 表与指标过滤
  → 补充日期、数据库方言和 schema
  → 生成 SQL
  → EXPLAIN 校验
  → 修正 SQL
  → 执行查询
  → SSE 返回进度和结果
```

主要技术：

- LangGraph `StateGraph`
- LangChain 1.x
- MySQL / SQLAlchemy async
- Qdrant 向量检索
- Elasticsearch 字段值检索
- Embedding 与多路召回
- FastAPI + SSE
- React + TypeScript
- YAML / OmegaConf 配置
- Repository / Service / Dependency Injection 分层
- Docker Compose 基础环境

它本质上是一个“确定性图编排 + 少量 LLM 节点”的业务系统。路径由代码定义，模型主要参与关键词抽取、候选过滤、SQL 生成和修正。

### 2. 深度研搜

核心问题是：用户提出开放式研究任务后，主智能体如何规划、调用不同信息来源、读取附件、汇总结果并生成 Markdown/PDF。

```text
开放式任务
  → 主智能体规划
  → 网络搜索子智能体 / 数据库子智能体 / 知识库子智能体
  → 多轮工具调用与上下文管理
  → 汇总和写作
  → Markdown / PDF 交付
  → WebSocket 推送进度、结果和文件事件
```

主要技术：

- DeepAgents `create_deep_agent`
- LangChain 1.x / LangGraph runtime
- 主智能体与专业子智能体
- 同步/异步执行和流式事件
- Checkpointer、长期记忆与 Backend 教学示例
- Human-in-the-loop、中断与恢复教学示例
- Middleware、调用限制和 Skills 教学示例
- Tavily 网络搜索
- MySQL 工具
- 本地知识库检索
- 上传文件读取、Markdown/PDF 生成
- FastAPI + WebSocket + 后台任务取消
- React + TypeScript + Ant Design
- `ContextVar` 会话上下文隔离

它本质上是“Agent Harness + Orchestrator/Workers + 多工具”的长任务系统。执行路径由模型和框架共同决定，因此能力更开放，但结果、耗时和成本也更不稳定。

## 四、日本现场实用性比较

### 1. 电商问数：实用性高

最适合的日本现场：

- 小売、EC、物流：销售额、库存、地区、商品、客户分析。
- 制造：生产、品质、设备、采购、库存和交期查询。
- 金融/保险：经过严格权限控制后的统计和管理报表辅助。
- SIer / 情报系统部门：既存数据库的自然语言查询 PoC。
- BI / 数据平台团队：把指标口径和元数据接入对话入口。

它的现场价值不在“电商”两个字，而在以下通用结构：

```text
业务术语
  → 元数据和指标口径
  → schema linking
  → Text-to-SQL
  → 校验
  → 只读查询
  → 可追踪结果
```

日本企业常有大量既存系统、数据库和部门报表。这个项目能自然延伸到“売上照会”“在庫照会”“経営指標照会”“社内データ検索”，业务方也容易理解节省了什么工作。

局限是：真正上线时，SQL 安全和权限比生成准确率更重要。当前代码会把生成 SQL 交给数据库执行，虽然先使用 `EXPLAIN` 校验，但没有形成完整的 SELECT-only AST 校验、表/字段白名单、行级权限、查询成本限制和审计闭环。因此它适合学习和 PoC，不能原样连接生产数仓。

### 2. 深度研搜：实用性中高

最适合的日本现场：

- 调查、咨询、市场分析、竞争情报。
- 制药、制造、金融等行业的法规和技术资料整理。
- 提案书、稟議资料、会议前调查和报告初稿。
- 社内知识库 + 外部公开资料的联合调查。
- IT 运维、障害调查、FAQ 和操作手册汇总。
- 多步骤后台业务自动化 PoC。

它的优势是横向能力强：一个主智能体可以连接外部搜索、内部数据库、知识库和用户附件，最后生成交付文件。这比单一聊天机器人更能展示 Agent 的实际价值。

但日本现场通常会立即追问：

- 外部搜索数据能否发送给第三方服务？
- 内部文件和个人信息如何隔离？
- 报告中的事实、引用和更新时间如何验证？
- 长任务能否持久化、恢复和审计？
- 单次任务的 token、API 调用和费用上限是多少？
- 哪些工具必须经过人工批准？

当前完整项目实现了会话目录、异步后台任务、取消、日志和 WebSocket 事件，但主检查点仍是 `InMemorySaver`，活跃任务也保存在进程内；用户认证、多租户、持久任务队列、全量审计、系统化评测和生产监控尚未实现。仓库 README 也明确把这些列为能力边界。

## 五、业务价值比较

| 维度 | 电商问数 | 深度研搜 |
| --- | --- | --- |
| 可量化收益 | 高：减少 SQL/报表请求和分析等待 | 中高：减少检索、阅读和报告整理时间 |
| 业务方理解成本 | 低：输入问题、得到数据结果 | 中：需要解释 Agent 计划、来源和可信度 |
| 输出确定性 | 中高：结果最终来自数据库 | 中低：多来源、模型写作带来波动 |
| 数据时效性 | 高：查询实时或准实时数据库 | 取决于网络搜索、知识库更新时间 |
| 错误影响 | SQL 查错、指标口径错，可能影响经营判断 | 事实错误、引用错误、遗漏，可能影响报告决策 |
| ROI 验证方式 | 查询准确率、SQL 执行成功率、节省工时 | 任务完成率、引用正确率、报告质量、节省工时 |
| PoC 范围控制 | 容易：限定库表和问题集 | 较难：必须限定工具、来源和任务类型 |
| 面试说明价值 | 很高：业务、数据、后端、Agent 都能讲 | 很高：多 Agent、长任务、工具与实时 UI 都能讲 |

对日本 SIer / SES 案件而言，电商问数更容易转化为具体要件定义和测试用例；深度研搜更适合作为技术提案、先进 PoC 或 Agent 平台能力展示。

## 六、可比性与互补关系

### 可以直接比较的共同部分

两者约有 35%—45% 的知识可以横向比较：

- Python 3.12 和异步编程。
- LangChain 1.x、LangGraph runtime。
- MySQL 工具或仓储访问。
- FastAPI 后端。
- React/Vite 前端。
- 流式进度展示。
- 配置、日志、会话上下文和模型供应商接入。
- LLM 调用失败、外部依赖失败和异常处理。

适合做对比学习的问题：

- 固定图与模型自主规划怎样取舍？
- SSE 与 WebSocket 各适合什么场景？
- Repository 型数据访问与 Tool 型数据访问有什么差异？
- 状态 schema 与 Agent message/context 如何管理？
- 确定性流程怎样测试，开放式 Agent 怎样评估？

### 不宜直接比较的部分

- 电商问数解决的是受约束分析查询，深度研搜解决的是开放式长任务。
- 电商问数的核心质量是 schema linking、指标口径和 SQL 正确性。
- 深度研搜的核心质量是任务规划、路由、来源覆盖、事实性和报告质量。
- 电商问数适合明确图结构；深度研搜适合动态工具调用和上下文隔离。

因此，不应因为深度研搜用了更多 Agent 能力，就认为它是电商问数的“高级替代版”。对于高风险数据查询，固定 LangGraph 工作流通常比自主多智能体更合适。

## 七、通用性比较

### 电商问数的通用性

业务名称可替换，但架构高度通用：

| 原场景 | 可迁移场景 |
| --- | --- |
| 商品、订单、销售额 | 制造产品、生产实绩、不良率 |
| 地区销售 | 门店、支店、工厂、営業所分析 |
| 库存 | 部品、原材料、药品、备件库存 |
| 业务指标 | KPI、原価、稼働率、納期遵守率 |
| 电商数仓 | 企业 DWH、Data Mart、BI 语义层 |

迁移成本主要在元数据、业务指标、表关系、同义词和权限规则，不在 LangGraph 本身。

### 深度研搜的通用性

工具替换后可迁移到多种知识劳动场景：

| 原工具 | 可替换对象 |
| --- | --- |
| Tavily | 企业许可的搜索 API、新闻库、法规库 |
| MySQL | PostgreSQL、数据 API、业务系统 API |
| 本地知识库 | SharePoint、Box、Confluence、社内 Wiki |
| 文件上传 | 稟議书、规格书、议事录、合同、调查资料 |
| Markdown/PDF | Word、PowerPoint、Excel、工单或邮件草稿 |

深度研搜的框架通用性高于电商问数，但每增加一个工具，都要新增权限、数据边界、失败模式和评测成本。

### 覆盖结论

- 电商问数的知识覆盖更“纵深”：围绕企业数据查询完成从元数据到 SQL 的完整链路。
- 深度研搜的知识覆盖更“横向”：覆盖多智能体、工具、文件、长任务和实时前端。
- 两者一起学习能覆盖 2026 路线的大部分 Agent 应用开发知识，但仍不能替代测试、评估、安全、云部署和运维专题。

## 九、技术版本与“是否最新”判断

> 这里的版本判断仅作为 2026-06-21 的分析快照，不代表当前仓库中的路线完成度，也不作为后续开发验收依据。

- LangGraph 最新稳定版为 `1.2.4`（2026-06-02）。[LangGraph PyPI](https://pypi.org/project/langgraph/)

### 2. 技术路线是否过时

不算过时。两个项目都采用 2026 年仍然有效的主流路线：

- LangChain 1.x。
- LangGraph 1.x 的状态图和持久执行思想。
- FastAPI 异步接口。
- React 19。
- 结构化数据、向量检索、全文检索和流式 UI。
- DeepAgents 的规划、子智能体、文件系统、长期记忆和 HITL。

LangChain 官方仍将 DeepAgents 定位为适合复杂、多步骤、上下文量大、需要子智能体和跨会话记忆的 Agent Harness，并说明其底层使用 LangGraph 提供持久执行、流式输出和 HITL。[Deep Agents 官方概览](https://docs.langchain.com/oss/python/deepagents/)

### 3. 为什么不能称为“最新版本案例”

- 两个项目的锁文件都落后于 2026-06-21 的最新稳定版本。
- DeepAgents 在 2026 年 5—6 月快速迭代，从 `0.5.7` 升到 `0.6.8`，API 和默认行为存在变化风险。
- 深度研搜教程部分章节仍使用“RAGFlow 子智能体”的名称，而当前完整工程和 README 已改为本地知识库实现；学习时要区分教程演进版本与当前成品。
- 深度研搜教程对 Backend、HITL、Middleware 和 Skills 的覆盖强于最终成品集成，不能把章节示例能力全部视为成品能力。
- 电商问数 `pyproject.toml` 使用宽松的 `>=` 约束，但 `uv.lock` 固定在较旧小版本；直接重新锁依赖可能跨 minor 升级，需要运行回归测试。

正确表述应是：

> 两者都是 2026 年的新架构案例，技术思想和主要 API 代际仍然有效；但依赖不是分析日的最新稳定版本，升级前必须建立测试并逐项验证。

## 十、工程质量与生产差距

### 电商问数需要补强

P0：

- SQL 只允许单条 `SELECT` / CTE 查询。
- 使用 SQL parser/AST 校验，禁止 DDL、DML、存储过程和多语句。
- 数据库账号必须只读，限制 schema、表、字段和行权限。
- 设置超时、最大扫描量、最大返回行数和并发限制。
- 对个人信息和敏感字段做 masking。
- 保存用户问题、候选 SQL、修正记录、最终 SQL、耗时和操作者审计日志。
- 建立 NL2SQL 评测集：执行正确率、结果正确率、指标口径一致率。

P1：

- 增加日语业务术语、片假名、缩写、全半角和同义词处理。
- 将电商指标替换为日本现场常见的売上高、粗利、在庫回転率、欠品率等。
- 增加 SSO、角色权限和租户隔离。
- 增加 pytest、数据库 fixture、Golden SQL 和 API 回归测试。
- 增加 LangSmith/OpenTelemetry 或等价 tracing。

### 深度研搜需要补强

P0：

- 用户认证、角色权限、会话和文件的所有权校验。
- 文件类型、大小、恶意内容和路径穿越防护。
- 外部搜索、数据库查询、文件写入等工具的独立权限策略。
- 高风险工具使用 HITL；官方也明确要求 DeepAgents 的安全边界应落实在工具和 sandbox，而不是信任模型自我约束。[DeepAgents PyPI 安全说明](https://pypi.org/project/deepagents/)
- 把 `InMemorySaver` 换成持久 checkpointer；任务状态和事件持久化。
- 使用任务队列或 durable execution 替代单进程 `asyncio.create_task`。
- 设置最大模型调用、最大工具调用、超时、取消和预算上限。
- 建立事实性、引用正确率、来源覆盖率和报告完整度评测集。

P1：

- 将 Tavily 替换或封装成企业允许的搜索服务。
- 对内部资料和外部资料做明确标签，输出中保留引用时间和 URL/文档 ID。
- 增加 prompt injection 和恶意文档测试。
- 增加用户级、组织级 memory namespace；官方新版已强调按用户隔离记忆。[Deep Agents 官方 Memory](https://docs.langchain.com/oss/python/deepagents/memory)
- 使用官方较新的 subagent 结构化输出、permissions 和独立 middleware 能力，但需在升级后验证。[Deep Agents 官方 Subagents](https://docs.langchain.com/oss/python/deepagents/subagents)

## 十一、推荐学习路线

### 阶段 1：先学电商问数

重点不是逐行照抄，而是掌握四条主线：

1. 业务线：事实表、维度表、指标口径和元数据。
2. 检索线：Qdrant 字段/指标召回 + Elasticsearch 字段值召回。
3. Agent 线：LangGraph state、并行节点、汇合、条件边和修正闭环。
4. 工程线：Repository、Service、FastAPI Depends、SSE、React 和 Docker。

学习验收：

- 能画出完整节点图并说明每个节点为什么由代码控制。
- 能新增一个业务指标和一张维度表。
- 能解释错误 SQL 如何进入修正分支。
- 能加入 SELECT-only、limit、timeout 和只读账号。
- 能准备 20 条中日双语问题和预期结果进行回归。

### 阶段 2：再学深度研搜

先学章节 1—7 的框架能力，再学章节 8—14 的完整工程：

1. `create_deep_agent`、stream/astream 和 chunk。
2. 子智能体职责、上下文隔离和异步执行。
3. LangGraph/LangChain 子智能体包装。
4. HITL、中断和恢复。
5. Backend、长期记忆和 namespace。
6. Middleware、调用限制和 Skills。
7. 网络、数据库、知识库、附件和文件交付工具。
8. FastAPI 后台任务、WebSocket、取消和前端事件流。

学习验收：

- 能说明为什么简单任务不应该使用多 Agent。
- 能新增一个日本法规/社内 Wiki 子智能体。
- 能给数据库写操作和文件生成加入人工审批。
- 能把任务从内存状态改成持久状态，并在重启后恢复。
- 能对报告中的每个核心结论保存来源和检索时间。
- 能限制模型调用次数、工具调用次数、总时间和预算。

### 阶段 3：做一个组合改造项目

推荐作品集题目：

```text
日本小売经营分析 Agent

结构化数据：
  电商问数负责销售、库存、地区和商品指标查询

非结构化信息：
  深度研搜负责市场趋势、竞争情报、社内资料和附件分析

最终交付：
  生成带数据库结果、外部引用、风险提示和人工确认记录的日文报告
```

这个组合比单独展示任一原项目更适合日本现场：既有可验证数据，又有调查和报告能力，同时能展示何时使用固定图、何时使用自主 Agent。
