## 1. 项目定位

本系统是面向日本企业内部知识工作的“社内文書 RAG + 承認 Workflow Agent”。它以企业受控文档为证据生成回答草案，并依据问题主题、检索证据和业务规则决定能否直接发布。高风险问题必须由指定承認者审核，系统保存从提问到最终决定的完整审计链。

系统价值不以回答文字是否流畅衡量，而以以下结果衡量：

- 员工能否快速定位现行且适用的内部规定。
- 回答能否展示文档、版本、章节和检索依据。
- 高风险回答能否在发布前被正确拦截。
- 审批者能否理解风险、证据和变更内容后作出决定。
- 情シス和管理者能否还原谁在何时基于什么证据作出什么决定。

## 3. 业务背景

日本企业内部资料往往分散在文件服务器、门户、工单系统、会议记录和部门共享目录中。相同主题可能存在多个版本，员工难以判断哪个版本有效。一般问答可以通过检索提升效率，但契約、個人情報、セキュリティ、経費、法務和障害対応等内容若未经确认直接发布，可能造成合规、财务或运营影响。

因此，本系统把知识检索和审批治理合并为一条受控流程：低风险且证据充分的问答可自动完成，高风险或证据不足的问题转交负责人，不以生成能力绕过现有组织责任。

## 4. 用户角色

| 角色 | 日文职责 | 系统权限边界 |
| --- | --- | --- |
| 一般社員 | 質問者 | 提交问题、查看自己有权访问的引用与回答、补充信息 |
| 部門担当者 | 文書・業務担当 | 维护部门文档元数据、确认业务适用性、处理一般差戻し |
| 承認者 | リスク領域責任者 | 审核指定领域的高风险草案，承認、差戻し或却下 |
| 情シス | システム運用担当 | 配置数据源、监控服务、处理故障；默认不能代替业务审批 |
| 管理者 | システム管理者 | 管理用户、角色、策略和保留期；高权限操作同样被审计 |

角色不等同于个人。一个用户可以具备多个角色，但每次动作必须记录当时生效的角色、部门、授权范围和策略版本。

## 5. 核心业务流程

1. 一般社員提交问题，系统建立 `question_id`、`workflow_id` 和 `request_id`。
2. 权限过滤器先确定该用户可检索的文档范围。
3. Query Analyzer 识别主题、部门、时间范围和风险关键词。
4. Retriever 取得候选文档片段，Reranker 重新排序。
5. Context Builder 去重、控制长度并保留 source/version/section。
6. Static Answer Provider 在当前阶段生成带引用的固定格式草案。
7. Risk Classifier 综合主题规则、证据置信度和动作影响输出风险等级与理由。
8. 低风险且证据充分时进入发布节点；高风险、证据不足或分类不确定时进入承認待ち。
9. 承認者查看问题、草案、引用、风险理由和历史决定，选择承認、差戻し或却下。
10. 承認后 Workflow 从 checkpoint 恢复并发布正式回答；差戻し回到草案修订；却下终止发布。
11. SSE 向有权限的客户端发送状态变化，Audit Log 保存所有关键动作。

## 6. 技术栈

| 技术 | 责任 | 当前阶段策略 |
| --- | --- | --- |
| React | 提问、进度、审批、回答与引用界面 | 下一阶段建立最小页面 |
| FastAPI | typed API、依赖注入、SSE、健康检查 | 下一阶段建立 HTTP 边界 |
| LangGraph | 显式 State、条件路由、interrupt/resume | 下一阶段实现单一受控图 |
| RAG | 基于企业文档形成证据上下文 | 使用固定本地语料验证流程 |
| Retriever | 权限过滤后的候选检索 | 先实现 SQLite/内存关键词检索 |
| Rerank | 对候选片段进行二次排序 | 先用确定性规则评分 |
| Approval Workflow | 人工审批、差戻し、却下、恢复 | 使用持久 checkpoint 和版本校验 |
| SSE | 单向推送任务及审批进度 | 事件只含最小必要字段 |
| SQLite | 本地任务、文档元数据、审批和审计存储 | 单进程基线 |
| Docker | 一致的本地构建和运行环境 | 后续提供 Compose 拓扑 |

本阶段不接真实 LLM，不调用 OpenAI API，不配置 API Key。

## 8. Production Gap

| 能力 | 当前缺口 | 生产要求 |
| --- | --- | --- |
| SSO | 无企业身份联合 | 对接 Entra ID、Okta 或企业 IdP，支持 OIDC/SAML |
| RBAC | 只有角色模型设计 | 实现角色、部门、资源范围和策略版本校验 |
| Audit Log | 只有事件模型设计 | 追加写、完整性保护、长期保留、受控查询 |
| VectorDB | 未接入向量索引 | 根据数据规模、日文检索质量和运维能力选型 |
| OpenSearch | 未接入全文检索 | 建立日文 analyzer、索引生命周期和 ACL 过滤 |
| PostgreSQL | 当前规划 SQLite | 支持并发事务、HA、备份恢复和迁移 |
| Redis | 无缓存和分布式协调 | 仅在测量到热点、锁或事件分发需求后引入 |
| Queue | 后台任务仍未实现 | 增加可靠投递、重试、死信、幂等和背压 |
| OpenTelemetry | 未建立跨服务 trace | 统一 trace/log/metric，定义 SLO 和告警 |

此外还缺少：多租户隔离、数据分类、DLP、密钥管理、文档病毒扫描、提示注入防御、离线检索评价、灾备、容量测试、供应链安全和正式变更管理。

## 10. 统一术语

| 标准术语 | 日文/英文 | 定义 | 禁止混用 |
| --- | --- | --- | --- |
| 问题 | 質問 / Question | 用户提交的原始查询 | 不称 Prompt |
| 回答草案 | 回答案 / Draft Answer | 尚未正式发布的回答 | 不称正式回答 |
| 正式回答 | 正式回答 / Final Answer | 通过自动发布条件或人工批准的结果 | 不称模型原文 |
| 文档 | 社内文書 / Document | 有权限、版本和有效期的知识来源 | 不称任意文件 |
| 文档片段 | Chunk | 可检索且保留来源坐标的最小单元 | 不称 Answer |
| 检索 | Retrieval | 在 ACL 约束内寻找候选片段 | 不等同生成 |
| 再排序 | Rerank | 对候选片段二次评分排序 | 不等同检索 |
| 引用 | Citation | 文档 ID、版本、章节和片段位置 | 不只保存 URL |
| 风险等级 | Risk Level | low/medium/high/critical | 不称模型感觉 |
| 承認 | Approval | 有权角色允许草案发布 | 不与 review 混用 |
| 差戻し | Return for Revision | 要求补充证据或修改草案 | 不等同却下 |
| 却下 | Rejection | 禁止当前草案发布 | 不等同失败 |
| 审计记录 | 監査ログ / Audit Record | 可追溯业务行为证据 | 不等同运行日志 |
| 运行日志 | Application Log | 用于故障诊断的技术记录 | 不替代审计记录 |
| Workflow | ワークフロー | 有状态、可恢复的业务执行图 | 不称简单函数链 |

## 11. 系统边界与不变量

以下规则在实现中不可被绕过：

1. 检索前执行 ACL 过滤，禁止“先检索再隐藏”。
2. 无引用、引用不可访问、文档失效或置信度不足时，不得形成确定性正式回答。
3. high/critical 风险必须人工审批；分类异常按高风险处理。
4. 审批者不能审批自己无权负责的领域；自问自批由策略明确禁止。
5. 审批决定必须绑定草案版本。草案变化后原批准失效。
6. Workflow 恢复操作必须幂等，重复回调不能重复发布回答。
7. SSE 不是事实来源；客户端重连后从持久事件和状态恢复。
8. Audit Record 不允许普通业务更新或物理覆盖。
9. 日志、事件和审计记录不得保存密钥或不必要的敏感正文。
10. 任何 Provider 超时或失败都必须收敛到可解释状态，并提供人工路径。

## 12. 验收基准

- 能以固定输入演示低风险自动完成和高风险等待审批两条路径。
- 每个回答都包含用户有权访问的引用。
- 承認、差戻し、却下均可从 checkpoint 正确恢复或终止。
- 重复审批请求不产生重复正式回答。
- SSE 断线重连不改变业务事实。
- request_id、workflow_id、question_id、approval_id 可关联日志与审计记录。
- 当前范围和 Production Gap 在界面、文档和面试说明中保持一致。
