## 1. 项目概要

本项目为日本企业构建社内知识检索与回答审批系统。一般社員从统一入口提出业务问题，系统在其权限范围内检索社内文書，对候选片段进行再排序，形成带引用的回答草案。涉及契約、個人情報、セキュリティ、経費、法務和障害対応时，Workflow 暂停并向相应承認者发起审批。只有满足自动发布条件或取得有效批准后，系统才返回正式回答。

项目不是单纯的聊天界面。业务交付对象包括检索证据、风险理由、审批状态、正式回答和审计轨迹。

### 现场范围定义

| 项目 | 范围 |
| --- | --- |
| 対象部門 | 总务、人事、法务、财务、信息系统及业务部门 |
| 対象利用者 | 一般社員、部門担当者、承認者、情シス、管理者 |
| 対象文書 | 有所有者、版本、有效期和访问级别的内部资料 |
| 対象処理 | 提问、检索、草案、风险判断、审批、发布、审计 |
| 対象外 | 自动签约、自动付款、自动变更生产系统、无审批的高风险建议 |

## 2. 业务背景

客户内部知识分散在多个部门和存储位置。员工通常通过询问熟悉业务的人员、搜索共享目录或参考旧邮件获取答案，导致响应时间依赖个人经验。与此同时，内部规定会改版，旧文档仍可能被检索到；部分员工也可能没有访问某类文档的权限。

对于普通操作问题，统一检索可以缩短查找时间。对于高风险问题，若系统仅追求自动回答率，可能把过期条款、缺少上下文的操作步骤或受限信息当作正式依据。项目因此采用“低风险自动化、高风险人工确认、全流程可追溯”的控制模式。

## 3. 客户课题

### 3.1 业务课题

- 社内規程和手順書分散，查找耗时。
- 同名文档存在多个版本，员工难以判断现行版。
- FAQ 更新滞后，重复问题持续占用担当者时间。
- 障害票和議事録含有重要经验，但缺少统一检索入口。
- 高风险回答依赖口头确认，审批过程和责任边界不清楚。

### 3.2 系统课题

- 文档访问权限无法仅靠回答页面隐藏，必须在检索阶段实施。
- 关键词搜索对日文同义词、复合词和上下文支持不足。
- 生成结果可能缺少依据或混用不同版本。
- 长流程需要处理中断、恢复、重复请求和审批超时。
- 运行日志无法回答“谁批准了哪一版草案”。

### 3.3 管理课题

- 缺少检索命中率、拒答率、审批等待时间等指标。
- 无法识别长期没有维护的知识领域。
- 发生误回答时，难以还原当时使用的文档和策略版本。

## 4. 系统目标

| 目标 | 评价指标示例 |
| --- | --- |
| 缩短资料查找时间 | 问题提交至草案生成的 P50/P95 时间 |
| 提高回答可验证性 | 带有效引用的回答比例、引用可访问率 |
| 防止高风险直出 | high/critical 未审批发布数必须为 0 |
| 提高审批效率 | 等待时间、差戻し率、超时件数 |
| 改善知识维护 | 零命中文档领域、过期文档引用拦截数 |
| 支持事后追溯 | 审计链完整率、按 workflow_id 还原成功率 |

目标之间可能冲突。例如，提高自动回答率不能牺牲风险控制；减少延迟不能跳过 ACL 或 Rerank。上线判断应同时看效率、质量、安全和运营负荷。

## 5. 用户角色

### 一般社員

提交问题、查看本人可访问的引用和正式回答。不能查看其它部门的受限文档，也不能把草案当作正式回答。

### 部門担当者

维护文档分类、所有者、有效期和适用范围；对差戻し问题补充资料。不能代替法务或安全承認者批准不属于本部门的高风险内容。

### 承認者

按风险领域审核草案。审批画面必须显示问题、草案版本、引用、风险理由、相关历史和截止时间。决定包括承認、差戻し和却下。

### 情シス

负责服务运行、数据源连接、故障处理和性能监控。情シス可维护系统，但不因技术管理员身份自动取得所有业务文档的阅读权限。

### 管理者

管理用户、角色、策略、保留期和紧急停止。管理动作同样进入 Audit Log，并接受最小权限和职责分离约束。

## 6. 文档类型

| 文档类型 | 主要用途 | 典型风险 | 必要元数据 |
| --- | --- | --- | --- |
| 社内規程 | 公司规则与控制要求 | 版本过期、适用范围错误 | document_id、version、effective_from/to、owner |
| 業務マニュアル | 标准业务操作 | 步骤遗漏、部门差异 | department、process、revision |
| FAQ | 高频问答 | 依据不足、更新滞后 | category、reviewed_at、source_refs |
| 障害票 | 故障现象与处理记录 | 暴露系统细节、照搬临时措施 | severity、system、closed_at、visibility |
| 議事録 | 决策和讨论背景 | 非正式结论被误当规则 | meeting_date、participants、decision_status |
| 手順書 | 具体操作步骤 | 生产变更、安全影响 | environment、approver、rollback |
| 申請書類 | 申请条件与格式 | 个人信息、财务数据 | form_type、retention、confidentiality |

导入时必须区分“可作为正式依据的文档”和“仅供参考的记录”。議事録和障害票默认不能覆盖现行社内規程。

## 7. RAG 检索流程

1. **Query normalization**：保留原问题，生成规范化查询；识别部门、系统、日期和主题。
2. **ACL filter**：根据用户、部门、角色、文档等级和有效期构造过滤条件。
3. **Retrieve**：从当前可用索引中取得候选片段。当前最小实现采用 SQLite 关键词匹配。
4. **Rerank**：综合关键词覆盖、文档类型优先级、版本有效性和章节匹配重新评分。
5. **Deduplicate**：合并相同文档的重叠片段，避免上下文重复。
6. **Evidence gate**：检查最低分数、引用数量、版本一致性和用户可访问性。
7. **Context build**：在长度预算内生成结构化上下文，每段保留 citation 坐标。

关键控制点：ACL 必须在 Retrieve 前或检索查询内部生效；Rerank 不能把无权限候选重新带回结果；Context Builder 不接受无 source_id 的文本。

## 8. 承認 Workflow

### 状态

```text
received → retrieving → drafting → classifying
  ├─ low + sufficient evidence → publishing → completed
  └─ high / uncertain / insufficient evidence → approval_pending
       ├─ approved → publishing → completed
       ├─ returned → drafting（新版本）
       ├─ rejected → rejected
       └─ expired → escalated / cancelled
```

### 审批输入

- 原始问题与提问者部门。
- 回答草案及 `draft_version`。
- 引用文档、版本、章节和摘要。
- 风险等级、命中规则和分类理由。
- 建议审批角色与截止时间。
- 历史差戻し和修改差异。

### 审批控制

- 决定必须绑定 `approval_id`、`workflow_id`、`draft_version` 和策略版本。
- 只有责任领域与资源范围同时匹配的用户可以决定。
- 重复提交同一决定返回原结果，不重复发布。
- 草案被修改后旧批准失效。
- 超时不自动视为批准；按策略升级或取消。

## 9. 高风险问题判定

### 分类维度

| 维度 | 示例 | 处理 |
| --- | --- | --- |
| 主题 | 契約、個人情報、セキュリティ、経費、法務、障害対応 | 至少 medium，按规则进入指定审批组 |
| 动作影响 | 签署、付款、删除、权限变更、生产操作 | high/critical |
| 数据等级 | 个人、机密、客户数据 | high |
| 证据质量 | 无命中、低分、版本冲突 | 转人工或拒答 |
| 时间敏感 | 法规、紧急故障、临时措施 | 要求有效期和负责人确认 |
| 分类不确定 | 规则冲突或异常 | fail closed，进入审批 |

当前阶段使用可解释规则，不让回答生成器自行决定是否需要审批。未来即使引入分类模型，也必须保留规则上限、理由记录、离线评价和人工兜底。

## 10. 回答生成流程

1. Context Builder 输出结构化证据列表。
2. Answer Provider 仅依据证据生成草案，当前使用固定模板。
3. Citation Validator 检查每个关键结论是否能映射到 citation。
4. Risk Classifier 在草案和原问题基础上执行独立判断。
5. Policy Engine 决定自动发布、等待审批或拒答。
6. 发布节点写入不可变的 Final Answer 版本。

回答格式至少包含：结论、适用条件、操作步骤、注意事项、引用、文档版本和“需要担当者确认”的明确标识。证据不足时不补造事实，而是列出缺少的信息。

## 11. SSE 进度通知

建议事件类型：

| event | 用途 | 关键字段 |
| --- | --- | --- |
| `status` | received/retrieving/drafting 等普通进度 | sequence、workflow_id、status、message |
| `approval_required` | 进入等待审批 | approval_id、risk_level、due_at |
| `approval_updated` | 承認/差戻し/却下结果 | decision、draft_version |
| `done` | 正式回答可读取 | answer_path、completed_at |
| `error` | 流程失败 | error_code、retryable |

SSE 是通知通道，不是业务事实存储。事件必须持久化并带 sequence；客户端用 Last-Event-ID 或 `after` 参数重连。事件不得包含完整受限文档正文。

## 12. Audit Log

### 审计对象

- 问题创建和访问范围。
- 文档导入、更新、停用和权限变化。
- Retriever/Rerank 使用的版本与结果摘要。
- 风险规则、策略版本和判定理由。
- 草案创建、修改和正式发布。
- 承認、差戻し、却下、超时和管理员操作。

### 最小字段

`audit_id`、`occurred_at`、`actor_id`、`actor_role`、`action`、`resource_type`、`resource_id`、`workflow_id`、`before_hash`、`after_hash`、`policy_version`、`request_id`、`result`。

Audit Log 采用追加写语义，业务表更新不能覆盖历史。敏感正文只保存哈希或受控引用；审计查询本身也必须记录。

## 13. Production Gap

当前架构资产和后续本地垂直切片与生产系统之间仍有明确差距：

- 身份：SSO、MFA、离职停权、服务账号治理。
- 权限：资源级 RBAC/ABAC、文档 ACL 同步、职责分离。
- 数据：PostgreSQL、VectorDB、OpenSearch、备份恢复、加密与保留期。
- 异步：Queue、幂等消费、重试、死信、优先级和背压。
- 观测：OpenTelemetry、SLO、告警、容量和成本监控。
- 安全：提示注入、恶意文件、DLP、密钥管理、供应链扫描。
- 质量：日文检索基准集、Rerank 指标、引用正确性、风险分类召回率。
- 运维：多环境、CI/CD、灰度、回滚、灾备和审计导出。

## 14. 日本现场面试介绍

### 60 秒版本（日语）

> 社内規程や業務マニュアルなどを権限範囲内で検索し、根拠付きの回答ドラフトを作成するシステムです。契約、個人情報、セキュリティなどの高リスク質問は、LangGraph のワークフローを中断し、担当承認者の確認後に再開します。検索では Retriever と Reranker を分離し、回答・引用・リスク判定・承認履歴を監査可能な形で管理します。現段階では外部 LLM を利用せず、固定 Provider と SQLite を前提に API、SSE、承認状態遷移の境界を設計しています。本番化には SSO、RBAC、OpenSearch、PostgreSQL、Queue、OpenTelemetry などが必要だと整理しています。

### 说明重点

1. 先讲客户问题，不先罗列框架名。
2. 强调高风险不直接发布和“为什么”。
3. 说明自己负责的边界：Workflow、RAG、审批状态、审计。
4. 主动区分当前范围和生产差距。
5. 用指标说明成功标准，而不是只说“提高效率”。

## 15. TL 追问

| TL 追问 | 回答要点 |
| --- | --- |
| 为什么不让生成器判断风险？ | 职责分离；风险需稳定、可解释、可测试，异常时 fail closed |
| ACL 在哪里执行？ | Query 构造和检索引擎过滤阶段；Rerank/生成不能恢复被排除内容 |
| 审批后如何恢复？ | 持久 checkpoint + approval token/version + 幂等 resume |
| SSE 断线会丢状态吗？ | SSE 仅通知；事件持久化，客户端按 sequence 续传，API 可查当前状态 |
| 如何防止审批旧草案？ | 决定绑定 draft_version；草案变化后旧审批自动失效 |
| SQLite 的限制是什么？ | 单机并发、HA、迁移和锁竞争；只作为当前本地基线 |
| 检索质量怎么测？ | Recall@K、MRR/NDCG、引用正确率、无答案识别，按文档类型分层 |
| 高风险分类怎么测？ | 以漏判成本优先看 recall，建立边界样例和策略回归集 |
| 什么时候需要 Queue？ | 长任务、跨实例、可靠重试和背压成为明确需求时 |
| Audit Log 与日志区别？ | 前者是业务证据、追加写和长期保留；后者用于技术诊断，可采样和轮转 |
| 如何应对提示注入？ | 文档视为不可信数据、指令隔离、工具白名单、输出校验、敏感动作审批 |
| 为什么不拆多个 Agent？ | 当前责任可由单一显式图和确定性组件表达；拆分会增加上下文、权限和运维复杂度 |
