# 04_日本TL_CodeReview

## 命名

名称必须体现业务含义，不能只体现技术实现。

## 架构

检查职责分离、依赖方向、開発と運用版边界。

## RAG

检查 Chunk、TopK、Citation、ACL、评估和无答案策略。

## API

检查 schema、错误码、Health、Reload、认证、日志、超时。

## Streaming

检查 status/token/error/done、断线、心跳、代理缓冲。

## LangGraph

检查状态、节点、边、checkpoint、终止条件、人工审批。

## MCP

检查 Tool allowlist、权限、审批、审计、幂等。

## Docker / 安全 / 性能 / 日志 / 异常 / 缓存

进入小売業向け AI 経営分析システム之前，这些必须作为 Review checklist。

# Agent / LangGraph Code Review 5 条

> 定位：本文件属于小売業向け AI 経営分析システム开发知识库。内容用于架构讨论、日本现场面试准备和 TL Review。


## REV-AGENT-001 Agent / LangGraph Review 1

### Review 视角

如果我是日本现场 TL，我会检查这段实现是否能从学习版进入運用版，是否有明确边界，是否会在改修、保守、障害対応中变成风险。

### 风险等级

Medium

### 关联项目

PRJ-010

### 关联知识点

KN-AGENT-003

### 问题说明

当前教学版为了理解流程会简化配置、日志、异常、缓存、测试和权限。进入小売業向け AI 経営分析システム前必须明确这些简化点。

### 为什么在小売業向け AI 経営分析システム中是风险

日本项目通常多人维护，后续会有レビュー票、障害票、改修和交接。如果设计理由不清楚，维护成本会迅速上升。

### 日本 TL 会怎么指出

“この実装は基本構成として整理できています。運用設計では、ログ、例外処理、設定外出し、テスト観点を追加してください。”

### 修改建议

把硬编码参数外部化，增加结构化日志，补充异常分支，补充测试用例，记录设计决策。

### 運用版改法

拆分 service/repository/config/schema 层，引入监控、鉴权、缓存、CI 测试和可回滚部署。
