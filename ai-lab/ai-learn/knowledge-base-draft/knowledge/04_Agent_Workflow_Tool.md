# KN-AGENT-001 Tool Calling

> 定位：本文件属于小売業向け AI 経営分析システム开发知识库。内容用于架构讨论、日本现场面试准备和 TL Review。


## 元数据

- ID: KN-AGENT-001
- 领域: Agent
- 状态: 初版
- 关联项目: PRJ-005, PRJ-025, PRJ-027
- 关联设计决策: DES-MCP-002

## 5分钟定位

Tool Calling 是小売業向け AI 経営分析システム 项目中用于把业务输入变成可控制工程单元的核心节点。5 分钟内应能回答：它解决什么问题、为什么当前项目需要它、规模扩大后风险在哪里。

## 企业为什么需要

小売業向け AI 経営分析システム需要稳定、可审计、可扩展的系统边界。Tool Calling 的价值不是概念本身，而是让需求、数据、模型、API、运维和面试说明可以连接起来。没有这个节点，后续 Review、改修和运用会变得困难。

## 日本现场怎么解释

在日本现场应使用“業務要件から見た必要性”解释：这个设计是为了降低障害风险、提高レビュー性、让基本設計和詳細設計能落到代码，而是为了让设计理由、Review 和运用边界更清晰。

## 什么时候使用

- 业务需要可追溯、可复用、可测试时。
- 项目需要从 開発 进入 API、运维或多人员协作时。
- 面试中需要说明设计理由和替代方案时。

## 什么时候不用

- 临时脚本、无复用、无团队维护要求时可以简化。
- 需求还没有确认时，不要过早引入复杂抽象。
- 数据量和风险很低时，先用教学版验证链路。

## 关联项目

- PRJ-025
- PRJ-027

## 关联设计决策

- DES-MCP-002

## 面试会怎么问

- 为什么这里需要 Tool Calling？
- 不使用 Tool Calling 会有什么风险？
- 如果数据量扩大 100 倍，你会怎么调整？
- 日本 TL Review 时会重点看什么？

## TL Review 会关注什么

TL 会检查命名是否准确、边界是否清晰、是否有日志和异常处理、是否可以测试、是否会在规模扩大后成为性能瓶颈。

## 企业案例

典型案例是社内文档问答、AI 问答 API、流式聊天 UI、审批 Agent 或 PR 审查 Agent。它们都要求知识点能连接到项目、设计决策和面试解释。

## 常见误区

- 只解释技术定义，不解释业务价值。
- 没有关联设计决策。
- 没有说明什么时候不用。
- 没有和日本现场交付物建立关系。

# 小売業向け AI 経営分析システム Handbook Extension

## Design Decision

- 为什么 Tool Calling：模型不能直接访问文件、DB、Office、GitHub，需要受控工具边界。
- 为什么固定 Workflow：关键业务流程要求可测试、可审计、可重复。
- 为什么 Memory：多轮任务需要保留状态，但生产版必须控制隐私和过期策略。
- 什么时候不用 Agent：固定规则、固定 SQL、简单 CRUD、低不确定性任务不要强行 Agent 化。

### Alternatives

规则引擎、传统工作流、Batch job、RPA、普通 API 编排。

### Fit / Not Fit

- 适用场景：系统开发已经证明业务价值，需要进入架构评审、面试说明或運用版设计。
- 不适用场景：临时脚本、无多人维护、无审计需求、无需长期运用的场景。

### Pros / Cons / Cost

- 优点：可解释、可 Review、可扩展。
- 缺点：比最小 demo 多设计和治理成本。
- 成本：需要维护 schema、日志、测试、评估、部署和监控。
- 维护成本：随着数据、模型、工具和组织复杂度增加，需要定期 Review。
- 扩展性：应通过分层、接口契约、状态管理和观测性扩展。

## TL Review

日本企业 TL 最可能追问：

- なぜこの設計にしましたか。
- なぜ他の方式を使わなかったのですか。
- 本格運用に向けた拡張ポイントはどこですか。
- 性能、復旧、セキュリティ、ログ、監視はどう設計しますか。
- 100 倍のデータ量、1000 TPS、外部サービス障害にどう対応しますか。

## Enterprise Practice

审批 Agent、Office 文档 Agent、PR 审查 Agent、SES 营业支援、客服转人工。

真实项目通常先明确业务范围、系统边界和 Review 观点，再由 TL 和 Architect 判断扩展路线。日本现场会要求基本设计、详细设计、レビュー票、障害票、运维和改修路线。

## Production Gap

缺 tool allowlist、权限、审批、审计、幂等、重试、超时、失败补偿、工具结果 schema。

这些是未来扩展目标，これらは本格運用に向けて設計・拡張する項目です。

## Continue Learning

继续学习 MCP、tool schema、human approval、idempotency、agent eval、tool-use tracing。

为什么值得学：这些内容决定候选人能否从“会跑 demo”升级为“能做小売業向け AI 経営分析システム开发、能参与架构评审、能通过日本现场面试”的工程师。
