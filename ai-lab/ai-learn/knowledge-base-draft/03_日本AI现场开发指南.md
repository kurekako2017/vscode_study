# 03_日本AI现场开发指南

## 基本设计书

用于描述业务背景、系统目标、架构、API、数据、权限、非功能需求和 担当範囲。

## 詳細設計書

用于描述函数、类、接口、错误处理、日志、测试点和模块职责。

## Review

レビュー票必须记录指摘、原因、対応、确认者和状态。

## 障害票

障害票记录发生时间、影响范围、临时处理、根因、永久对应和再发防止。

## 日报 / 周报

简洁报告：做了什么、遇到什么问题、明天做什么、需要谁确认。

## 工数 / 見積

小売業向け AI 経営分析システム 估算必须考虑调查、验证、模型不确定性、评估和 Review。

## SES / 自社开发 / 受托开发

SES 强调现场适应和沟通，自社开发强调长期维护，受托开发强调交付物和验收。

## Jira / Confluence / Teams

Jira 管任务，Confluence 管设计和会议纪要，Teams 管日常沟通。

## Git Flow / Docker / Kubernetes / CI/CD

Git 管变更，Docker 保证复现，Kubernetes 管部署，CI/CD 管质量门禁。

# JP-CONTRACT-001 SES

> 定位：本文件属于小売業向け AI 経営分析システム开发知识库。内容用于架构讨论、日本现场面试准备和 TL Review。


## 日本现场怎么做

SES 是日本 AI 开发现场的重要交付或沟通主题。它必须和项目、Review、面试、障害対応连接，不能只作为流程名出现。

## 典型交付物

- 目的
- 范围
- 责任者
- 输入输出
- Review 记录
- 变更履历

## PM / TL / SE / PG 分工

PM 确认业务优先级，TL 确认技术风险和 Review 标准，SE 整理设计，PG 实现和测试。

## Jira / Confluence / Teams / SharePoint 使用方式

Jira 管任务，Confluence 管设计和会议纪要，Teams 管沟通，SharePoint 管正式资料和附件。

## Review 方式

必须留下指摘、対応、確認者和状态。AI 项目还要补充模型、数据、权限和运维风险。

## 常见问题

- 只写实现，不写业务背景。
- 没有 Review 记录。
- 没有说明障害时如何处理。

# Enterprise Handbook Alignment

## Design Decision

TL 会检查本手册是否能支持三件事：项目能讲清楚、设计决策能讲清楚、本格運用に向けた拡張ポイント能讲清楚。

## Enterprise Practice

企业 项目文档应服务于内部研发、架构讨论、Code Review、面试说明和后续本格運用化计划，而服务于面试表达和长期维护。

## Production Gap

本手册中的运用设计内容用于说明本格運用に向けた扩展思路，包括权限、审计、监控、CI/CD 和 Kubernetes。
