## 3分钟项目介绍（日语）

```text
小売業向け AI 経営分析システムは、日本の小売・EC 業務を想定した経営分析システムです。
多店舗、多商品、多地域の売上・在庫データをもとに、経営会議前の分析レポート作成を支援します。

設計上のポイントは、固定 KPI 分析と Research Agent を分けている点です。
売上、粗利、在庫、地域別差異のような KPI は、数値の正確性と再現性が重要なため、固定 SQL テンプレートと固定 workflow で処理します。
Agent に自由に SQL を生成させると、SQL Injection、存在しない項目の参照、集計口径のブレ、重いクエリ、権限違反のリスクがあるためです。

一方、市場トレンド、競合情報、社内ポリシーの確認は、質問によって必要な情報が変わるため Research Agent に分けています。
Research Agent は tool を選択し、調査結果と根拠をレポートに反映します。

バックエンドは FastAPI で、TaskService がタスク作成、状態更新、イベント保存を担当します。
LangGraph StateGraph が route、data workflow、research agent、report generation を制御し、SSE / WebSocket で進捗をフロントエンドへ返します。
最終的には日本語 Markdown レポートを生成し、データ結果、調査結果、リスク、人工確認事項を分けて表示します。
```

## 5分钟项目介绍（日语）

```text
小売業向け AI 経営分析システムは、日本の小売業を想定した経営分析システムです。
業務背景として、店舗、地域、商品、在庫、販促、競合、市場トレンドの情報を経営会議前に整理する必要があります。
このシステムでは、CSV や DB 由来の売上・在庫データ、社内ポリシー、競合調査、市場メモを組み合わせ、日本語の経営分析レポートを生成します。

アーキテクチャは、React フロントエンド、FastAPI バックエンド、TaskService、EventBus、SSE / WebSocket、LangGraph StateGraph、固定 KPI workflow、Research Agent、Report Composer、Docker で構成しています。
API ではタスク作成、状態取得、イベント配信、レポート取得を扱います。
TaskService は HTTP 層と workflow 実行を分離し、LangGraph は route、data workflow、research agent、report generation の状態遷移を管理します。

KPI 分析では、売上、粗利、在庫、地域別差異を固定 SQL テンプレートと whitelist workflow で処理します。
これは経営数字の正確性、再現性、監査性を担保するためです。
Research Agent は、市場トレンド、競合情報、社内ポリシー確認など、質問ごとに調査内容が変わる領域を担当します。

私は、API 設計、workflow 設計、Research Agent、固定 KPI 分析、Streaming、レポート生成、Docker 構成、設計ドキュメント整理を担当しました。
今後の拡張として、SSO、RBAC、多テナント、SQL AST 設計、DWH 接続、企業検索、VectorDB、監査ログ、OpenTelemetry、CI/CD、Kubernetes を想定しています。
```

## 技术亮点（日语）

```text
技術的なポイントは、固定 workflow と Research Agent の責務を分けている点です。
KPI は固定 SQL と whitelist で安定性を重視し、市場調査や社内情報確認は Research Agent で柔軟性を持たせています。
また、FastAPI、TaskService、SSE / WebSocket、LangGraph、checkpoint、Report Composer を組み合わせ、タスク実行型の経営分析システムとして設計しています。
```

## 为什么做这个项目（日语）

```text
小売業では、売上、在庫、商品、地域、販促、市場、競合情報を横断して確認する必要があります。
人手で Excel、CSV、社内資料を整理すると時間がかかり、根拠や集計口径も分散しやすいです。
そのため、固定 KPI 分析と Research Agent を組み合わせた経営分析システムとして設計しました。
```

## 为什么这样设计（日语）

```text
設計上は、安定性が必要な処理と柔軟性が必要な処理を分けました。
売上や在庫などの経営 KPI は、結果が毎回変わると業務上問題になるため固定 workflow にしています。
市場や競合、社内資料の調査は、質問によって必要な情報が変わるため Research Agent にしています。
この分離により、経営数字の安定性と調査業務の柔軟性を両立させています。
```

## 本格運用に向けて追加する機能（日语）

```text
本格運用に向けて、SSO、OAuth、RBAC、多テナント、監査ログ、SQL AST 設計、行列レベル権限を追加します。
その上で、CSV / SQLite を PostgreSQL や DWH に置き換え、社内資料検索を SharePoint、Confluence、社内 Wiki、OpenSearch、VectorDB に拡張します。
さらに Redis、durable queue、OpenTelemetry、CloudWatch、CI/CD、Kubernetes、load test、backup、rollback も設計します。
```

## 面试官继续追问与推荐回答（日语）

### なぜ固定 SQL と Agent を分けましたか。

```text
経営 KPI は正確性と再現性が重要なので固定 SQL にしました。
Agent に自由に SQL を生成させると、集計口径の不一致や SQL Injection、存在しない項目の参照が起きる可能性があります。
一方で、市場や競合、社内資料の調査は柔軟性が必要なので Research Agent に分けています。
```

### なぜ LangGraph を使いましたか。

```text
処理が route、data workflow、research agent、report generation の複数ステップに分かれ、状態を持って進むためです。
LangGraph を使うことで State、Node、Edge、条件分岐、checkpoint を明示でき、workflow の見通しがよくなります。
```

### RAG はどこに使っていますか。

```text
社内情報や市場調査メモを参照し、レポートに根拠を付ける部分で使っています。
文書量が増える場合は、Chunk、Embedding、Retriever、VectorDB、Rerank、ACL filter を組み合わせます。
```

### 本格運用の最初の一歩は何ですか。

```text
最初は認証、認可、tenant_id、監査ログ、SQL 安全性、評価セットです。
小売業の売上、在庫、販促情報は権限管理が重要なので、誰がどのデータにアクセスしたかを追跡できる設計にします。
```

### 改善点は何ですか。

```text
改善点は、SSO、RBAC、多テナント、SQL AST 設計、DWH 接続、企業検索、監視、CI/CD、負荷試験の強化です。
Research Agent の tool layer を SharePoint、Confluence、外部検索 API、MCP Tool と接続し、引用と権限管理も強化します。
```
