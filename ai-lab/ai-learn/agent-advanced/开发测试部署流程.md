# Agent 开发、测试与部署流程

这份文档专门回答一个问题：

> **同一套代码，怎么从“本地开发”一路走到“上线前验收”？**

`agent-advanced/` 里的很多 demo 都遵循同一个思路：

- **开发阶段**：先用 Mock LLM，把流程和结构跑通
- **集成阶段**：切到本地模型、便宜模型或兼容 API，验证真实调用链路
- **E2E 阶段**：用少量真实 API 样本做端到端测试
- **上线前阶段**：再做一次真实 API 抽样和部署验收

这样做的好处是：

- 代码主逻辑尽量不改
- 只通过环境变量或命令行开关切换模式
- 先保证“能跑”，再保证“质量够”
- 真正上线前，能把风险尽量压缩到可控范围

## 1. 同一套代码，四种运行模式

这套学习线里，最重要的不是“写出多个版本的代码”，而是**同一套代码在不同阶段切换不同模式**。

| 模式 | 目标 | 典型用途 |
| --- | --- | --- |
| Mock LLM | 不依赖外部 API，先验证流程 | 开发期、离线学习、单元测试 |
| 本地模型 / 便宜模型 | 验证真实推理链路，但控制成本 | 集成测试、联调、日常调试 |
| 少量真实 API E2E | 验证真实供应商、真实网络、真实返回 | 小规模端到端测试 |
| 上线前真实 API 抽样 | 上线前再确认关键路径没坏 | 发布前检查、回归、验收 |

### 这四种模式分别看什么

- **Mock LLM**：看输入输出结构、分支流程、解析逻辑、错误处理
- **本地模型 / 便宜模型**：看真实模型返回是否符合预期，顺便控制费用
- **少量真实 API E2E**：看供应商接口、认证、网络、延迟、返回格式
- **上线前抽样**：看最关键的业务样本有没有回归，别把明显问题带上线

## 2. 推荐的开发到部署流程

下面是我建议的标准顺序。

### 第 1 步：本地开发

先把代码写成“可切换模式”的形状：

- 把模型调用封装成一个函数
- 把 prompt、检索、重排、引用等步骤拆开
- 给每一步都留出可测试入口
- 默认先支持 Mock

适合做的检查：

- `python3 -m py_compile ...`
- `ruff` / `flake8` / `black` 这类格式和静态检查
- 本地纯函数测试

### 第 2 步：单元测试

单元测试优先测**不依赖真实 API**的逻辑。

建议测的对象：

- prompt 拼接
- query rewrite
- 关键词提取
- 检索排序
- rerank 规则
- 权限过滤
- 引用拼接
- 输出解析

推荐模式：

- **Mock LLM**
- 固定输入
- 固定输出
- 尽量避免外部网络

### 第 3 步：集成测试

集成测试的重点是：**代码已经连成一条链，但不一定要用昂贵的真实 API。**

可以用这些方式：

- 本地模型
- 便宜模型
- OpenAI 兼容接口
- OpenRouter / DeepSeek / 其他兼容服务

这一步主要验证：

- 环境变量能否被正确读取
- 模型客户端能否正常初始化
- 检索 / 生成 / 解析之间能否顺利接上
- 服务启动后接口是否通

### 第 4 步：少量 E2E 测试

这一步开始使用**真实 API**，但样本数量不要大。

建议只跑少量关键样本，比如：

- 3 到 10 个核心问题
- 覆盖最常见的用户路径
- 覆盖最容易出错的边界问题

重点看：

- 返回内容是否完整
- 引用是否存在
- 结构化输出是否稳定
- latency 是否可接受
- 是否出现明显的幻觉或跑题

### 第 5 步：上线前抽样测试

上线前不要重新发明一套测试，而是做一轮“抽样复核”。

建议检查：

- 真实 API 是否可用
- 关键路径是否通过
- 配置是否正确
- 容器是否能启动
- 健康检查是否正常
- smoke test 是否通过

上线前更关注：

- 能不能启动
- 能不能返回
- 关键样本有没有回归
- 这次改动是否影响已有场景

## 3. 测试方案总表

| 阶段 | 模式 | 目标 | 测试内容 | 通过标准 |
| --- | --- | --- | --- | --- |
| 开发期 | Mock LLM | 快速验证流程 | prompt、解析、分支、引用拼接 | 不依赖外部 API 也能跑通 |
| 单元测试 | Mock LLM | 验证纯逻辑 | query rewrite、过滤、rerank、格式化 | 输入固定时输出稳定 |
| 集成测试 | 本地模型 / 便宜模型 | 验证真实链路 | 模型初始化、请求发送、返回解析 | 能正常调用并得到合理结果 |
| 小型 E2E | 真实 API | 验证真实服务 | 关键问答、引用、权限、错误处理 | 样本结果可接受，关键路径可用 |
| 上线前抽样 | 真实 API | 验证发布风险 | 核心样本回归、容器启动、健康检查 | 不阻塞发布，没有明显回归 |

## 4. 适合 Agent 的测试层次

`agent-advanced/` 里建议把测试分成这几层。

### 4.1 单元测试：Mock LLM

适合测：

- prompt 模板是否正确
- 输出解析是否正确
- 检索结果是否排序正确
- 权限过滤是否正确
- 结构化字段是否齐全

特点：

- 快
- 稳
- 成本低
- 适合频繁跑

### 4.2 集成测试：本地模型 / 便宜模型

适合测：

- 真实模型返回是否符合接口预期
- 兼容 API 是否能连通
- 请求参数是否设置正确
- 模型输出是否足够稳定

特点：

- 比单元测试慢一点
- 比真实 API 便宜
- 适合 CI 或本地验证

### 4.3 小量 E2E：真实 API

适合测：

- 问答结果
- 引用来源
- 多步工作流
- 多 Agent 之间的协作结果

建议：

- 样本少
- 只测关键链路
- 不要把大量回归全压在真实 API 上

### 4.4 上线前抽样：真实 API + 部署验收

适合测：

- 容器能不能正常启动
- 健康检查能不能通过
- 服务能不能返回
- 关键输入是否仍然正确

这一步更像“发布门禁”，不是完整测试。

## 5. `agent-advanced` 里可以怎么落地

结合当前目录结构，我建议按下面方式做。

### `projects/`

这里放业务 demo，本身就应该支持：

- `--mock`
- `--real`
- 或者通过环境变量切换

例如：

- `langchain_chain_demo`：Mock LLM / 真实 API
- `advanced_rag_pipeline_demo`：本地资产 / 真实检索源
- `internal_hybrid_rag_demo`：本地模拟 / 真实企业知识源

### `eval/`

这里放评估脚本和测试口径。

可以负责：

- coverage
- precision
- answer completeness
- citation coverage

### `deployment/`

这里放上线前的检查示例。

可以负责：

- Docker
- 健康检查
- smoke test
- 端口与环境变量

### `frontend/`

这里放前端联调和 UI 验证。

建议测：

- 消息发送
- 来源展示
- loading 状态
- 错误状态

## 9. 实际部署平台清单

日本项目里常见的部署平台，通常先看这几类。

### 日本项目里常见的云平台

| 平台 | 常见承载方式 | 适合放什么 |
| --- | --- | --- |
| AWS | EC2、ECS、EKS、Lambda、S3、RDS | API 服务、容器、向量库、对象存储、定时任务 |
| Azure | VM、App Service、AKS、Functions、Blob Storage、Azure OpenAI | 企业内部 AI 服务、前后端、企业认证集成 |
| GCP | Compute Engine、Cloud Run、GKE、Cloud Storage、Vertex AI | 容器化服务、推理服务、检索服务 |
| Oracle Cloud / OCI | Compute、OKE、Object Storage | 传统企业系统周边服务 |
| 其他常见日本企业环境 | 内网服务器、Kubernetes、虚拟机、SaaS 集成 | 企业内部 PoC、权限严格环境、旧系统周边服务 |

### 私有化 / 企业内部平台

| 平台 | 常见承载方式 | 适合放什么 |
| --- | --- | --- |
| 本地服务器 | Linux 主机、Docker、systemd | 学习环境、小型 PoC、内网服务 |
| VMware / 虚拟化环境 | 虚拟机、内网网段 | 企业内部隔离环境 |
| Kubernetes 集群 | 自建 K8s、OpenShift、Rancher | 多服务 Agent、RAG、前后端联动 |
| Windows Server / 内网机房 | IIS、Windows 服务、容器 | 老系统周边集成、权限严格环境 |

### 典型部署组合

| 场景 | 常见组合 |
| --- | --- |
| 个人学习 / PoC | 本地 Python + Mock LLM + 本地文件 |
| 团队联调 | Docker + 本地模型 / 便宜模型 + 简单前端 |
| 企业内部验证 | 内网服务器 + 统一登录 + RAG API + 前端页面 |
| 正式生产 | Kubernetes / 云容器 + 监控 + 日志 + 评估 + 回滚 |

### 选平台时先看什么

- 是否能开出稳定的网络出口
- 是否能放环境变量和密钥
- 是否支持容器化
- 是否支持健康检查
- 是否方便接日志和监控
- 是否能满足权限和审计要求

### 在 `agent-advanced` 里怎么对照

- `deployment/container_demo`：先练 Docker 和健康检查
- `frontend/chat_ui_demo`：先练前端联调
- `eval/rag_eval_demo`：先练评估报告
- `projects/internal_hybrid_rag_demo`：先练企业知识检索链路

如果后面要贴近真实项目，可以继续把这些 demo 对应到：

- AWS ECS / EKS
- Azure App Service / AKS
- GCP Cloud Run / GKE
- 公司内部 K8s 或虚拟机环境

## 10. 交付前检查清单

这是一页版的执行清单，适合你在真正跑项目时直接勾选。

### 开发完成

- [ ] 代码能通过 `py_compile`
- [ ] 主要文件都有中文注释或说明
- [ ] `README.md` 能告诉我怎么跑
- [ ] 默认有 mock 或离线模式

### 单元测试

- [ ] prompt / 解析 / 检索 / 权限过滤都有固定样本
- [ ] 关键纯函数有稳定输出
- [ ] 失败分支也能被手动触发
- [ ] 不依赖真实 API 也能验证核心逻辑

### 集成测试

- [ ] 本地模型或便宜模型能正常启动
- [ ] 环境变量能正确读取
- [ ] 请求能发出去，返回能解析
- [ ] 基本链路没有明显报错

### 小量 E2E

- [ ] 3 到 10 个核心问题已经跑过
- [ ] 引用来源能返回
- [ ] 答案没有明显跑题
- [ ] 延迟在可接受范围内

### 部署前

- [ ] Dockerfile / 启动脚本可用
- [ ] `/health` 或等价健康检查可用
- [ ] 目标平台能注入环境变量
- [ ] 关键配置已经和日本项目环境对齐

### 上线前抽样

- [ ] 核心样本再跑一遍
- [ ] 没有明显回归
- [ ] 没有敏感配置泄露
- [ ] 日志和监控能接上

### 日本项目平台判断

- [ ] AWS / Azure / GCP / OCI / 内网平台已确认
- [ ] 容器、虚拟机、K8s 其中哪一种已经定下来
- [ ] 网络出口、权限、审计要求已经明确

## 6. 一个最小可执行的测试顺序

如果你现在想从头开始跑，我建议按这个顺序：

1. `python3 -m py_compile` 检查语法
2. 跑 Mock 模式，验证流程
3. 跑本地模型或便宜模型，验证集成
4. 跑少量真实 API 样本，验证 E2E
5. 跑 eval 脚本，观察 coverage / precision
6. 打包容器，做健康检查和 smoke test
7. 上线前再抽样验证关键路径

## 7. 一个实用原则

**开发时追求速度，测试时追求稳定，上线前追求保守。**

具体来说：

- 开发期尽量用 Mock
- 集成期尽量用便宜模型
- E2E 只跑少量真实样本
- 上线前只抽样关键路径

这样你会发现，Agent 项目不会因为“每次都必须真实 API”而被卡死，也不会因为“只测 Mock”而对真实环境失去把握。
