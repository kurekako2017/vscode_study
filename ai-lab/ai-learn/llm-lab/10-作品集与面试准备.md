# 作品集与面试准备

这一章解决的问题是：把已经做过的 demo 整理成能对外说明的 PoC 和作品集。

作品集不是堆项目，而是讲清楚：

- 解决什么问题
- 为什么这样设计
- 输入输出是什么
- 当前限制是什么
- 下一步怎么企业化

## 1. 推荐作品集组合

最小组合：

1. [structured_output_demo](../agent-lab/projects/structured_output_demo/README.md)
2. [doc_qa_agent](../agent-lab/projects/doc_qa_agent/README.md)
3. [rag_api_demo](../agent-lab/projects/rag_api_demo/README.md)
4. [rag_eval_notes.md](./rag_eval_notes.md)

进阶追加：

- [tool_agent_demo](../agent-lab/projects/tool_agent_demo/README.md)
- [workflow_agent](../agent-lab/projects/workflow_agent/README.md)

优先打磨 `doc_qa_agent` 和 `rag_api_demo`，因为它们最贴近：

- 社内検索
- ナレッジ検索
- RAG API
- 生成 AI 后端 PoC

## 2. 每个案例建议准备什么

| 材料 | 作用 |
| --- | --- |
| README | 说明项目目标、运行方式、功能范围 |
| 需求概要 | 说明背景、用户、输入输出、范围 |
| 基本设计 | 说明架构、数据流、API、主要函数 |
| 测试观点 | 说明要验证什么 |
| 简单测试用例表 | 说明如何确认功能 |
| 试验结果报告 | 说明评估结果、失败点、改进方向 |

如果项目很小，至少也要写清：

- 背景
- 目标
- 技术构成
- 功能范围
- 限制与后续课题

## 3. 面试说明顺序

介绍项目时建议按这个顺序：

1. 背景
2. 目标用户
3. 解决的问题
4. 技术构成
5. 你负责的部分
6. 当前限制
7. 后续扩展

介绍 `rag_api_demo` 可以这样说：

```text
这是一个面向社内文档问答场景的最小 RAG API PoC。
后端使用 FastAPI，支持 md / txt / pdf 文档读取。
服务启动时构建本地 chunks，/ask 接口接收问题后检索相关片段，
再把上下文传给模型生成回答并返回 sources。
当前版本重点是验证链路，后续会补认证、日志、向量检索和评估。
```

## 4. 每个案例突出什么

| 案例 | 展示能力 |
| --- | --- |
| `structured_output_demo` | schema、Pydantic、稳定 JSON 输出 |
| `doc_qa_agent` | 文档读取、切分、检索、来源返回 |
| `rag_api_demo` | FastAPI、API 化、PDF、文档重载 |
| `rag_eval_notes` | 评估意识、失败记录、改进方向 |
| `tool_agent_demo` | Tool Calling、工具 schema、安全边界 |
| `workflow_agent` | 固定工作流、中间状态、结构化计划 |

## 5. 常见面试问题

- 为什么先做 RAG，不先做复杂 Agent？
- 为什么当前版本先用关键词检索？
- 如果文档变多，怎么优化？
- 如果回答错了，怎么评估和改进？
- 如果接企业系统，认证和权限怎么做？
- API Key 和配置如何管理？
- 当前 PoC 离正式上线还差什么？

## 6. 简历写法

中文：

- 基于 Python 和 OpenAI API 实现结构化输出 PoC
- 基于本地文档实现最小 RAG 问答系统，支持来源返回
- 使用 FastAPI 将文档问答能力封装为后端接口
- 整理 RAG 评估观点、测试问题和改进方向

日语：

- `Python / OpenAI API を用いた生成AIアプリ PoC を実装`
- `RAG による社内文書検索・問い合わせ対応機能を試作`
- `FastAPI により RAG 問い合わせ API を実装`
- `評価観点、テストケース、改善課題を整理`

## 7. 不要这样讲

| 不推荐 | 更好的说法 |
| --- | --- |
| 我做了一个 AI demo | 我做了一个面向社内文档问答的 RAG PoC |
| 我用了 Agent | 我用 Tool Calling 做了一个受限工具调用助手 |
| 这个可以上线 | 当前是 PoC，后续需要补认证、日志、评估和权限 |
| 模型会自己回答 | 系统会检索资料，并要求模型基于上下文回答 |

## 8. 最后检查清单

- README 能不能 3 分钟读懂？
- 命令能不能用 mock 模式跑通？
- 有没有说明输入、输出、限制？
- 有没有至少一份测试观点或评估记录？
- 有没有写清楚下一步企业化要补什么？

完成这一步后，再进入 [../agent-lab/README.md](../agent-lab/README.md) 做 Agent 进阶，会更容易讲成完整学习路线。
