# 07_面试口头训练

本文件只用于开口训练。练习时先遮住回答，听完问题后立即开口；说完再对照关键词，不逐字追求一致。

## 第一章 30秒回答训练

### 1. 自己紹介をお願いします。

**日语问题**

自己紹介をお願いします。

**30秒日语回答**

小売業の経営分析を支援する Retail Insight AI を設計・実装しました。私はバックエンドを中心に、Task API、TaskService、LangGraph Workflow、Research Agent、SSE、Report Generator を担当しました。KPI は Fixed KPI Workflow で安定して処理し、調査は Research Agent に分けています。

**中文理解**

先说项目价值，再说担当，最后说最重要的职责拆分。

**背诵重点**

小売業、担当範囲、KPI 与 Research 分离。

### 2. プロジェクトの概要を説明してください。

**日语问题**

Retail Insight AI の概要を説明してください。

**30秒日语回答**

Retail Insight AI は、小売業向け AI 経営分析システムです。POS、在庫、商品、会員、販促のデータと市場・競合の調査結果をまとめ、経営判断に使える日本語レポートを生成します。タスクの受付から分析、進捗通知、結果保存まで追跡できる構成です。

**中文理解**

不是单次问答，而是完整的经营分析任务流程。

**背诵重点**

数据来源、日文报告、全流程可追踪。

### 3. どの部分を担当しましたか。

**日语问题**

担当範囲を教えてください。

**30秒日语回答**

バックエンドを中心に、FastAPI の API 設計、Task API、TaskService、LangGraph Workflow、Fixed KPI Workflow、Research Agent、SSE、Report Generator、Docker、アーキテクチャレビューを担当しました。特に、長時間タスクの状態管理と KPI・Research の責務分離を担当しました。

**中文理解**

担当要落到具体模块，并指出自己的核心贡献。

**背诵重点**

模块清单不要漏；最后收束到状态管理和职责分离。

### 4. なぜ Task API にしたのですか。

**日语问题**

なぜ同期 API ではなく Task API にしたのですか。

**30秒日语回答**

KPI 計算、外部調査、レポート生成を含み、処理時間が一定ではないためです。同期 API ではタイムアウトや再送による二重実行が起きやすいので、Task API で task_id を先に返し、TaskService が状態を管理する構成にしました。

**中文理解**

重点是耗时不确定、超时、重复执行，不要只说异步更快。

**背诵重点**

長時間処理、task_id、TaskService。

### 5. TaskService はなぜ必要ですか。

**日语问题**

TaskService はなぜ必要なのですか。

**30秒日语回答**

API と分析処理の間で、タスクのライフサイクルを一元管理するためです。タスク作成、状態更新、Workflow 起動、イベント発行、結果保存を API から分離しました。TaskService は調整を担当し、KPI や Research の内部ロジックは持ちません。

**中文理解**

TaskService 是用例协调层，不是塞入所有逻辑的大 Service。

**背诵重点**

生命周期、API 分离、只协调不计算。

### 6. なぜ LangGraph を使ったのですか。

**日语问题**

なぜ LangGraph Workflow を採用したのですか。

**30秒日语回答**

依頼内容の判定、KPI、Research、Report Generator に条件分岐と失敗時の制御が必要だからです。LangGraph Workflow で State、Node、Edge を明示し、各処理の責務、途中状態、失敗箇所を追いやすくしました。

**中文理解**

不是因为流行，而是因为状态、分支和失败恢复需要显式化。

**背诵重点**

State、Node、Edge、追跡性。

### 7. なぜ全部 Agent にしなかったのですか。

**日语问题**

なぜすべての処理を Agent にしなかったのですか。

**30秒日语回答**

すべてを Agent にすると、KPI の処理経路や結果が変わり、計算根拠を説明しにくくなるためです。経営判断に使う数値は Fixed KPI Workflow、情報源や調査手順が変わる処理だけを Research Agent に任せました。

**中文理解**

确定性业务留在固定流程，不确定性调查才交给 Agent。

**背诵重点**

再現性、説明可能性、Agent 的使用边界。

### 8. なぜ KPI と Research を分けたのですか。

**日语问题**

なぜ Fixed KPI Workflow と Research Agent を分けたのですか。

**30秒日语回答**

KPI は同じ入力から同じ結果を出す必要があります。一方、市場・競合調査は質問によって情報源や手順が変わります。そこで、KPI は固定処理、調査は Research Agent と分け、正確性と柔軟性を両立しました。

**中文理解**

回答“为什么分”，而不是只重复两个模块的名字。

**背诵重点**

正確性と柔軟性。

### 9. なぜ SSE を使ったのですか。

**日语问题**

進捗通知に SSE を選んだ理由は何ですか。

**30秒日语回答**

主にサーバーから画面へ進捗を通知する一方向通信だからです。SSE で status、error、done を送り、接続が切れてもタスク自体は継続します。再接続できない場合も、状態確認 API から最終結果を取得できます。

**中文理解**

SSE 是通知通道，不是任务状态存储。

**背诵重点**

一方向、连接与任务分离、状态 API 兜底。

### 10. Production Gap は何ですか。

**日语问题**

Production Gap について説明してください。

**30秒日语回答**

正式運用に向けて追加が必要な項目です。優先しているのは SSO、RBAC、Audit Log、PostgreSQL、Redis、OpenTelemetry です。機能を増やす前に、権限、追跡性、永続化、障害調査を整える方針です。

**中文理解**

Production Gap 不是逃避，而是明确区分当前实现和正式运用要求。

**背诵重点**

优先级和业务理由。

## 第二章 1分钟回答训练

### 1. このシステムはどんな課題を解決しますか。

**日语问题**

Retail Insight AI は、どのような業務課題を解決しますか。

**1分钟日语回答**

小売業の経営企画では、経営会議の前に POS、在庫、商品、会員、販促などのデータを集め、市場や競合の情報も調査します。データが CSV、Excel、API に分かれていると、整理に時間がかかり、担当者によって分析の観点も変わります。Retail Insight AI は、データ検証、KPI 計算、Research、Report Generator を一つの流れにまとめ、進捗と根拠を追跡できる日本語レポートを生成します。

**面试官想确认什么**

是否真正理解零售业务，而不是从框架名开始介绍。

**不能说什么**

不要只说“使用 AI 自动生成报告”，也不要省略数据来源和业务使用者。

### 2. システム全体の処理を説明してください。

**日语问题**

タスク作成からレポート取得までの流れを説明してください。

**1分钟日语回答**

利用者が分析条件を送ると、Task API が task_id を返します。TaskService がタスクを running にし、LangGraph Workflow を起動します。Workflow では依頼内容を判定し、Fixed KPI Workflow と Research Agent を実行します。両方の結果を Report Generator がまとめ、保存後に completed へ更新します。進捗は SSE で画面へ通知し、画面は完了後にレポートを取得します。

**面试官想确认什么**

是否能不看图也说清调用顺序和职责边界。

**不能说什么**

不要把 API、TaskService、Workflow、Report Generator 混成一个处理。

### 3. 一番工夫した設計は何ですか。

**日语问题**

このシステムで一番工夫した点は何ですか。

**1分钟日语回答**

正確性が必要な処理と、柔軟性が必要な処理を分けた点です。売上や在庫の KPI は経営判断に使うため、同じ入力から同じ結果を出す必要があります。そのため Fixed KPI Workflow で処理します。一方、市場・競合調査は質問によって情報源が変わるため、Research Agent が担当します。Research が失敗しても KPI を残せるため、品質と障害隔離の両方に効果があります。

**面试官想确认什么**

架构选择是否来自业务要求，并能说明收益。

**不能说什么**

不要把“采用了新技术”当作最大亮点。

### 4. TaskService の設計を説明してください。

**日语问题**

TaskService の責務と設計上の注意点を説明してください。

**1分钟日语回答**

TaskService はタスク作成、状態更新、Workflow 起動、イベント発行、結果保存、失敗処理を調整します。API 層から長時間処理を分離し、状態遷移を queued、running、completed、failed に限定します。一方、KPI 計算、Research の Tool 選択、レポート文面は持たせません。将来、実行基盤をキューへ移しても API 契約への影響を小さくできる境界にしています。

**面试官想确认什么**

TaskService 是否会变成职责过大的类。

**不能说什么**

不要回答“Service 层通常都这样写”。

### 5. LangGraph Workflow の設計を説明してください。

**日语问题**

LangGraph Workflow をどのように設計しましたか。

**1分钟日语回答**

State には task_id、入力条件、KPI 結果、Research 結果、出典、エラー、レポート情報を持たせます。Node は route、KPI、research、report に分け、必要な項目だけを読み書きします。Edge では KPI のみ、Research のみ、両方という経路を切り替えます。State を大きくしすぎず、失敗した Node と再実行範囲を追えることを重視しました。

**面试官想确认什么**

是否真正理解 State、Node、Edge，而不是只会说框架名称。

**不能说什么**

不要说“所有数据都放进 State 比较方便”。

### 6. Research Agent の安全性をどう考えましたか。

**日语问题**

Research Agent の自由度と安全性をどう両立しましたか。

**1分钟日语回答**

Research Agent は市場、競合、内部資料の調査に限定しています。利用可能な Tool、入力、出力、最大実行回数、timeout を明確にし、結果には出典と更新日を持たせます。内部資料を検索する場合は、利用者の部門や店舗スコープで絞り込みます。KPI の値は変更させず、調査結果だけを Report Generator へ渡します。

**面试官想确认什么**

Agent 是否有权限、终止条件、来源和失败边界。

**不能说什么**

不要说“Agent 会自己判断，所以不需要限制”。

### 7. 障害時はどう調査しますか。

**日语问题**

タスクが失敗した場合、どのように調査しますか。

**1分钟日语回答**

まず影響範囲、発生時刻、継続中かを確認し、必要であれば処理を停止します。その後、request_id、task_id、trace_id を使い、API、TaskService、LangGraph Workflow の Node、Research Tool、Report Generator の順に追います。復旧を優先し、原因分析と恒久対応は分けます。入力エラー、権限エラー、一時的な外部エラーも区別します。

**面试官想确认什么**

是否先止血，再定位，再做永久修复。

**不能说什么**

不要只说“先把所有日志从头看一遍”。

### 8. 権限と監査をどう設計しますか。

**日语问题**

RBAC と Audit Log はどのように設計しますか。

**1分钟日语回答**

RBAC では、経営層、経営企画、商品部、在庫管理、店舗担当を役割へ整理し、タスク作成、データ閲覧、レポート閲覧などの権限へ対応させます。さらに部門、地域、店舗のデータスコープも確認します。Audit Log には user_id、task_id、操作、対象、結果、時刻、trace_id を記録し、機密データ本文は残しません。

**面试官想确认什么**

权限是否贯穿 API、数据、Research、报告和审计。

**不能说什么**

不要把前端隐藏按钮当作权限控制。

### 9. PostgreSQL へどう移行しますか。

**日语问题**

SQLite から PostgreSQL へ移行する場合、どう進めますか。

**1分钟日语回答**

まず Repository を通じてアクセスし、業務ロジックから SQLite 依存を外します。販売、在庫、商品、会員などの業務データと、タスク、イベント、レポート、Audit Log を分けてモデル化します。その後、データ移行、件数と内容の整合性確認、切替手順、バックアップ、ロールバックを準備します。接続先を変えるだけではなく、型とトランザクションの差も確認します。

**面试官想确认什么**

是否理解数据库迁移不仅是修改连接字符串。

**不能说什么**

不要说“把 SQLite 文件导入 PostgreSQL 就完成了”。

### 10. 運用に向けて何を優先しますか。

**日语问题**

Production Gap の中で、何を優先しますか。

**1分钟日语回答**

最初に SSO、RBAC、Audit Log、PostgreSQL、OpenTelemetry を優先します。小売業の売上や会員データを扱うため、誰が何を見られるか、誰がどのデータでレポートを作ったか、障害時にどこで失敗したかを追えることが重要だからです。その後、負荷に応じて Redis、RabbitMQ、Kubernetes、バックアップ、ロールバックを段階的に整備します。

**面试官想确认什么**

是否能按业务风险排序，而不是先堆基础设施。

**不能说什么**

不要回答“先上 Kubernetes，之后再考虑权限和监控”。

## 第三章 深掘り追问训练

### 1. 同期 API でも timeout を長くすればよいのでは？

**日语问题**

同期 API の timeout を長くすれば解決できませんか。

**推荐回答**

timeout を延ばしても、切断後の状態確認、再送による二重実行、進捗表示は解決できません。そのため、HTTP リクエストと分析タスクのライフサイクルを分けました。

**如果继续被追问怎么答**

短時間の入力検証や状態取得は同期のままです。長時間処理だけを Task API にしています。

**NG回答**

非同期の方が常に高速だからです。

### 2. 二重実行をどう防ぎますか。

**日语问题**

利用者が同じタスクを二回送った場合はどうしますか。

**推荐回答**

Task API で冪等キーを受け取り、同じ利用者と同じ条件の受付済みタスクを確認します。結果保存も task_id とバージョンで重複を防ぎます。

**如果继续被追问怎么答**

画面のボタン無効化だけに依存せず、サーバー側で保証します。

**NG回答**

利用者に二回押さないようお願いすれば十分です。

### 3. TaskService が大きくなりませんか。

**日语问题**

TaskService に責務が集中しすぎませんか。

**推荐回答**

TaskService は状態遷移とコンポーネント間の調整に限定します。KPI 計算、Research の Prompt、Report Generator のテンプレートは各モジュールが持ちます。

**如果继续被追问怎么答**

TaskService のテストは、状態順序、失敗収束、イベント発行に集中させます。

**NG回答**

中心 Service なので、すべての処理を持たせます。

### 4. State が大きくなりませんか。

**日语问题**

LangGraph Workflow の State が大きくなる問題をどう防ぎますか。

**推荐回答**

State にはルーティングと後続 Node に必要な情報だけを持たせます。原データや全文は Repository に保存し、State には参照 ID と要約を保持します。

**如果继续被追问怎么答**

新しい項目を追加するときは、作成者、利用者、保存期間、checkpoint の必要性を確認します。

**NG回答**

Node 間で共有しやすいので、すべて State に入れます。

### 5. Node の境界をどう決めましたか。

**日语问题**

LangGraph Workflow の Node は、どのように分割しましたか。

**推荐回答**

route、KPI、research、report の業務責務で分けました。各 Node は必要な State だけを読み、担当結果だけを更新します。

**如果继续被追问怎么答**

小さな関数をすべて Node にせず、独立して失敗・再実行したい単位で分けます。

**NG回答**

関数一つにつき Node 一つにしました。

### 6. Checkpoint は何のためですか。

**日语问题**

Checkpoint を使う目的は何ですか。

**推荐回答**

途中状態を保存し、失敗箇所の確認と安全な再開を可能にするためです。完了済み Node の無条件な再実行も防ぎやすくなります。

**如果继续被追问怎么答**

正式運用では PostgreSQL などへ持続化し、State バージョンと復旧試験も必要です。

**NG回答**

LangGraph にある機能なので、とりあえず有効にします。

### 7. 条件分岐に該当しない入力はどうしますか。

**日语问题**

route Node で判定できない入力はどう扱いますか。

**推荐回答**

安全な fallback を定義し、入力不足として終了させます。running のまま残る経路は作りません。

**如果继续被追问怎么答**

TaskService が最終状態を failed または確認待ちへ収束させ、理由を利用者へ返します。

**NG回答**

Agent に自由に判断させます。

### 8. Research Agent が timeout したら？

**日语问题**

Research Agent がタイムアウトした場合はどうしますか。

**推荐回答**

Tool 単位と Research 全体に timeout を設定します。一時的な通信エラーだけを限定的に再試行し、失敗時は KPI を残して調査不足を明記します。

**如果继续被追问怎么答**

認証、権限、入力エラーは再試行せず、どの Tool で失敗したかを記録します。

**NG回答**

完了するまで timeout を延ばします。

### 9. Report Generator が失敗したら？

**日语问题**

Report Generator だけが失敗した場合はどうしますか。

**推荐回答**

KPI と Research の中間結果を保存し、Report Generator だけを再実行できるようにします。入力、テンプレート、モデル、出力検証の順に切り分けます。

**如果继续被追问怎么答**

一時的なモデルエラーと、テンプレート不整合のような恒久エラーを分けます。

**NG回答**

最初から全タスクを再実行します。

### 10. SSE が切断されたら？

**日语问题**

SSE が途中で切れた場合、タスクはどうなりますか。

**推荐回答**

タスクは継続します。クライアントは task_id と最後の event_id で再接続し、復旧できない場合も状態 API から結果を確認できます。

**如果继续被追问怎么答**

複数インスタンス化では Redis に短期イベントを保存し、PostgreSQL を最終状態の記録先にします。

**NG回答**

画面を更新して最初から実行します。

### 11. done と error をどう区別しますか。

**日语问题**

SSE の done と error の意味を説明してください。

**推荐回答**

done は成功完了だけを表し、error は失敗終了を表します。error の後に done を送らない契約にします。

**如果继续被追问怎么答**

最終状態は SSE ではなく状態 API と持続化データで確認します。

**NG回答**

どちらも処理終了なので、同じ意味です。

### 12. KPI の数字が間違っていたら？

**日语问题**

KPI の計算ミスが見つかった場合、どう調査しますか。

**推荐回答**

task_id からデータバージョン、対象期間、KPI 定義バージョンを特定し、前処理、集計条件、丸め処理の順に再計算します。

**如果继续被追问怎么答**

## 第六章 最终背诵版

### 1. 30秒自己紹介

小売業の経営分析を支援する Retail Insight AI を設計・実装しました。私はバックエンドを中心に、Task API、TaskService、LangGraph Workflow、Research Agent、SSE、Report Generator を担当しました。KPI は Fixed KPI Workflow で安定して処理し、調査は Research Agent に分けています。

### 2. 1分プロジェクト紹介

Retail Insight AI は、小売業向け AI 経営分析システムです。POS、在庫、商品、会員、販促のデータと、市場・競合の調査結果をまとめ、経営判断に使える日本語レポートを生成します。私はバックエンドを担当し、長時間処理には Task API を使い、TaskService が状態を管理し、SSE で進捗を通知する構成にしました。KPI は Fixed KPI Workflow、調査は Research Agent と役割を分けています。

### 3. 3分プロジェクト紹介

私が担当した Retail Insight AI についてご説明します。

小売業の経営企画では、経営会議の前に POS、在庫、商品、会員、販促など複数のデータを確認し、市場や競合の情報も一つの資料にまとめます。そのため、作業負荷が高く、担当者によって分析の観点が変わりやすいという課題があります。

Retail Insight AI は、この経営分析を一つの流れにまとめ、根拠を追跡できる日本語レポートを生成するシステムです。私は Task API、TaskService、LangGraph Workflow、Fixed KPI Workflow、Research Agent、SSE、Report Generator、Docker、アーキテクチャレビューを担当しました。

設計の一つ目のポイントは、長時間処理を同期 API で待たせないことです。Task API が task_id を返し、TaskService が状態を管理し、SSE で進捗を通知します。

二つ目は、KPI と Research の分離です。KPI は再現性が必要なので Fixed KPI Workflow、市場・競合調査は情報源が変わるため Research Agent が担当します。

三つ目は、LangGraph Workflow で State、Node、Edge を明示したことです。条件分岐、途中状態、失敗箇所を追いやすくしました。

Production Gap として、RBAC、Audit Log、PostgreSQL、Redis、OpenTelemetry を優先しています。機能追加より先に、権限、追跡性、永続化、障害調査を整える方針です。

### 4. 担当範囲

バックエンドを中心に、FastAPI の API 設計、Task API、TaskService、LangGraph Workflow、Fixed KPI Workflow、Research Agent、SSE、Report Generator、Docker、アーキテクチャレビューを担当しました。特に、長時間タスクの状態管理、KPI と Research の責務分離、進捗通知からレポート保存までの流れを設計・実装しました。

### 5. なぜ全部 Agent にしないのか

すべてを Agent にすると、同じ入力でも処理経路や結果が変わり、KPI の計算根拠を説明しにくくなります。経営判断に使う数値は Fixed KPI Workflow で処理し、情報源や調査手順が変わる市場・競合調査だけを Research Agent に任せました。

### 6. なぜ LangGraph を使うのか

依頼内容の判定、KPI、Research、Report Generator に条件分岐と失敗時の制御が必要だからです。LangGraph Workflow で State、Node、Edge を明示し、各処理の責務、途中状態、失敗箇所を追いやすくしました。

### 7. なぜ Task API を使うのか

KPI 計算、外部調査、レポート生成を含み、処理時間が一定ではないためです。同期 API ではタイムアウトや二重実行が起きやすいので、Task API で task_id を先に返し、TaskService が状態を管理する構成にしました。

### 8. Production Gap

正式運用に向けて、SSO、RBAC、Audit Log、PostgreSQL、Redis、OpenTelemetry を優先します。誰が何を見られるか、誰がどのデータでレポートを作ったか、障害時にどこで失敗したかを追えることが重要だからです。その後、負荷に応じて RabbitMQ、Kubernetes、バックアップ、ロールバックを段階的に整備します。

### 9. 実務経験を聞かれた場合

Retail Insight AI では、Task API、TaskService、LangGraph Workflow、Fixed KPI Workflow、Research Agent、SSE、Report Generator を設計・実装しました。一方、商用運用に向けた RBAC、Audit Log、OpenTelemetry などは Production Gap として整理しています。実装した範囲と運用に向けて設計した範囲を分けて、具体的にご説明します。

### 10. 弱点を聞かれた場合

現時点で最も強化が必要なのは、複数インスタンスを前提としたタスク実行基盤です。改善する場合は、PostgreSQL を事実の保存先、Redis を短期状態、RabbitMQ を実行キューに分け、冪等性、再試行上限、監視、復旧手順を追加します。
