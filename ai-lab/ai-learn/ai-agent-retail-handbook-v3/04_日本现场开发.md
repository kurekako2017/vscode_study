# 04_日本现场开发

## 第一章 日本现场开发总流程

Retail Insight AI 按日本现场流程推进：需求整理、基本設計、詳細設計、API 設計、開発、単体試験、結合試験、レビュー、部署、保守改修。

【TL Review】

- 每个工程活动是否能对应交付物。
- 是否能说明課題管理、影響調査、障害対応。

【后续扩展】

- 增加各交付物模板。

## 第二章 需求整理

需求整理围绕经营会议进行。确认经营企画、商品部、库存管理、会员运营和门店负责人需要哪些 KPI、哪些报告、哪些数据来源。

【设计决策】

先确认业务问题，再决定 API 和 Workflow。否则系统会变成技术堆叠，无法支撑客户 Review。

【TL Review】

- KPI 定义是否明确。
- 使用者是否明确。
- 数据来源是否明确。

【后续扩展】

- Jira 課題管理。
- 要件定義書。

## 第三章 基本设计

基本設計定义系统范围、整体架构、外部接口、数据来源、权限、非機能要件和保守改修方针。

【设计决策】

基本設計必须能解释为什么采用 Task API、TaskService、LangGraph、Research Agent、SSE 和 Report Store。

【TL Review】

- 架构图是否能对应业务流程。
- 非機能要件是否包含性能、可用性、運用監視。

【后续扩展】

- 增加系统构成图。
- 增加部署构成图。

## 第四章 详细设计

詳細設計落到源码结构：API、schema、service、workflow、kpi、research、report、streaming、db、tests。

【设计决策】

详细设计必须让开发人员知道代码放哪里、异常在哪里处理、日志在哪里记录、测试如何覆盖。

【TL Review】

- 类和函数职责是否单一。
- state 是否可读。
- 失败路径是否明确。

【后续扩展】

- 详细设计书按模块维护。

## 第五章 测试与 Review

测试覆盖単体試験、結合試験、异常系、Streaming 断线、Report 生成、Research timeout、KPI 计算口径。

【设计决策】

Retail Insight AI 的测试不能只确认 API 返回成功，还要确认经营指标、来源、报告结构和状态流。

【TL Review】

- 是否有异常系。
- 是否有結合試験。
- 是否有回归确认。

【后续扩展】

- 自动化测试。
- Load Test。
- 回归基准数据。

## 第六章 部署与保守改修

部署关注 Docker、环境变量、health check、日志、监控、回滚。保守改修关注 KPI 追加、数据源追加、报告格式调整、Research tool 扩展和障害対応。

【设计决策】

部署和保守必须在设计阶段考虑。否则后续性能改善、可観測性、ログ設計、障害対応会成本很高。

【TL Review】

- 环境差异是否控制。
- 日志是否能定位 task_id。
- 改修影响范围是否能评估。

【后续扩展】

- Kubernetes。
- OpenTelemetry。
- 障害票。
- 保守改修フロー。
