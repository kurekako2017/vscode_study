# 03_AI核心知识

## 第一章 知识服务于项目

本书中的知识点只围绕 Retail Insight AI 展开。FastAPI、LangGraph、RAG、Streaming、Docker 都不是孤立知识，而是服务于日本小売業客户的经营分析任务。

【TL Review】

- 是否能把知识点连接到项目模块。
- 是否能说明客户为什么需要该设计。

【后续扩展】

- 每个知识点增加源码目录映射。

## 第二章 FastAPI

客户要求经营分析任务由前端发起，并能查询状态、订阅进度、读取报告。FastAPI 承担 Task API、状态 API、SSE API、Report API 和 health check。

【为什么这样设计】

同步等待完整报告会带来 HTTP Timeout。FastAPI 负责立即承接请求并返回 task_id，后续由 TaskService 和 SSE 处理。

【源码映射】

```text
app/api/tasks.py
app/api/streams.py
app/api/reports.py
app/api/health.py
app/schemas/
```

【TL Review】

- API schema 是否清晰。
- 错误码是否统一。
- request_id 是否进入日志。
- health check 是否覆盖依赖组件。

【后续扩展】

- API Gateway
- SSO
- Rate Limit
- OpenAPI Review

## 第三章 LangGraph Workflow

Retail Insight AI 的任务包含 route、KPI workflow、Research Agent、Report Generation。LangGraph 用 State、Node、Edge、checkpoint 表达业务顺序。

【为什么这样设计】

普通函数可以完成简单调用，但难以表达状态、条件路由、失败恢复和 Review 边界。LangGraph 让 Workflow 可視化、可测试、可保守改修。

【源码映射】

```text
app/workflow/state.py
app/workflow/graph.py
app/workflow/nodes/
app/workflow/checkpoint.py
```

【TL Review】

- state 是否过大。
- node 是否职责单一。
- route 是否有 fallback。
- checkpoint 是否能支持障害対応。

【后续扩展】

- interrupt / resume
- retry policy
- node latency metrics

## 第四章 Fixed KPI Workflow

销售、库存、商品、会员、促销 KPI 用于经营判断。系统使用 Fixed KPI Workflow 保证口径稳定，避免每次分析逻辑变化。

【为什么这样设计】

全部交给 Agent 会增加数字口径漂移风险。固定 Workflow 让 KPI 可测试、可 Review、可追溯。

【源码映射】

```text
app/kpi/sales.py
app/kpi/inventory.py
app/kpi/product.py
app/kpi/member.py
app/kpi/promotion.py
```

【TL Review】

- KPI 公式是否有业务确认。
- 异常值是否处理。
- 促销期间和通常期间是否区分。

【后续扩展】

- PostgreSQL
- materialized view
- batch aggregation

## 第五章 Research Agent

Research Agent 处理市场趋势、竞品调查、内部资料检索。它通过 Tool Layer 获取信息，并保留来源进入报告。

【为什么这样设计】

市场和竞品信息具有不确定性，固定 Workflow 无法覆盖全部调查路径。Research Agent 负责弹性信息收集，但不接管 KPI 计算。

【源码映射】

```text
app/research/agent.py
app/research/tools.py
app/research/sources.py
app/research/prompts.py
```

【TL Review】

- tool 是否有 timeout。
- 来源是否保留。
- 内部资料是否有权限过滤。
- Research 失败是否影响报告主流程。

【后续扩展】

- OpenSearch
- VectorDB
- Tool Allowlist
- Prompt Injection Guard

## 第六章 RAG

RAG 用于内部资料、月报、商品资料、促销规则和会议资料检索。Retail Insight AI 中，RAG 服务于 Research Agent 和 Report Generation。

【为什么这样设计】

经营报告需要根拠。没有检索和来源保留，管理层无法 Review 报告结论。

【源码映射】

```text
app/rag/ingest.py
app/rag/chunk.py
app/rag/retriever.py
app/rag/rerank.py
app/rag/citation.py
```

【TL Review】

- Citation 是否来自真实来源。
- 权限过滤是否在检索前执行。
- TopK 是否通过评估调整。

【后续扩展】

- Hybrid Search
- Rerank
- ACL Filter
- Evaluation Set

## 第七章 Streaming / SSE

Streaming 让前端显示任务进度。SSE 承接 status、token、error、done。

【为什么这样设计】

经营分析任务持续时间不固定。SSE 能避免用户长时间等待无反馈，也能降低同步 API 超时风险。

【源码映射】

```text
app/streaming/events.py
app/streaming/sse.py
app/service/event_bus.py
```

【TL Review】

- error 后是否停止 loading。
- done 是否只在成功时发送。
- 断线后能否恢复状态。

【后续扩展】

- heartbeat
- reconnect
- Redis pub/sub

## 第八章 Docker

Docker 统一 Backend、Frontend、依赖和环境变量，使 Review、結合試験、部署准备更稳定。

【为什么这样设计】

日本现场需要可重复运行环境。Docker 是从本地开发到 ECS / EKS / Kubernetes 的基础。

【源码映射】

```text
Dockerfile
docker-compose.yml
.env.example
deployment/
```

【TL Review】

- secret 是否进入镜像。
- health check 是否存在。
- log 是否输出到 stdout。

【后续扩展】

- CI/CD
- image scan
- Kubernetes manifest
