このように、Retail Insight AI では、AI を使うこと自体ではなく、小売業の経営分析を正確で追跡可能な業務プロセスにすることを重視して設計・実装しました。

## 第四章 技术设计问答

<a id="quick-task-api"></a>

### なぜ同期 API ではなく Task API にしましたか。

```text
経営分析は、KPI 計算、外部調査、Report Generator の処理を含み、完了時間が一定ではありません。
同期 API で完了まで待つと、HTTP タイムアウトや再送による二重実行が起きやすく、利用者も進捗を確認できません。
そのため、Task API で task_id を先に返し、TaskService がライフサイクルを管理し、SSE で進捗を通知する構成にしました。
```

### なぜ FastAPI を使いましたか。

```text
分析処理と LangGraph Workflow を Python で実装しているため、API 層も同じ言語にするとデータモデルや例外処理を共有しやすいからです。
また、Pydantic による入力検証、非同期 API、SSE を整理して実装でき、Task API の要件に合っていると判断しました。
```

<a id="quick-langgraph"></a>

### なぜ LangGraph を使いましたか。

```text
依頼内容の判定、KPI 分析、Research、Report Generator という複数の処理に、条件分岐と失敗時の制御が必要だからです。
LangGraph Workflow で State、Node、Edge を明示し、各 Node の入出力を限定することで、レビューや障害調査をしやすくしました。
```

【TL Review】

- 技术理由是否来自客户需求。
- 是否说明 trade-off。
- 是否能说明不用同步 API、不用全部 Agent 的理由。

【后续扩展】

- 增加性能改善回答。
- 增加障害対応回答。

## 第五章 Agent 与 Workflow 问答

### なぜ Fixed KPI Workflow と Research Agent を分けましたか。

```text
KPI は経営判断の根拠になるため、計算式や集計条件を固定し、同じ入力から同じ結果を出す必要があります。
一方、市場調査や競合調査は、質問によって情報源や調査手順が変わります。
そのため、KPI は Fixed KPI Workflow、調査は Research Agent と役割を分け、正確性と柔軟性を両立させました。
```

### Research Agent は何をしますか。

```text
市場トレンド、競合調査、内部資料の検索を担当します。
Tool Layer を通じて情報を取得し、出典、更新日、要約を保持した上で Report Generator へ渡します。
また、利用できる Tool、実行回数、タイムアウトを制限し、調査処理の境界を明確にしています。
```

【TL Review】

- Agent 的职责是否清晰。
- KPI 是否避免自由生成逻辑。
- Research 超时是否有 fallback。

【后续扩展】

- 实际面谈前，按目标现场补充 Tool Allowlist 与权限规则的具体名称。

## 第六章 Production Gap 问答

### 運用拡張ポイントは何ですか。

```text
将来的な運用を見据え、まず SSO、RBAC、Audit Log、PostgreSQL、Redis、OpenTelemetry を優先します。
小売業の売上や会員データを扱うため、誰が何を閲覧できるか、どのデータからレポートを作ったか、障害時に処理を追跡できるかが重要だからです。
その後、負荷と可用性の要件に応じて、タスクキュー、CI/CD、Kubernetes、バックアップ、ロールバックを段階的に整備します。
```

### 障害が起きた場合はどう調査しますか。

```text
まず影響範囲、発生時刻、障害が継続しているかを確認し、必要であれば処理の停止や切り戻しを判断します。
その後、task_id と trace_id を基に、API、TaskService、LangGraph Workflow の Node、KPI、Research Tool、Report Generator の順に追います。
復旧を優先し、原因分析と恒久対応は切り分けて進めます。
```

## 第七章 深掘り質問集

### 1. この案件で一番工夫した点は何ですか。

**面接官質問（日语）**

この案件で一番工夫した点は何ですか。

**回答例（日语）**

一番工夫したのは、正確性が必要な KPI 分析と、柔軟性が必要な市場・競合調査を分けた点です。KPI は Fixed KPI Workflow で計算し、Research Agent は調査に専念させました。これにより、数値の再現性と調査の柔軟性を両立しました。

**回答ポイント（中文）**

先说业务上的“准确性与灵活性冲突”，再说如何通过职责分离解决，不要把亮点说成技术堆砌。

### 2. なぜ小売業をテーマにしましたか。

**面接官質問（日语）**

なぜ小売業をテーマにしたのですか。

**深掘りされた場合（日语）**

特に、経営会議前の情報整理は、複数の部門とデータソースをまたぎます。そのため、データの追跡性、部門ごとの権限、レポートの一貫性まで含めて考える必要があり、実際の業務に近い設計課題を扱える点を重視しました。

**回答ポイント（中文）**

### 3. なぜ全部 Agent にしなかったのですか。

**面接官質問（日语）**

なぜすべての処理を Agent にしなかったのですか。

**回答例（日语）**

すべてを Agent にすると、同じ入力でも処理経路や結果が変わり、KPI の計算根拠を説明しにくくなるためです。売上や在庫など、経営判断に直接使う数値は固定ロジックで処理し、情報の探し方が変わる市場・競合調査だけを Research Agent に任せました。

**深掘りされた場合（日语）**

Agent の入力と出力、利用可能な Tool、タイムアウト、最大実行回数を明確にしています。調査結果は出典付きで Report Generator へ渡します。

**NG回答**

**回答ポイント（中文）**

从可重复、可解释、可测试角度说明边界设计，不要表现成能力不足或工期妥协。

<a id="quick-fixed-kpi"></a>

### 4. なぜ KPI は Fixed KPI Workflow にしましたか。

**面接官質問（日语）**

なぜ KPI は Fixed KPI Workflow にしたのですか。

**回答例（日语）**

KPI は経営判断の根拠になるため、計算式、対象期間、集計条件を明確にし、同じ入力から同じ結果を出す必要があります。Fixed KPI Workflow にすることで、単体テストと変更時の影響調査をしやすくしました。

**深掘りされた場合（日语）**

KPI ごとに定義、入力項目、欠損値の扱い、丸め規則、バージョンを管理する想定です。レポートには利用した定義バージョンとデータ期間を紐づけ、後から数字を再現できるようにします。

**NG回答**

固定処理の方が Agent より簡単だからです。

**回答ポイント（中文）**

重点是经营数字的口径、版本和可追溯性，体现企业系统思维。

<a id="quick-research-agent"></a>

**深掘りされた場合（日语）**

分離することで、Research Agent がタイムアウトしても KPI 分析を継続できます。また、内部資料検索と外部調査で Tool や権限を分け、Audit Log も残しやすくなります。

Agent という名前の独立した機能を作りたかったからです。

**回答ポイント（中文）**

回答职责、故障隔离、工具权限和审计，不要只说“灵活”。

### 7. LangChain ではなく LangGraph を使う理由は何ですか。

**面接官質問（日语）**

LangChain ではなく、LangGraph を使う理由は何ですか。

**回答例（日语）**
