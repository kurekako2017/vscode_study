## 第四章 技术设计问答

### なぜ同期 API ではなく Task API にしましたか。

```text
经营分析任务可能持续几十秒甚至更长。同步 API 会带来 HTTP Timeout 和用户等待体验问题。
因此 Task API 立即返回 task_id，TaskService 管理生命周期，前端通过 SSE 获取状态。
```

### なぜ FastAPI を使いましたか。

```text
FastAPI は Task API、状態確認 API、SSE、レポート取得 API、health check を整理しやすいです。
Python の分析処理、LangGraph Workflow、Streaming とも相性が良いためです。
```

### なぜ LangGraph を使いましたか。

```text
Retail Insight AI の処理は route、KPI workflow、Research Agent、report generation に分かれます。
LangGraph を使うことで State、Node、Edge、checkpoint を明確にし、Review と障害対応をしやすくしています。
```

【TL Review】

- 技术理由是否来自客户需求。
- 是否说明 trade-off。
- 是否能说明不用同步 API、不用全部 Agent 的理由。

## 第五章 Agent 与 Workflow 问答

### なぜ固定 KPI Workflow と Research Agent を分けましたか。

```text
KPI は経営判断に使われるため、計算口径の安定性と再現性が重要です。
市場調査や競合調査は情報源が変わるため Research Agent に任せます。
固定 Workflow と Agent を分けることで、安定性と柔軟性を両立しています。
```

### Research Agent は何をしますか。

```text
市場トレンド、競合調査、内部資料検索を担当します。
Tool Layer を通じて情報を取得し、来源を保留した上で Report Generator に渡します。
```

【TL Review】

- Agent 的职责是否清晰。
- KPI 是否避免自由生成逻辑。
- Research 超时是否有 fallback。

【后续扩展】

- 增加 Tool Allowlist 回答。
- 增加权限过滤回答。

## 第六章 Production Gap 问答

### 運用拡張ポイントは何ですか。

```text
SSO、RBAC、Audit Log、PostgreSQL、Redis、RabbitMQ、VectorDB、OpenSearch、OpenTelemetry、CI/CD、Kubernetes、Secrets Manager、Load Test、Backup、Rollback です。
特に小売業では、売上や会員データの権限、監査、可観測性が重要です。
```

### 障害が起きた場合はどう調査しますか。

```text
task_id、request_id、API log、SSE event、LangGraph state、KPI workflow、Research Agent tool、Report Generation の順に確認します。
OpenTelemetry を導入する場合は trace_id で全体を追跡します。
```
