# 05_TL代码审查

## 第一章 Review 总原则

TL Review 的目标是确认 Retail Insight AI 能支撑日本小売業客户的经营分析、运用监视、障害対応和保守改修。

【TL Review】

- 是否从客户业务出发。
- 是否有职责分离。
- 是否有日志、异常、测试、权限和 Production Gap。

【后续扩展】

- 增加 Review 票格式。

## 第二章 API Review

Task API 必须立即返回 task_id。状态查询、SSE、报告获取和 health check 必须职责清晰。

【Review 重点】

- API 是否幂等。
- 错误码是否统一。
- request_id 是否记录。
- 同步等待是否被避免。

【后续扩展】

- API Gateway。
- SSO。
- Rate Limit。

## 第三章 TaskService Review

TaskService 管理任务生命周期，不能让 API 层直接执行 Workflow。

【Review 重点】

- task status 是否完整。
- 失败是否记录。
- 重试是否可扩展。
- report 失败是否可追踪。

【后续扩展】

- Redis。
- RabbitMQ。
- retry policy。

## 第四章 LangGraph Review

LangGraph 负责 route、KPI workflow、Research Agent、Report Generation。

【Review 重点】

- State 是否过大。
- Node 是否职责单一。
- Edge 是否可读。
- checkpoint 是否能帮助障害対応。

【后续扩展】

- interrupt / resume。
- node metrics。

## 第五章 Research Agent Review

Research Agent 负责市场、竞品、内部资料检索，不能接管 KPI 计算。

【Review 重点】

- Tool 是否有 timeout。
- 来源是否保留。
- 权限是否可扩展。
- Prompt 是否限制输出格式。

【后续扩展】

- OpenSearch。
- VectorDB。
- Tool Allowlist。

## 第六章 Streaming Review

SSE 承接 status、token、error、done。前端必须正确处理失败和断线。

【Review 重点】

- error 后是否停止 loading。
- done 是否只代表成功。
- 断线后是否能恢复状态。
- proxy buffering 是否考虑。

【后续扩展】

- heartbeat。
- reconnect。
- Redis pub/sub。

## 第七章 Report Review

Report Generator 生成日文经营分析报告，必须保留 KPI、Research、来源、风险和确认点。

【Review 重点】

- 报告结构是否稳定。
- 来源是否可追踪。
- KPI 口径是否明确。
- 日语表达是否适合管理层。

【后续扩展】

- report template version。
- Audit Log。
- approval workflow。
