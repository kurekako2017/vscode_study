# 试验结果报告

## 1. 目的

本报告用于记录 `rag_api_demo` 最小 PoC 的试验结果。

试验重点不是证明该系统已经达到生产可用，而是确认以下基础链路已经成立：

- 服务可启动
- 文档可加载
- `RAG` 问答接口可调用
- 来源信息可返回
- `PDF` 文档可纳入处理范围

## 2. 试验对象

- 项目：`rag_api_demo`
- 形态：本地 `FastAPI` 后端 PoC
- 主要功能：
  - `GET /health`
  - `POST /ask`
  - `POST /reload`
  - `md / txt / pdf` 文档读取
  - 基于关键词重叠的最小检索

## 3. 试验范围

本次试验覆盖以下内容：

- 启动确认
- 健康检查接口
- 正常问答
- 来源返回
- 文档重载
- `PDF` 读取
- 基础异常系

本次不覆盖：

- 认证与权限
- 向量检索
- 并发性能
- 企业网络接入
- 持久化日志

## 4. 试验环境

| 项目 | 内容 |
|---|---|
| 运行方式 | 本地运行 |
| 应用框架 | `FastAPI` |
| 模型调用 | OpenAI Responses API |
| 文档类型 | `md / txt / pdf` |
| 检索方式 | 关键词重叠 |

## 6. 总体判定

本次 PoC 的总体判定为：

- `通过（PoC 级）`

原因如下：

- 目标中的最小链路已经成立
- `RAG API` 的服务化结构已经具备
- 文档来源返回和 `PDF` 支持已经接上

但需要明确：

- 这不是企业正式版验收结果
- 这只是学习版和 PoC 级验证结果

## 7. 发现的问题与限制

### 7.1 检索质量有限

当前检索方式是关键词重叠，不是向量检索，因此存在：

- 表述稍有变化就可能命中不足
- 同义表达支持较弱
- 多文档场景下排序能力有限

### 7.2 `PDF` 抽取质量受文件本身影响

对于扫描版或排版复杂的 `PDF`，文本抽取质量可能较差，因此：

- 不保证所有 `PDF` 都能得到高质量结果
- 正式项目中通常还要补 OCR、版面处理或更稳定的文档解析方案

### 7.3 企业级要件尚未覆盖

当前版本未覆盖：

- 认证
- 权限
- 审计日志
- 监控
- 性能压测
- 多用户隔离

## 8. 后续课题

如果继续往日本现场可交付 PoC 的方向推进，建议下一步优先补：

1. 向量检索或混合检索
2. 简化版评估记录
3. 认证和访问控制设计
4. 日志与错误追踪
5. 更稳定的 `PDF / Office` 文档解析

## 9. 结论

`rag_api_demo` 已经可以作为一个贴近日本现场案件方向的最小 `RAG API` 学习样例使用。

它当前最适合用于说明这些能力：

- `FastAPI + RAG`
- 文档问答 API 化
- `PDF` 纳入文档处理范围
- 来源返回与最小可追溯性

如果作为作品集展示，建议明确表述为：

- 一个面向社内搜索 / 文档问答场景的 `RAG API` PoC
- 已完成最小服务化和基础试验
- 后续仍需补企业级能力
