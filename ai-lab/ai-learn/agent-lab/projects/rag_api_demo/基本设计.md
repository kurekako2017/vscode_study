# rag_api_demo 基本设计

## 1. 系统概要

`rag_api_demo` 是一个将本地文档问答能力服务化的最小 `FastAPI` 后端。

其目标是提供：

- 文档加载
- 文档检索
- 文档问答
- HTTP 接口调用

从角色上看，`FastAPI` 应用是服务端，`curl`、浏览器、前端或其他系统是客户端。

## 2. 系统构成

| 模块 | 作用 |
|---|---|
| FastAPI 应用 | 提供 HTTP 接口 |
| 文档加载模块 | 读取 `md / txt / pdf` |
| 切分模块 | 按固定大小切分文本 |
| 检索模块 | 基于关键词重叠检索片段 |
| 模型调用模块 | 调用 OpenAI Responses API |
| 状态保持模块 | 将文档目录、片段、客户端保存在 `app.state`，并支持按需加载 |

## 3. 接口设计

### 3.1 `GET /health`

用途：

- 健康检查
- 确认服务当前状态

返回：

- `status`
- `docs_dir`
- `chunk_count`

### 3.2 `POST /ask`

用途：

- 根据问题查询本地资料并返回回答

请求项：

- `question`
- `model`

返回项：

- `answer`
- `model`
- `docs_dir`
- `source_count`
- `sources`

### 3.3 `POST /reload`

用途：

- 重载文档目录中的资料
- 更新内存中的片段数据

返回项：

- `docs_dir`
- `chunk_count`

## 4. 处理流程

### 4.1 启动流程

1. 启动 `uvicorn`
2. 创建 `FastAPI` 应用
3. 不立即扫描文档，不立即预热索引
4. 等首次请求进入后再执行 `ensure_state_loaded()`
5. 再由 `load_state()` 完成文档扫描、切分和缓存
6. 如果是 Real 模式，再初始化 OpenAI 客户端

### 4.2 `/ask` 流程

1. 接收问题
2. 调用 `ensure_state_loaded()`
3. 基于问题执行本地检索
4. 取 Top-K 片段
5. 构建提示上下文
6. 调用模型生成回答
7. 返回答案和来源列表

### 4.3 `/reload` 流程

1. 重新扫描文档目录
2. 重建片段数据
3. 更新 `app.state`
4. 返回新的片段数

### 4.4 `GET /health` 流程

1. 调用 `ensure_state_loaded()`
2. 确认服务已经完成状态初始化
3. 返回 `status`、`docs_dir`、`chunk_count`

## 5. 文档读取设计

支持文件类型：

- `.md`
- `.txt`
- `.pdf`

读取方式：

- `md / txt`：按 `UTF-8` 文本读取
- `pdf`：使用 `pypdf` 抽取页面文本

对于无法读取的文件：

- 跳过

## 6. 检索设计

本样例采用最简单的关键词重叠检索：

- 对问题和片段分别提取 token
- 根据重叠 token 数量计算分数
- 取得分最高的前 `TOP_K`

该方式适合学习和 PoC，后续可扩展为：

- 向量检索
- 混合检索
- 外部搜索引擎

## 7. 状态管理设计

服务运行时将以下对象保存在 `app.state`：

- `client`
- `docs_dir`
- `chunks`

这样可以避免每次请求都重新扫描目录。
在当前实现里，这些状态不是在启动时一次性准备好，而是首次访问相关接口时按需生成。

## 8. 错误处理

| 场景 | 处理方式 |
|---|---|
| `OPENAI_API_KEY` 未配置 | 启动失败 |
| `OPENAI_API_KEY` 未配置但使用 Mock 模式 | 可以正常运行 |
| `RAG_API_DOCS_DIR` 不存在 | 启动失败 |
| 无可读资料 | 启动失败 |
| 模型调用失败 | 返回 `500` |
| 服务状态未初始化 | 首次请求时自动初始化 |

## 9. 限制事项

- 只适合本地 PoC
- 不支持认证
- 不支持多用户隔离
- 不支持向量检索
- 不支持持久化日志
