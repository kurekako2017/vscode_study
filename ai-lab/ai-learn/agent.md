# ai-learn 示例文档流程图展开规则

这个文件专门记录 `ai-learn` 目录下示例文档的统一写法，重点是把“流程图”写成读者能直接顺着看懂的链路。

## 1. 统一原则

- 只要示例里出现流程图，就要同时写出“图中节点对应代码 / 概念 / 数据结构”的对照表。
- 流程图不要只画抽象箭头，必须展开到可落地的步骤，例如 `Loader`、`Chunk`、`Retriever`、`Context`、`LLM`、`Output`。
- 如果示例是 RAG，优先按这条主链路展开：

`文档目录 -> 文档切分 -> chunk 集合 -> 用户提问 -> 检索 -> Top-K -> 上下文拼接 -> LLM -> 输出答案`

- 如果示例里还有 `索引构建`、`重排`、`权限过滤`、`工具调用`、`记忆`、`状态机` 等步骤，就把它们插到主链路中间，再补对应说明。

## 2. 每个例子必须使用的解释模板

每个可运行示例的主 `README.md` 都必须按下面的顺序解释。不能只放代码、命令或一张抽象 Mermaid 图。

### 2.1 示例目标

- 用一句话说明这个例子解决什么问题。
- 给出一个现实业务场景，并说明不使用这个方案时的困难。

### 2.2 最小输入示例

- 展示一条可以直接运行的命令、HTTP 请求或函数调用。
- 输入依赖文件时，同时展示最小目录树和一小段文件内容。
- 至少解释一个正常输入；存在分支、失败或安全边界时，再增加一个对照输入。

### 2.3 处理前的数据

- 展示进入流程前最重要的数据形态，例如文档目录、JSON、State、工具参数或消息列表。
- 只保留理解流程所需字段，不堆砌无关内容。

### 2.4 整体流程图（纯文本展开版）

- 主流程图必须使用 `text` 代码块和树状连接符，保证 VS Code、GitHub 和普通 Markdown 阅读器都能显示。
- 从真实入口开始，展开到最终输出；不能省略检索、校验、路由、状态更新、工具执行、上下文组装等关键中间步骤。
- 每个关键节点优先标出对应的函数、类、文件或框架。
- 有分支和循环时必须画出分支条件、回流位置和终止条件。
- Mermaid 或 SVG 可以保留作辅助图，但不能代替纯文本主流程图。

统一样式：

```text
用户输入 / 文件 / HTTP 请求
│
▼
入口：main() / FastAPI route / graph.invoke()
│
▼
参数或数据校验
├── 失败 -> 错误 / 拒绝 / fallback
└── 成功
    │
    ▼
核心处理函数
│
▼
模型 / 工具 / 检索器 / 状态图
│
▼
结果整理与来源说明
│
▼
最终输出
```

### 2.5 输出与逐节点解释

- 展示与最小输入对应的最小输出，必要时注明输出为示意值。
- 紧跟“流程节点 -> 文件/函数/类 -> 输入 -> 输出 -> 为什么需要”的对应表；节点很少的最小示例可以改用一段 `节点对应：` 说明，但必须点出真实函数或组件。
- 说明输出中的哪些部分是确定字段、哪些部分会随输入、模型或数据变化。

### 2.6 下一步理解与企业级演进（强制）

解释完当前例子后，必须继续写“下一步理解”以及“企业级 / 现实案例中的完整流程”。这一节不是普通的学习建议，不能省略，也不能只列几个框架名称。

必须同时完成下面四件事：

1. 用一句等式或映射说明当前教学实现的本质，例如 `retrieve() = 关键词匹配`、`list[Chunk] = 内存索引`。
2. 说明真实项目会把当前哪一个函数、数据结构或本地组件替换成什么生产组件。
3. 使用 `text` 代码块画出企业级流程图，从真实数据源一直展开到输出、评估和运维环节。
4. 给出一个现实业务案例，解释数据从哪里来、谁发起请求、经过哪些组件、失败时如何处理，以及最终交付给谁。

统一写法：

```text
当前教学例子
核心函数 / 本地数据结构
=
它所模拟的真实能力

↓ 下一步替换或扩展

真实数据源
│
▼
生产级处理组件
│
▼
检索 / 模型 / 工具 / 工作流
├── 成功 -> 业务输出
└── 失败 -> 重试 / fallback / 人工处理
    │
    ▼
评估 + 监控 + 审计 + 持续改进
```

企业级流程图必须根据例子类型继续展开：

- `RAG`：数据接入、清洗、Chunk、Embedding、向量库、关键词索引、Metadata/ACL、Hybrid Search、Rerank、Top-K、Context、LLM、引用、评估、监控和索引更新。
- `Tool Calling / MCP`：身份认证、工具发现、Schema、权限策略、参数校验、执行、超时重试、人工审批、结果回填和审计。
- `Workflow / LangGraph`：State、Node、条件路由、循环终止、Checkpoint、Interrupt、持久化、失败恢复和人工介入。
- `结构化输出`：Schema 版本、模型生成、校验、修复/重试、业务规则、持久化、下游 API 和质量监控。
- `模型调用 / LangChain`：Prompt 管理、模型网关、Provider 回退、限流、缓存、Tracing、成本统计和内容安全。
- `API / Streaming / Frontend`：认证、请求校验、任务状态、流式协议、断线恢复、错误合同、可观测性和前端状态管理。
- `Multi-Agent`：Supervisor、角色边界、共享状态、消息协议、预算、终止保护、质量门禁和人工审批。

RAG 类示例至少要给出下面这种从教学实现到真实系统的认知桥梁：

```text
当前例子
retrieve()
=
关键词匹配

↓ 企业级 RAG

文档 / Wiki / 数据库 / 工单
│
▼
清洗与 Chunk
│
▼
Embedding + 关键词索引
│
▼
向量数据库（FAISS / Chroma / Qdrant 等）
│
▼
Metadata / ACL 过滤
│
▼
Hybrid Search -> Rerank -> Top-K
│
▼
Context 组装 -> LLM -> 答案与引用
│
▼
检索评估 + 答案评估 + 监控 + 索引更新
```

不能只写“下一步学习 LangChain / LangGraph”。必须解释学习它们以后，当前例子中的哪一部分会被替换、增强或纳入更完整的工程链路。

## 3. RAG 类示例的固定展开顺序

对于 `doc_qa_agent`、`rag_api_demo`、`advanced_rag` 这类例子，建议统一用下面的顺序写：

1. 资料目录
2. 读取与切分
3. chunk 集合
4. 用户问题
5. 检索与打分
6. Top-K 选择
7. 上下文拼接
8. 模型生成
9. 答案与来源输出

## 4. 写法要求

- README 先让人看懂全链路，再看代码。
- 流程图要能单独解释每一步，不要只留下“输入 -> 输出”这种过于粗的图。
- 如果同一主题有多个示例，尽量复用同一套节点命名，方便横向对比。
- 图里的每个节点最好都能在正文里找到对应的函数名或模块名。
- 同一示例存在 `README.md` 和 `README_LEARN.md` 时，完整模板放在主 `README.md`；学习补充文档通过链接引用，不复制一套容易失效的流程。
- 每个主 `README.md` 的“图片式模板解释”后面必须出现“下一步理解与企业级演进”，两节共同构成一个完整示例说明。
- 企业级流程不能为了显得复杂而堆组件；每个新增组件都要对应当前例子的真实限制、失败模式或业务需求。

## 5. Mermaid 显示兼容规则

- 辅助 Mermaid 流程图统一使用 ```` ```mermaid ```` 代码块，不使用大小写变体或自定义围栏。
- VS Code `1.121` 及以上版本使用内置 Mermaid 渲染器，不安装或推荐已废弃的 `bierner.markdown-mermaid`，避免两个渲染器同时激活后出现空白图表。
- 含函数调用、路径、括号、斜杠、井号或 HTML 换行的节点文本统一放在双引号中，例如 `A["build_context()<br/>拼接上下文"]`。
- 新增或修改流程图后，至少在 VS Code Markdown 预览中检查一次；如果只显示空白画布，先检查是否存在重复 Mermaid 扩展，再检查图表语法。

## 6. 测试清单的两列规则

- 只要文档里出现“测试清单”或“测试用例表”，必须优先写成两列对照表，表头固定为：
  - `真实模型 / 真实服务命令`
  - `纯 Mock / 本地命令`
- 左列只放会触发真实模型、真实服务或真实外部依赖的命令，例如真实 API、真实向量库、真实 MCP 服务、真实数据库、真实浏览器或真实 HTTP 服务。
- 右列只放不依赖外部真实服务的命令，例如 `--mock`、`--mode mock`、`--backend memory`、本地脚本、内存实现、纯单元测试或本地文件回放。
- 两列必须表达同一条测试意图的“真实版”和“本地版”对照关系，不要把不相关的命令拆成两个孤立列表。
- 如果某个示例本身就不调用 LLM 或不接真实服务，左列必须明确写 `不适用（不调用 LLM）`、`不适用（不调用真实服务）` 或同义表述，右列仍然写本地可运行命令或本地测试命令。
- 如果某个示例只有真实版或只有本地版，也要解释原因，不能把测试清单写成单列；必要时用“说明”补一句为什么另一列不适用。
- 每一行最好对应同一个输入或同一个业务场景，便于读者直接比较真实模式和本地模式的差异。
- 不要把“安装命令”“环境准备命令”和“功能测试命令”混在同一行，测试清单优先展示可验证行为本身。

## 7. Codex 执行时的强制检查规则

当 Codex 修改 `ai-learn` 下任意示例文档、README、教程或流程图说明时，必须主动检查是否已经写入“继续学习 / 扩展延伸 / 企业级演进”的理解部分。没有写入时，必须追加；写得太浅时，必须补全。

重点不是简单写一句“下一步学习 LangChain / LangGraph / MCP”，而是要把当前教学例子和真实企业级系统之间的关系讲清楚。

### 7.1 必须检查的位置

优先检查并补全每个示例的主文档：

```text
README.md
README_LEARN.md
基本设计.md
项目简介.md
测试说明.md
```

如果同一示例同时存在多个文档：

- 主流程、完整解释、企业级演进优先写进 `README.md`。
- `README_LEARN.md` 可以补充学习笔记，但不要和 `README.md` 重复一整套内容。
- 图片、Mermaid、SVG 下面必须有纯文本流程图或逐节点解释，不能只留图。

### 7.2 必须写出的认知桥梁

每个例子都必须出现类似下面的“当前例子 -> 真实系统”的映射：

```text
当前教学例子
某个函数 / 类 / 本地数据结构
=
它模拟的真实工程能力

↓ 下一步替换或增强

真实项目中的生产组件 / 工程链路
```

示例：

```text
当前例子
retrieve()
=
关键词匹配检索

↓ 企业级 RAG

Embedding + 向量数据库 + Hybrid Search + Rerank + 权限过滤 + 评估监控
```

```text
当前例子
RunnableLambda(mock_llm)
=
本地模拟模型调用

↓ 企业级 LangChain 调用链

Prompt 管理 + 模型网关 + Provider fallback + 限流 + 缓存 + Tracing + 成本统计
```

```text
当前例子
State 字典
=
最小工作流状态

↓ 企业级 LangGraph

Typed State + Node + 条件路由 + Checkpoint + Interrupt + 人工审批 + 失败恢复
```

### 7.3 企业级 RAG 延伸必须写完整

凡是 RAG、文档问答、知识库、检索增强生成相关示例，Codex 必须追加下面这种完整流程，不允许只写“以后换成向量数据库”。

```text
当前教学版 RAG
本地文档目录
│
▼
build_chunks()
读取 .md / .txt 并切分 chunk
│
▼
list[Chunk]
内存中的文档片段集合
│
▼
retrieve()
关键词重合度打分
│
▼
Top-K chunks
命中的少量片段
│
▼
build_context()
把片段拼成上下文
│
▼
LLM
基于上下文回答
│
▼
Answer + Sources
答案与来源

↓ 企业级 RAG

文档 / Wiki / 数据库 / 工单 / PDF / 网页
│
▼
数据接入
定时同步、增量更新、权限读取
│
▼
清洗与标准化
去重、去噪、格式统一、敏感信息处理
│
▼
Chunk 策略
按标题、段落、Token、语义边界切分
│
▼
Embedding
把 chunk 转成向量
│
▼
索引层
向量数据库 + 关键词索引 + Metadata
│
▼
权限过滤 ACL
只允许用户检索自己有权限看的资料
│
▼
Hybrid Search
向量检索 + 关键词检索
│
▼
Rerank
对候选结果重新排序
│
▼
Top-K
选出最有用的证据片段
│
▼
Context 组装
压缩、去重、引用标签、Token 控制
│
▼
LLM 生成
只基于上下文回答，不足时明确说明
│
▼
答案与引用
返回答案、来源、可信度或证据片段
│
▼
评估与监控
检索命中率、答案正确率、延迟、成本、失败日志
│
▼
索引更新与持续改进
新增文档同步、旧文档失效、反馈回流
```

### 7.4 现实业务案例必须落地

企业级演进后必须补一个现实案例，至少回答这几个问题：

- 数据从哪里来？例如公司 Wiki、FAQ、合同、客服工单、数据库。
- 谁发起请求？例如客服、社员、客户、管理员、业务系统。
- 中间经过哪些组件？例如 API、检索器、向量库、权限过滤、Rerank、LLM。
- 失败时怎么办？例如无结果、权限不足、模型失败、超时、人工处理。
- 最终交付给谁？例如前端页面、Slack、管理后台、客服系统。

RAG 示例可以使用下面这种案例结构：

```text
现实案例：公司内部知识库问答

社员在 Web 页面提问
│
▼
后端 API 接收问题并确认登录身份
│
▼
根据身份加载可访问部门和项目权限
│
▼
在 Wiki、FAQ、工单索引中做 Hybrid Search
│
▼
ACL 过滤掉无权限文档
│
▼
Rerank 选出最相关 Top-K
│
▼
组装 Context 并调用 LLM
├── 成功 -> 返回答案 + 引用来源
├── 无资料 -> 提示资料不足，并建议联系负责人
└── 模型或检索失败 -> fallback 到关键词搜索 / 人工工单
│
▼
记录日志、成本、命中率和用户反馈
```

### 7.5 Codex 修改完成后的自检清单

每次 Codex 修改示例文档后，必须自检：

- 是否有“整体流程图”？
- 是否有“逐节点解释”？
- 是否解释了打印结果或输出字段？
- 是否有“下一步理解与企业级演进”？
- 是否写出了 `当前函数 = 真实能力` 的映射？
- RAG 示例是否包含企业级 RAG 完整链路？
- 是否有现实业务案例？
- 测试清单是否是“真实模型 / 真实服务命令”和“纯 Mock / 本地命令”两列？
- Mermaid / SVG 在深色主题下是否看得清？看不清时是否提供纯文本流程图？
