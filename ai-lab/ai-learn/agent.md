# ai-learn 示例文档流程图展开规则

这个文件专门记录 `ai-learn` 目录下示例文档的统一写法，重点是把“流程图”写成读者能直接顺着看懂的链路。

## 1. 统一原则

- 只要示例里出现流程图，就要同时写出“图中节点对应代码 / 概念 / 数据结构”的对照表。
- 流程图不要只画抽象箭头，必须展开到可落地的步骤，例如 `Loader`、`Chunk`、`Retriever`、`Context`、`LLM`、`Output`。
- 如果示例是 RAG，优先按这条主链路展开：

`文档目录 -> 文档切分 -> chunk 集合 -> 用户提问 -> 检索 -> Top-K -> 上下文拼接 -> LLM -> 输出答案`

- 如果示例里还有 `索引构建`、`重排`、`权限过滤`、`工具调用`、`记忆`、`状态机` 等步骤，就把它们插到主链路中间，再补对应说明。

## 2. 每个例子必须使用的解释模板

每个可运行示例的主 `README.md` 都必须按下面的顺序解释。不能只放代码、命令或一张抽象 Mermaid 图。

### 2.1 示例目标

- 用一句话说明这个例子解决什么问题。
- 给出一个现实业务场景，并说明不使用这个方案时的困难。

### 2.2 最小输入示例

- 展示一条可以直接运行的命令、HTTP 请求或函数调用。
- 输入依赖文件时，同时展示最小目录树和一小段文件内容。
- 至少解释一个正常输入；存在分支、失败或安全边界时，再增加一个对照输入。

### 2.3 处理前的数据

- 展示进入流程前最重要的数据形态，例如文档目录、JSON、State、工具参数或消息列表。
- 只保留理解流程所需字段，不堆砌无关内容。

### 2.4 整体流程图（纯文本展开版）

- 主流程图必须使用 `text` 代码块和树状连接符，保证 VS Code、GitHub 和普通 Markdown 阅读器都能显示。
- 从真实入口开始，展开到最终输出；不能省略检索、校验、路由、状态更新、工具执行、上下文组装等关键中间步骤。
- 每个关键节点优先标出对应的函数、类、文件或框架。
- 有分支和循环时必须画出分支条件、回流位置和终止条件。
- Mermaid 或 SVG 可以保留作辅助图，但不能代替纯文本主流程图。

统一样式：

```text
用户输入 / 文件 / HTTP 请求
│
▼
入口：main() / FastAPI route / graph.invoke()
│
▼
参数或数据校验
├── 失败 -> 错误 / 拒绝 / fallback
└── 成功
    │
    ▼
核心处理函数
│
▼
模型 / 工具 / 检索器 / 状态图
│
▼
结果整理与来源说明
│
▼
最终输出
```

### 2.5 输出与逐节点解释

- 展示与最小输入对应的最小输出，必要时注明输出为示意值。
- 紧跟“流程节点 -> 文件/函数/类 -> 输入 -> 输出 -> 为什么需要”的对应表；节点很少的最小示例可以改用一段 `节点对应：` 说明，但必须点出真实函数或组件。
- 说明输出中的哪些部分是确定字段、哪些部分会随输入、模型或数据变化。

### 2.6 下一步理解与企业级演进（强制）

解释完当前例子后，必须继续写“下一步理解”以及“企业级 / 现实案例中的完整流程”。这一节不是普通的学习建议，不能省略，也不能只列几个框架名称。

必须同时完成下面四件事：

1. 用一句等式或映射说明当前教学实现的本质，例如 `retrieve() = 关键词匹配`、`list[Chunk] = 内存索引`。
2. 说明真实项目会把当前哪一个函数、数据结构或本地组件替换成什么生产组件。
3. 使用 `text` 代码块画出企业级流程图，从真实数据源一直展开到输出、评估和运维环节。
4. 给出一个现实业务案例，解释数据从哪里来、谁发起请求、经过哪些组件、失败时如何处理，以及最终交付给谁。

统一写法：

```text
当前教学例子
核心函数 / 本地数据结构
=
它所模拟的真实能力

↓ 下一步替换或扩展

真实数据源
│
▼
生产级处理组件
│
▼
检索 / 模型 / 工具 / 工作流
├── 成功 -> 业务输出
└── 失败 -> 重试 / fallback / 人工处理
    │
    ▼
评估 + 监控 + 审计 + 持续改进
```

企业级流程图必须根据例子类型继续展开：

- `RAG`：数据接入、清洗、Chunk、Embedding、向量库、关键词索引、Metadata/ACL、Hybrid Search、Rerank、Top-K、Context、LLM、引用、评估、监控和索引更新。
- `Tool Calling / MCP`：身份认证、工具发现、Schema、权限策略、参数校验、执行、超时重试、人工审批、结果回填和审计。
- `Workflow / LangGraph`：State、Node、条件路由、循环终止、Checkpoint、Interrupt、持久化、失败恢复和人工介入。
- `结构化输出`：Schema 版本、模型生成、校验、修复/重试、业务规则、持久化、下游 API 和质量监控。
- `模型调用 / LangChain`：Prompt 管理、模型网关、Provider 回退、限流、缓存、Tracing、成本统计和内容安全。
- `API / Streaming / Frontend`：认证、请求校验、任务状态、流式协议、断线恢复、错误合同、可观测性和前端状态管理。
- `Multi-Agent`：Supervisor、角色边界、共享状态、消息协议、预算、终止保护、质量门禁和人工审批。

RAG 类示例至少要给出下面这种从教学实现到真实系统的认知桥梁：

```text
当前例子
retrieve()
=
关键词匹配

↓ 企业级 RAG

文档 / Wiki / 数据库 / 工单
│
▼
清洗与 Chunk
│
▼
Embedding + 关键词索引
│
▼
向量数据库（FAISS / Chroma / Qdrant 等）
│
▼
Metadata / ACL 过滤
│
▼
Hybrid Search -> Rerank -> Top-K
│
▼
Context 组装 -> LLM -> 答案与引用
│
▼
检索评估 + 答案评估 + 监控 + 索引更新
```

不能只写“下一步学习 LangChain / LangGraph”。必须解释学习它们以后，当前例子中的哪一部分会被替换、增强或纳入更完整的工程链路。

## 3. RAG 类示例的固定展开顺序

对于 `doc_qa_agent`、`rag_api_demo`、`advanced_rag` 这类例子，建议统一用下面的顺序写：

1. 资料目录
2. 读取与切分
3. chunk 集合
4. 用户问题
5. 检索与打分
6. Top-K 选择
7. 上下文拼接
8. 模型生成
9. 答案与来源输出

## 4. 写法要求

- README 先让人看懂全链路，再看代码。
- 流程图要能单独解释每一步，不要只留下“输入 -> 输出”这种过于粗的图。
- 如果同一主题有多个示例，尽量复用同一套节点命名，方便横向对比。
- 图里的每个节点最好都能在正文里找到对应的函数名或模块名。
- 同一示例存在 `README.md` 和 `README_LEARN.md` 时，完整模板放在主 `README.md`；学习补充文档通过链接引用，不复制一套容易失效的流程。
- 每个主 `README.md` 的“图片式模板解释”后面必须出现“下一步理解与企业级演进”，两节共同构成一个完整示例说明。
- 企业级流程不能为了显得复杂而堆组件；每个新增组件都要对应当前例子的真实限制、失败模式或业务需求。

## 5. Mermaid 显示兼容规则

- 辅助 Mermaid 流程图统一使用 ```` ```mermaid ```` 代码块，不使用大小写变体或自定义围栏。
- VS Code `1.121` 及以上版本使用内置 Mermaid 渲染器，不安装或推荐已废弃的 `bierner.markdown-mermaid`，避免两个渲染器同时激活后出现空白图表。
- 含函数调用、路径、括号、斜杠、井号或 HTML 换行的节点文本统一放在双引号中，例如 `A["build_context()<br/>拼接上下文"]`。
- 新增或修改流程图后，至少在 VS Code Markdown 预览中检查一次；如果只显示空白画布，先检查是否存在重复 Mermaid 扩展，再检查图表语法。
