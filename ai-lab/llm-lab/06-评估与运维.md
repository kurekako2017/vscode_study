# 评估与运维

这一章解决的问题是：demo 能跑以后，怎么判断它是否可靠、可解释、可继续改进。

```text
问题集 -> 执行问答 -> 记录命中/回答/来源 -> 总结失败类型 -> 改进方案
```

## 1. 学完要会什么

- 知道“能跑”不等于“可交付”
- 能准备 10 到 20 个测试问题
- 能区分检索命中、回答正确、来源正确
- 能记录失败案例和改进点
- 能说明成本、延迟、日志、错误处理这些运维问题
- 能维护一份 [rag_eval_notes.md](./rag_eval_notes.md)

## 2. RAG 最小评估维度

| 维度 | 要看什么 |
| --- | --- |
| 检索命中 | 是否找到相关文档和正确片段 |
| 回答质量 | 是否基于资料回答，是否有幻觉 |
| 来源质量 | 返回来源是否能支撑答案 |
| 无命中处理 | 没资料时是否明确说不知道 |
| 稳定性 | 同类问题是否表现一致 |
| 运维性 | 日志、错误、成本、延迟是否可观察 |

## 5. 推荐评估流程

1. 先选一个 demo：`doc_qa_agent` 或 `rag_api_demo`。
2. 准备问题集，至少包含正常命中、模糊问题、无命中问题。
3. 写下每个问题理论上应该命中的资料。
4. 执行问答并记录答案、来源、失败点。
5. 把观察整理进 [rag_eval_notes.md](./rag_eval_notes.md)。
6. 根据失败类型决定下一步改检索、改 chunk、改 prompt，还是改 API。

## 6. 常见误区

| 误区 | 为什么危险 |
| --- | --- |
| 只看最终回答 | 不知道答案是否基于正确资料 |
| 只测容易问题 | 看不出系统边界 |
| 不测无命中 | 系统可能在没资料时乱答 |
| 不记录失败 | 下次无法比较是否改进 |
| 不看成本和延迟 | PoC 很难走向真实使用 |

## 7. 运维最小意识

学习阶段先记住这些问题：

- 出错时，调用方能看到什么？
- 服务日志能不能定位问题？
- 文档更新后怎么重载？
- 调用次数和成本能不能控制？
- API Key 和配置放在哪里？
- 回答错了以后，如何复现？

这些问题不一定马上全部实现，但面试和现场沟通时要能说明。

## 8. 练习任务

1. 给 `doc_qa_agent` 准备 10 个测试问题。
2. 给每个问题写“期望资料”。
3. 运行并记录是否命中、是否答对、来源是否合理。
4. 总结 3 类失败原因。
5. 把结果补充到 [rag_eval_notes.md](./rag_eval_notes.md)。
