## 2. 目的

本書は、機能要件以外の品質観点を整理し、性能、可用性、保守性、セキュリティ、運用性の要求水準を明確化することを目的とする。

## 3. 非機能要件一覧

| 区分 | 要件ID | 要件内容 | 現状 | 今後の対応方針 |
|---|---|---|---|---|
| 性能 | NFR-01 | 一般的な画面遷移は数秒以内に応答すること | 小規模データ前提では問題小 | 大量データ時はページング追加を検討 |
| 性能 | NFR-02 | 商品一覧・顧客一覧取得時に過度な SQL 発行を避けること | DAO 単位取得中心 | SQL 実行回数計測を追加 |
| 性能 | NFR-11 | 主要画面は性能試験条件一覧に基づき応答時間を測定できること | 未整備 | `82_性能テスト計画書` と CSV 管理表を整備 |
| 性能 | NFR-12 | 同時アクセス時でも許容エラー率内で処理継続できること | 未測定 | 一覧系・更新系を対象に負荷試験を実施 |
| 可用性 | NFR-03 | アプリ起動・停止手順が標準化されていること | 手順書あり | スクリプト化と監視連携を追加 |
| 可用性 | NFR-04 | 障害発生時に一次切り分け可能なこと | ログと障害対応手順あり | ログ出力粒度を強化 |
| セキュリティ | NFR-05 | ログイン機能で認証・権限分離を行うこと | 一般/管理者の分離あり | パスワード暗号化強化を検討 |
| セキュリティ | NFR-06 | 直接 URL 叩きによる管理画面流入を抑止すること | 一部 Session 前提 | Spring Security 導入を検討 |
| 保守性 | NFR-07 | Controller / Service / DAO / Model の責務分離が維持されること | 分層構造あり | 共通部品化と規約強化を継続 |
| 保守性 | NFR-08 | 設計書を元にコード追跡できること | 文書整備済み | 変更時の同期更新を徹底 |
| 運用性 | NFR-09 | 初期データ投入、試験前準備、リリース手順が明文化されていること | 手順書あり | 実施結果記録を継続 |
| 拡張性 | NFR-10 | 外部 API、決済、通知などの追加接続に耐えられること | 基本構造のみ整備 | 外部 IF 設計を別途追加 |

## 4. 区分別詳細

### 4.1 性能

- 商品一覧、分類一覧、顧客一覧の取得は、画面表示に必要な範囲で応答すること。
- 不要な全件取得、不要な多重クエリを避けること。
- 将来データ量増加時は、ページング、検索条件、索引追加を前提とする。
- 性能確認は `82_性能テスト計画書` と `82_性能テスト条件一覧.csv` を用いて管理すること。
- 日本案件では JMeter 等で実行し、Excel 管理表で条件、実績、判定を管理する運用を想定する。

### 4.2 可用性

- 起動手順、停止手順、障害初動手順を文書化すること。
- DB 接続不可、セッション不整合、アプリ起動不可時の切り分け観点を持つこと。
- リリース時にはロールバック可能な手順を持つこと。

### 4.3 セキュリティ

- 利用者区分は `ROLE_NORMAL` と `ROLE_ADMIN` を前提とする。
- 管理者向け機能は一般利用者から分離すること。
- 入力値チェック、認証失敗時のメッセージ、権限外アクセス時の制御方針を明確化すること。

### 4.4 保守性

- 文書番号、画面、URL、クラス、テーブルの追跡可能性を維持すること。
- 変更時は `変更履歴一覧`、`版数管理表`、関連設計書を更新すること。
- 共通化可能な処理は `共通部品一覧` に追記し、重複実装を減らすこと。

### 4.5 運用性

- 運用担当者がアプリ起動、ログ確認、障害一次対応を単独で実施できること。
- FAQ、引継ぎ資料、監視手順書を合わせて参照できる状態とすること。

## 5. 優先度評価

| 優先度 | 対象 | 理由 |
|---|---|---|
| 高 | セキュリティ、可用性 | ログイン、管理機能、障害対応に直結するため |
| 中 | 保守性、運用性 | 長期保守、引継ぎ品質に影響するため |
| 中 | 性能 | 現状小規模だが将来拡張で影響が出るため |
| 低 | 拡張性 | 現時点では外部連携が限定的なため |
