## 2.1 処理フロー図の記載方針

本書に記載する `処理フロー図` は、業務処理単位または画面機能単位の全体フローを対象とする。

対象例:

- ログイン処理
- カート追加処理
- 管理者商品更新処理

記載対象:

- 画面入力から画面返却までの主要分岐
- Service / DAO / DB 更新順序
- 正常系と代表的な異常系

記載対象外:

- 1 クラス内部だけの細かいメソッド処理
- 単一 DAO メソッド内部の詳細分岐

補足:

- 機能単位の観点は [13_機能設計書.md](D:/dev/source_code/vscode_study/java-projects/JtProject/doc/jp-docs/01_design/13_%E6%A9%9F%E8%83%BD%E8%A8%AD%E8%A8%88%E6%9B%B8.md) に記載する。
- クラス、メソッド単位の処理フローは [15_クラス詳細設計書.md](D:/dev/source_code/vscode_study/java-projects/JtProject/doc/jp-docs/02_class-design/15_%E3%82%AF%E3%83%A9%E3%82%B9%E8%A9%B3%E7%B4%B0%E8%A8%AD%E8%A8%88%E6%9B%B8.md) 配下に記載する。

## 3. 処理詳細

### 3.1 ユーザーログイン処理

対象メソッド：

- `UserController.userlogin()`
- `UserService.checkLogin()`
- `UserDao` ログイン照合処理

処理概要：

1. 画面から `username`, `password` を受信
2. `UserService.checkLogin()` 呼出
3. DB のユーザー情報照合
4. 成功時:
- Cookie `username` を保存
- Session に `username`, `userRole` を保存
- 商品一覧を取得し `index.jsp` を表示
5. 失敗時:
- `userLogin.jsp` にエラーメッセージ表示

入力:

| 項目 | 型 | 必須 | 説明 |
|---|---|---|---|
| username | String | 必須 | ログインユーザー名 |
| password | String | 必須 | パスワード |

出力:

| 条件 | 画面 | メッセージ |
|---|---|---|
| 成功 | `index.jsp` | なし |
| 認証失敗 | `userLogin.jsp` | `Please enter correct email and password` |
| 例外 | `userLogin.jsp` | `系统异常` 文言 |

処理手順詳細：

1. `UserController.userlogin()` が POST パラメータを受信する
2. Service 層へ認証依頼を行う
3. `UserDao.getUser()` は `username` で DB 検索を行う
4. DB 取得結果が空の場合は空 `User` を返却する
5. 取得結果がある場合は Java 側でパスワード文字列比較を行う
6. 認証成功時は Cookie / Session を設定する
7. 商品一覧を取得して `index.jsp` を返す
8. 認証失敗時は `userLogin.jsp` にエラーメッセージを設定する

SQL / HQL イメージ：

- HQL: `from User where username = :username`
- 認証判定: DB 取得後に `storedPassword.equals(inputPassword)` を実施

### 3.2 カート追加処理

対象メソッド：

- `UserController.addToCart()`

処理概要：

1. Cookie または Session からログインユーザー取得
2. ユーザー未取得の場合はメッセージセットして商品一覧へ戻る
3. ロールが `ROLE_ADMIN` の場合は管理画面へ誘導
4. ユーザーに紐づく Cart を検索
5. Cart がない場合は新規作成
6. Product を取得
7. `CartProduct` を追加
8. カート画面へリダイレクト

処理手順詳細：

1. `resolveLoggedInUsername()` により Session / Cookie からユーザー名取得
2. ユーザー未取得時は `cartMsg` に失敗理由を設定
3. `UserService.getUserByUsername()` で顧客情報を取得
4. 全カート一覧から該当顧客のカートを探索
5. カートが存在しなければ `CartService.addCart()` で新規作成
6. `ProductService.getProduct(id)` で商品を取得
7. `CartProductDao.addCartProduct()` で中間テーブル登録
8. Session メッセージを設定し `/user/cart` へ遷移

DB アクセス観点：

- `CART` から顧客別カート取得
- `PRODUCT` から対象商品取得
- `CART_PRODUCT` へ関連登録

### 3.3 管理者商品更新処理

対象メソッド：

- `AdminController.updateproduct()`
- `AdminController.updateProduct()`

処理概要：

1. 画面表示時に商品詳細とカテゴリ一覧取得
2. 更新時に入力値から Product を再構成
3. 画像未指定時は既存画像を引継ぐ
4. `productService.updateProduct()` 実行
5. 商品一覧画面へ遷移

処理手順詳細：

1. GET で対象商品を取得し、カテゴリ一覧と合わせて画面表示
2. POST で更新値を受信する
3. `categoryId` によるカテゴリ取得を行う
4. `existingProduct` を取得し、画像未入力時の引継ぎ判定を行う
5. 更新用 `Product` を生成し、項目を再設定する
6. `ProductService.updateProduct()` で更新
7. `/admin/products` にリダイレクト

## 4. 画面項目定義例

### SCR-01 ユーザーログイン画面

| 項目ID | 項目名 | 物理名 | 型 | 必須 | 備考 |
|---|---|---|---|---|---|
| ULG-01 | ユーザー名 | username | text | 必須 | `@RequestParam("username")` |
| ULG-02 | パスワード | password | password | 必須 | `@RequestParam("password")` |
| ULG-03 | ログインボタン | submit | button | - | `/userloginvalidate` POST |
| ULG-04 | メッセージ | msg | label | - | 失敗時表示 |

### SCR-10 商品追加画面

| 項目ID | 項目名 | 物理名 | 必須 | 備考 |
|---|---|---|---|---|
| PAD-01 | 商品名 | name | 必須 | 文字列 |
| PAD-02 | カテゴリID | categoryid | 必須 | 選択式 |
| PAD-03 | 価格 | price | 必須 | 数値 |
| PAD-04 | 重量 | weight | 必須 | 数値 |
| PAD-05 | 在庫数 | quantity | 必須 | 数値 |
| PAD-06 | 説明 | description | 任意 | テキスト |
| PAD-07 | 画像 | productImage | 任意 | 画像パス |

## 5. 注意事項

- 一般ユーザーと管理者でログイン管理方式が異なる
- 一部メッセージが英語、中国語混在のため、今后日文化余地あり
- 管理者プロフィール更新は実装コメントと実処理の乖離がないか継続確認が必要
- パスワード比較は暗号化ではなく平文比較である
- 一部入力チェックは画面側 `required` に依存している
