## 2. 目的

本書は、`BAT-004 アプリケーションログローテーション` の処理詳細、ファイル運用方針、異常時制御、保守観点を定義する。

目的は以下の通り。
- ログ肥大化防止
- 障害解析用ログの圧縮保管
- 実行後も現行ログを継続利用可能な状態へ戻す

## 3. 概要

### 3.1 バッチ概要

| 項目 | 内容 |
|---|---|
| バッチ ID | `BAT-004` |
| バッチ名 | アプリケーションログローテーション |
| 起動クラス | `AppLogRotationBatchApplication` |
| 実行クラス | `AppLogRotationBatchService` |
| 実行方式 | Spring Boot 非 Web 起動 |
| プロファイル | `batch` |
| 入力 | `logs/batch/*.log` |
| 出力 | `*.log.{timestamp}.zip`、空の現行 `.log` |
| 終了コード | `0`: 正常、`1`: システム異常 |

### 3.2 処理概要

```
開始
  ↓
ログディレクトリ存在チェック
  ↓
対象 .log ファイル探索（サイズ > 0）
  ↓
ファイル単位でローテーション
  ├─ zip 作成
  ├─ 元 log を zip へ格納
  └─ 元 log を truncate
  ↓
件数ログ出力
  ↓
終了コード返却
```

## 4. 処理詳細

### 4.1 対象抽出

| 条件 | 内容 |
|---|---|
| 拡張子 | `.log` |
| ファイル種別 | 通常ファイル |
| サイズ条件 | `> 0 byte` |

### 4.2 ローテーション処理

#### 4.2.1 命名規則

- zip ファイル名: `{logName}.{yyyyMMdd_HHmmss}.zip`
- 例: `product-inventory-check.log.20260430_220001.zip`

#### 4.2.2 実装方針

```java
ZipOutputStream zos = new ZipOutputStream(...);
zos.putNextEntry(new ZipEntry(logFile.getFileName().toString()));
in.transferTo(zos);
Files.newBufferedWriter(logFile, TRUNCATE_EXISTING).close();
```

#### 4.2.3 再実行時挙動

- タイムスタンプを含むためファイル名衝突は基本的に回避
- 空ログは再圧縮対象外

### 4.3 0件時の扱い

- 対象ディレクトリ不存在: WARN ログ後、正常終了
- 対象ファイル 0 件: INFO ログ後、正常終了

## 5. エラーハンドリング戦略

| 例外 | 例 | 終了コード |
|---|---|---|
| `IOException` | zip 作成失敗、ファイル書込失敗 | `1` |
| その他例外 | パス不正、権限問題 | `1` |

- 1 ファイルでも失敗した場合はバッチ全体を異常終了とする。

## 6. パフォーマンス・運用

### 6.1 パフォーマンス方針

- 対象はログファイル I/O が支配的
- ファイルごとに逐次処理し、メモリ消費を一定化

### 6.2 運用注意点

- 実行時にログ書込中のファイルを扱うため、運用上は低トラフィック帯で実行
- ローテーション先ディスク容量を監視

## 7. 出力仕様

### 7.1 生成物

| 種別 | 内容 |
|---|---|
| 圧縮ログ | `logs/batch/*.zip` |
| 現行ログ | 元 `.log`（空ファイル） |

### 7.2 終了コード

| コード | 意味 |
|---|---|
| 0 | 正常終了 |
| 1 | システム異常 |

## 8. テスト戦略

### 8.1 単体テスト観点

- `.log` 検索条件の確認
- zip 生成後に元ログが truncate されること
- ディレクトリ不存在時に `0` を返すこと

### 8.2 統合テスト観点

- 複数ログファイルでローテーション件数が一致する
- 生成 zip を展開し、元ログ内容と一致する
