## 3. 概要

### 3.1 バッチ概要

| 項目 | 内容 |
|---|---|
| バッチ ID | `BAT-005` |
| バッチ名 | 商品·分類マスタ整合チェック |
| 処理区分 | 監視処理（日次チェック） |
| 実行契機 | 日次固定時刻（運用想定） |
| 処理対象 | `PRODUCT` テーブル、`CATEGORY` テーブル |
| 出力形式 | CSV + ログ |

### 3.2 処理概要

```
入力（PRODUCT・CATEGORY マスタ）
      ↓
   [フェーズ 1] カテゴリ参照整合チェック
      ├─ 存在チェック（CATEGORY_ID が CATEGORY テーブルに存在するか）
      ├─ デリート前チェック（削除済みカテゴリ参照でないか）
      └─ NULL チェック
      ↓
   [フェーズ 2] 商品データ有効性チェック
      ├─ 必須項目チェック（商品名、価格）
      ├─ 形式チェック（価格は数値か、在庫数 ≥ 0 か）
      └─ 論理チェック（商品名の長さ制限）
      ↓
   [フェーズ 3] マスタ連動整合性チェック
      ├─ カテゴリの利用可否確認
      └─ 商品の販売可能状態確認
      ↓
   結果 CSV 出力 + ログ出力 + 終了コード返却
```

## 4. 処理詳細

### 4.1 フェーズ 1：カテゴリ参照整合チェック

#### 4.1.1 処理内容

各 `PRODUCT.CATEGORY_ID` に対して以下をチェック：

| チェック項目 | 条件 | NG判定 | エラーコード |
|---|---|---|---|
| 存在チェック | `CATEGORY.ID = PRODUCT.CATEGORY_ID` | 参照先 CATEGORY が存在しない | `ERR_CAT_NOT_FOUND` |
| DELETE フラグ | `CATEGORY.DELETE_FLG = 0` | DELETE_FLG = 1（削除済み） | `ERR_CAT_DELETED` |
| NULL チェック | `PRODUCT.CATEGORY_ID IS NOT NULL` | NULL | `ERR_CAT_NULL` |

#### 4.1.2 SQL（最適化版）

```sql
-- フェーズ 1: 参照整合チェック用の一括 LEFT OUTER JOIN
SELECT 
  p.ID,
  p.NAME,
  p.CATEGORY_ID,
  c.ID as CAT_ID,
  c.DELETE_FLG
FROM PRODUCT p
LEFT OUTER JOIN CATEGORY c 
  ON p.CATEGORY_ID = c.ID
WHERE p.DELETE_FLG = 0
ORDER BY p.ID;
```

**最適化ポイント:**
- 単一クエリで全チェック対象を取得（N+1 クエリ回避）
- LEFT OUTER JOIN で存在チェックと削除フラグを同時確認
- 結果をメモリ内キャッシュ、Java でチェック実施

#### 4.1.3 エラー時の処理フロー

```
チェック結果 = NG
  ├─ メッセージ文字列化: "ERR_CAT_NOT_FOUND | ERR_CAT_DELETED"
  ├─ ログに WARN 出力: 商品ID、エラーコード、カテゴリID
  └─ CSV 行に "NG" フラグと複数メッセージを記録
```

### 4.2 フェーズ 2：商品データ有効性チェック

#### 4.2.1 チェック項目

| 項目 | 条件 | NG判定 | エラーコード | 補足 |
|---|---|---|---|---|
| 商品名 | `TRIM(NAME) <> ''` | 空 or 空白のみ | `ERR_PROD_NAME_EMPTY` | 長さ MAX 100 文字 |
| 価格 | `PRICE > 0 AND PRICE <= 999999.99` | 価格 ≤ 0 or 超過 | `ERR_PRICE_INVALID` | DECIMAL(10,2) |
| 在庫数 | `QUANTITY >= 0 AND QUANTITY <= 999999` | 負数 or 超過 | `ERR_QUANTITY_INVALID` | INTEGER |
| 商品説明 | 形式チェック不要 | - | - | 任意項目 |

#### 4.2.2 実装例（Java）

```java
List<String> errors = new ArrayList<>();

// 商品名チェック
if (!InputCheckUtil.hasText(product.getName())) {
    errors.add("ERR_PROD_NAME_EMPTY");
} else if (product.getName().length() > 100) {
    errors.add("ERR_PROD_NAME_TOO_LONG");
}

// 価格チェック
if (product.getPrice() == null || product.getPrice() <= 0 
    || product.getPrice() > 999999.99) {
    errors.add("ERR_PRICE_INVALID");
}

// 在庫数チェック
if (product.getQuantity() < 0 || product.getQuantity() > 999999) {
    errors.add("ERR_QUANTITY_INVALID");
}
```

### 4.3 フェーズ 3：マスタ連動整合性チェック

#### 4.3.1 カテゴリの利用可否確認

| 条件 | 判定 | エラーコード |
|---|---|---|
| `CATEGORY.DELETE_FLG = 0` | ✅ 利用可 | - |
| `CATEGORY.DELETE_FLG = 1` | ❌ 削除済み | `ERR_CAT_DELETED` |

**実装:**
- フェーズ 1 の SQL 結果を再利用
- メモリ内マップ(`categoryMap`) で DELETE_FLG をキャッシュ

#### 4.3.2 商品の販売可能状態確認

```java
// 販売可否判定ロジック
boolean isSellable = category != null 
    && category.getDeleteFlg() == 0
    && product.getQuantity() > 0
    && product.getPrice() > 0;
```

**エラーケース:**
- 参照カテゴリが削除済み → `ERR_CAT_DELETED`
- 在庫が 0 → `ERR_OUT_OF_STOCK`（ ⚠️ 警告レベル、NG ではない）
- 価格 ≤ 0 → `ERR_PRICE_INVALID`

## 5. エラーハンドリング戦略

### 5.1 エラー分類

#### 業務 NG（終了コード = 2）

整合性問題だが処理続行可能：
- カテゴリ参照不整合
- 必須項目欠損
- 商品価格·在庫の論理エラー

#### システム異常（終了コード = 1）

処理継続不可能：
- DB 接続エラー
- SQL 実行エラー
- ファイル出力エラー
- 予期しない RuntimeException

#### 正常終了（終了コード = 0）

- NG 件数 = 0

### 5.2 トランザクション方針

```java
@Transactional(readOnly = true)
public int runBatch() {
    // 読み取り専用トランザクション
    // ロック不要、他プロセスの妨害なし
}
```

**理由:**
- マスタデータのみ読み取り
- 結果は CSV ファイルに出力（DB 更新なし）
- 並行実行による競合なし

### 5.3 例外ハンドリングパターン

```java
try {
    // フェーズ 1-3 の処理
} catch (DataAccessException e) {
    logger.error("DB アクセスエラー", e);
    return 1;  // システム異常
} catch (IOException e) {
    logger.error("ファイル出力エラー", e);
    return 1;  // システム異常
} catch (Exception e) {
    logger.error("予期しないエラー", e);
    return 1;  // システム異常
}
```

## 6. パフォーマンス·メモリ最適化

### 6.1 データ量別処理戦略

| データ量 | 戦略 | 理由 |
|---|---|---|
| ～ 10,000 件 | 全件メモリロード | 処理速度重視 |
| 10,001 ～ 100,000 件 | ストリーム + バッファ処理 | メモリ効率化 |
| 100,001 件～ | ページング + 定期フラッシュ | OOM 回避 |

### 6.2 メモリ効率化の具体例

```java
// ❌ 非効率: 全件メモリロード
List<Product> all = productDao.getAllProducts();  // メモリ爆増

// ✅ 効率的: バッチ処理
List<String> csvLines = new ArrayList<>();
int batchSize = 1000;
for (int offset = 0; ; offset += batchSize) {
    List<Product> batch = productDao.getProducts(offset, batchSize);
    if (batch.isEmpty()) break;
    
    processBatch(batch, csvLines);
    csvLines.clear();  // 定期的にメモリ解放
}
```

### 6.3 SQL 最適化チェックリスト

- ✅ INDEX: `PRODUCT.CATEGORY_ID`, `CATEGORY.ID`, `PRODUCT.DELETE_FLG`
- ✅ JOIN の LEFT OUTER JOIN 活用（参照有無判定）
- ✅ WHERE 句で削除フラグ事前フィルタリング
- ✅ ORDER BY で ソート効率化
