## 2. 目的

本書は、設計、実装、試験、運用準備の各工程で発生した課題を一覧管理し、対応漏れを防止するための管理表である。

## 3. 管理方針

- 障害票とは別に、仕様未確定、設計見直し、運用未整備などの課題を管理する。
- 課題は発生日、影響、優先度、担当、期限、対応状況を明記する。
- 高優先度課題は設計書、試験書、承認資料へ反映有無を確認する。

|---|---|---|---|---|---|---|---|---|---|
| ISSUE-001 | 2026-03-15 | 設計 | `PRODUCT.customer_id` の業務上の意味が不明確 | DB 設計、DDL、DAO | 高 | Codex | 未設定 | 対応中 | エンティティ意図を整理し、必要なら関連資料を修正する |

## 5. 優先度定義

| 優先度 | 定義 |
|---|---|
| 高 | 品質、セキュリティ、主要機能に大きく影響するため優先対応が必要 |
| 中 | 直ちに致命ではないが、設計・試験・運用に影響するため計画対応が必要 |
| 低 | 影響は限定的で、後続工程または改善対応でよい |

## 6. 状況定義

| 状況 | 意味 |
|---|---|
| 未着手 | 課題登録のみで対応未開始 |
| 対応中 | 調査、設計、改修、反映のいずれかを実施中 |
| 管理中 | すぐには対応しないが、継続監視対象 |
| 完了 | 課題対応完了、関連資料反映済み |
| 保留 | 外部要因または方針待ち |

## 7. 備考

- 実案件では課題起票者、会議体、決定日、エビデンス URL を追加する。
- 進捗会議用に「今週対応」「次週対応予定」列を追加してもよい。
