## 2. 目的

本書は、`JtProject` に対して実施する性能試験、負荷試験、耐久試験の基本方針を整理するための資料である。  
日本案件では、性能試験ツール本体とは別に、試験条件、シナリオ、実施結果を Excel で管理することが多いため、本資料と CSV 管理表をセットで扱う。

## 3. 対象範囲

- 一般ユーザーのログイン、商品一覧表示、カート追加、カート表示
- 管理者のログイン、商品一覧表示、商品登録
- 画面応答時間、同時アクセス時の安定性、エラー率、長時間稼働時の劣化有無

対象外:

- 外部連携性能
- 大規模データ移行
- バッチ性能

## 4. 試験種別

| 種別 | 日本案件での呼び方 | 目的 | 代表確認項目 |
|---|---|---|---|
| 性能試験 | `性能試験` | 通常時の応答性能確認 | 応答時間、画面表示時間 |
| 負荷試験 | `負荷試験` | 想定同時接続での安定性確認 | 同時ユーザー数、TPS、エラー率 |
| 耐久試験 | `耐久試験` / `連続試験` | 長時間運転での劣化確認 | メモリ増加、応答悪化、接続枯渇 |
| ストレス試験 | `限界試験` / `ストレス試験` | 想定超過時の壊れ方確認 | 失敗時の挙動、復旧性 |

## 5. 想定観点

| 観点 | 内容 |
|---|---|
| 応答時間 | 主要画面が目標秒数以内に返ること |
| 同時利用 | 想定ユーザー数でもエラーなく処理継続できること |
| DB 負荷 | 一覧取得、登録、更新で異常に SQL が増えないこと |
| 長時間安定性 | 一定時間継続後も応答劣化や異常終了がないこと |
| エラー率 | HTTP エラー、例外件数が許容範囲内であること |
| 復旧性 | 試験後にアプリ、DB 接続が正常状態へ戻ること |

## 6. 試験対象シナリオ

| シナリオ ID | 業務区分 | 対象 URL / 機能 | 試験内容 | 主な測定値 |
|---|---|---|---|---|
| PERF-01 | 一般ユーザー | `GET /index` | 商品一覧表示を繰り返し実行する | 平均応答、最大応答、エラー率 |
| PERF-02 | 一般ユーザー | `POST /userloginvalidate` | ログインを繰り返し実行する | 平均応答、成功率 |
| PERF-03 | 一般ユーザー | `POST /products/addtocart` | カート追加を繰り返し実行する | 平均応答、DB 更新成功率 |
| PERF-04 | 一般ユーザー | `GET /user/cart` | カート表示を繰り返し実行する | 平均応答、取得件数整合 |
| PERF-05 | 管理者 | `POST /admin/loginvalidate` | 管理者ログインを繰り返し実行する | 平均応答、成功率 |
| PERF-06 | 管理者 | `GET /admin/products` | 商品管理一覧表示を繰り返し実行する | 平均応答、エラー率 |
| PERF-07 | 管理者 | `POST /admin/products/add` | 商品登録を一定間隔で実行する | 平均応答、登録成功率 |

## 7. 合否判定の目安

| 項目 | 目安 |
|---|---|
| 通常画面応答 | 3 秒以内を目安とする |
| 一覧系応答 | 5 秒以内を目安とする |
| エラー率 | 0% を目標、許容は 1% 未満 |
| 長時間試験 | 30 分から 1 時間の継続で大きな劣化がないこと |

注記:

- `JtProject` は学習用構成のため、本番案件の厳密 SLA ではなく、設計学習向けの目安値を記載する。
- 実案件では顧客基準値、同時接続数、ピーク時件数に合わせて再設定する。

## 8. 日本案件でよく使う性能試験ツール

| ツール / 方式 | 区分 | 用途 | 備考 |
|---|---|---|---|
| Apache JMeter | 実行ツール | HTTP、DB、API の性能試験 | 日本案件で比較的よく見かける定番 |
| Gatling | 実行ツール | シナリオベースの負荷試験 | コード管理しやすい |
| k6 | 実行ツール | スクリプトベースの負荷試験 | 軽量、CI 連携しやすい |
| LoadRunner | 商用ツール | 大規模性能試験 | 商用案件で採用されることがある |
| Excel 管理表 | 管理資料 | 条件、シナリオ、結果、証跡管理 | 実行ツールではなく管理台帳 |

補足:

- 日本案件で「Excel で性能試験」と言う場合、実際には JMeter などで試験を実行し、Excel で試験条件一覧、実施記録、結果集計、グラフ化を行うケースが多い。
- 顧客提出物としては `性能テスト計画書`、`性能テスト条件一覧`、`性能テスト結果報告書` を Excel または Markdown から転記して作ることが多い。

## 10. JtProject 向け実施方針

- まず `GET /index` と `GET /admin/products` の一覧系応答を測定する。
- 次に `POST /userloginvalidate`、`POST /products/addtocart`、`POST /admin/products/add` の更新系を測定する。
- 画面応答だけでなく、DB 更新件数や登録結果が崩れていないことを合わせて確認する。
- 学習用プロジェクトのため、試験条件は小規模から開始し、必要に応じて段階的に同時数を上げる。
