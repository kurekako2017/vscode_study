## 3. 目的

Controller、Service、DAO 単位で、各クラスの責務に対する正常系・異常系・境界値の挙動を確認する。  
日本プロジェクトの設計書ベース開発を前提とし、実装コードが詳細設計書・機能設計書どおりに動作することを検証対象とする。

## 4. 対象クラス

- `UserController`
- `AdminController`
- `UserService`
- `ProductService`
- `CartService`
- `UserDao`
- `ProductDao`
- `CartProductDao`

## 5. テスト実施方針

- Controller 単体では、リクエスト受領、入力値受け渡し、遷移先、メッセージ設定を確認する。
- Service 単体では、業務ロジック、入力判定、重複確認、更新可否判定を確認する。
- DAO 単体では、検索条件、更新対象、戻り値、件数整合を確認する。
- 異常系では、`null`、不正 ID、権限不足、対象データなし、更新失敗時のハンドリングを確認する。

## 6. テスト環境・前提条件

| 項目 | 内容 |
|---|---|
| 実行環境 | Windows 11 / Java / Spring Boot / Hibernate / JSP |
| DB | H2 またはプロジェクト既定の開発 DB |
| 事前データ | `25_テストデータ一覧.md` に記載の顧客、管理者、カテゴリ、商品データを投入済みであること |
| セッション | ログイン済みケースでは `user` または `admin` セッションを事前設定すること |
| ファイル | 商品画像更新試験では既存画像あり・画像未選択・不正ファイルの各条件を準備すること |

## 7. 観点対応方針

- 単体テストケースは [81_単体テスト観点一覧](../../../../JtProject/doc/jp-docs/04_testing/81_単体テスト観点一覧.md) の観点をもとに作成すること。
- 1 テストケースにつき 1 主観点を原則とし、必要に応じて関連観点を備考または設計根拠へ補記すること。
- 設計書に定義がある観点は、`機能設計書`、`入力チェック一覧`、`例外設計一覧`、`画面項目定義書`、`状態遷移一覧`、`入力項目・DB項目対応一覧` のいずれかを根拠として明記すること。

## 10. 実施時チェックポイント

- 期待どおりの画面名またはリダイレクト先が返却されること
- Model / Session / Request に必要な値が設定されること
- DB 更新件数、取得件数が想定どおりであること
- 例外発生時に業務メッセージへ変換され、システム異常終了しないこと
- `18_入力チェック一覧.md` の定義と整合していること
- [81_単体テスト観点一覧](../../../../JtProject/doc/jp-docs/04_testing/81_単体テスト観点一覧.md) に未対応の観点が残っていないこと
