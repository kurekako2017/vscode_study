# 57. コーディング規約

## 1. 文書情報

| 項目 | 内容 |
|---|---|
| 文書名 | コーディング規約 |
| 対象システム | JtProject |
| 文書区分 | 開発標準 |
| 版数 | v1.0 |

## 2. 目的

本書は、`JtProject` における Java / JSP / SQL / 設定ファイルの記述ルールを統一し、可読性、保守性、レビュー効率を高めることを目的とする。

## 3. 基本方針

- 既存の `MVC + Service + DAO` 構成を前提に責務を分離すること。
- 画面制御は `Controller`、業務判断は `Service`、DB 取得更新は `DAO` に限定すること。
- 命名、インデント、コメント、例外処理、ログ出力ルールを統一すること。
- 設計書記載内容と実装内容の不一致を放置しないこと。

## 4. Java コーディング規約

### 4.1 命名規則

| 対象 | 規約 |
|---|---|
| クラス名 | `PascalCase` を使用する |
| メソッド名 | `camelCase` を使用する |
| 変数名 | 意味の分かる `camelCase` を使用する |
| 定数名 | `UPPER_SNAKE_CASE` を使用する |
| DAO メソッド | `get`, `save`, `update`, `delete` を基本動詞とする |

### 4.2 Java 基本記述規則

| 対象 | 規約 |
|---|---|
| インデント | 半角スペース 4 個を基本とし、タブ文字は使用しない |
| 1 行の長さ | 必要以上に長くしない。長い式は改行して可読性を優先する |
| 波括弧 | `if`、`for`、`while`、`else` では波括弧を省略しない |
| 空行 | メソッド間、論理ブロック間に適切に空行を入れる |
| import | 未使用 import を残さない。順序は `java` / `javax` / 外部ライブラリ / 自プロジェクトを基本とする |
| 1 ファイル 1 クラス | 原則としてトップレベル public class は 1 ファイル 1 つとする |
| 末尾空白 | 行末空白は残さない |
| 改行コード | プロジェクト標準に合わせる。混在させない |

### 4.3 層ごとの責務

| 層 | 記述してよい内容 | 記述しない内容 |
|---|---|---|
| Controller | リクエスト受付、入力受領、画面遷移、Session 操作 | 複雑な業務判定、SQL |
| Service | 業務ルール、入力補正、DAO 呼出順序制御 | 画面遷移、直接 SQL |
| DAO | Entity 永続化、検索条件、更新処理 | 画面判定、権限判定 |
| Model | 項目定義、関連、業務意味表現 | 業務ロジックの実装 |

### 4.4 メソッド実装ルール

- 1 メソッド 1 目的を原則とすること。
- 入力値前提、戻り値意味、異常時動作が読み取れる実装にすること。
- 判定が多い場合は、設計書の処理フロー図と対応を取ること。
- null 前提処理を行う場合は、判定理由を明確にすること。
- ネストが深くなる場合は private メソッドへ分割すること。
- 同一処理が複数箇所に出る場合は共通化を検討すること。
- 戻り値がない場合でも、副作用が分かるメソッド名にすること。

### 4.5 コメント規約

#### 4.5.1 基本方針

- コメントは「コードだけでは伝わらない意図」を補足するために書くこと。
- コードを読めば分かる逐語説明は書かないこと。
- コメントと実装が不一致になった場合は、実装と同時にコメントも修正すること。

#### 4.5.2 JavaDoc の書き方

- `public` クラス、`public` メソッドには原則として JavaDoc を付与すること。
- JavaDoc では少なくとも「何をするか」「主要な入力」「戻り値の意味」を読み取れるようにすること。
- `@param`、`@return`、`@throws` は必要なものだけを記載し、未使用タグは残さないこと。
- 1 行目は概要文とし、文末表現を統一すること。

記載例:

```java
/**
 * ユーザー名から利用者情報を取得する。
 *
 * @param username ユーザー名
 * @return 利用者情報。存在しない場合は null
 */
public User getUserByUsername(String username) {
    ...
}
```

#### 4.5.3 行コメントの書き方

- 行コメント `//` は、複雑な判定理由、暫定対応理由、設計上の制約説明に限定すること。
- 1 行コメントは対象コードの直前に書き、行末コメントの多用は避けること。
- `TODO`、`FIXME` を使う場合は「何を」「なぜ」を書くこと。

記載例:

```java
// 管理者は通常の購入導線を使わないため、管理画面へ戻す
if (RoleConstants.isAdmin(user.getRole())) {
    return "redirect:/admin/products";
}
```

#### 4.5.4 禁止または非推奨のコメント

- `// 変数に値を代入する`
- `// if 文開始`
- `// for ループ`
- `// return する`

上記のような逐語説明コメントは記載しないこと。

#### 4.5.5 修正履歴コメントの書き方

日本案件や保守案件では、差分箇所に修正履歴コメントを残す運用がある。
本プロジェクトでも必要な場合は、以下のルールで統一する。

用途:

- 一時調査対応
- 顧客要望による局所修正
- 設計書番号や障害票番号に紐づくピンポイント改修
- 後続保守で「どこを何のために触ったか」を即座に追いたい箇所

基本ルール:

- 開始コメントと終了コメントを対で記載すること
- 日付は `YYYY/MM/DD` 形式で統一すること
- 担当者は名前または ID を記載すること
- 修正理由または経緯を 1 行で分かるように書くこと
- 広範囲改修では乱用せず、チケット・履歴管理で追える場合は最小限にすること

記載例:

```java
// --- MOD START 2026/03/20 USER_A: 顧客情報修正受付対応 ---
// 旧フォーマットコメントアウト
// String name = "Old Name";
String name = "New Name"; // 新しいコード
// --- MOD END 2026/03/20 USER_A ---
```

コメントに残す推奨項目:

- `START` / `END` の識別子
- 日付
- 担当者
- 修正内容・経緯
- 必要に応じて障害票番号、改修票番号、設計書番号

よく使う略称:

| 略称 | 意味 | 用途例 |
|---|---|---|
| `MOD` | Modified | 既存処理の修正 |
| `ADD` | Added | 処理追加 |
| `DEL` | Deleted | 削除対象の明示 |
| `FIXME` | 要修正 | 後続で修正が必要 |
| `TODO` | 要対応 | 将来対応予定 |

運用上の注意:

- 修正履歴コメントは JavaDoc の代替ではない。メソッド説明は JavaDoc、差分経緯は修正履歴コメントで使い分けること。
- 既に Git、障害票、改修票で十分追跡可能な場合は冗長に残さないこと。
- 修正完了後に不要になった暫定コメントは放置せず削除すること。
- 大量の `MOD START` / `MOD END` が並ぶ状態は可読性を落とすため、恒久対応時に整理すること。

### 4.6 Checkstyle を意識した Java 常識規約

| 観点 | 規約 |
|---|---|
| マジックナンバー | 意味のある固定値は定数化する |
| 文字列直書き | Session キー、ロール名、メッセージキーなどの繰返し文字列は定数化する |
| `final` | 再代入しないローカル変数、定数、Utility クラスには `final` を検討する |
| Utility クラス | `final class` + private constructor を基本とする |
| null 判定 | null 許容理由がある場合のみ扱い、必要な箇所で明示的に判定する |
| コレクション判定 | `null` と空を区別する必要がない場合は共通関数で統一する |
| 例外捕捉 | `catch (Exception e)` の多用は避け、可能な限り具体的な例外を捕捉する |
| ログ出力 | `System.out.println` より Logger を優先する |
| 比較 | 定数との比較は NPE 回避のため `CONSTANT.equals(value)` 形式を優先する |
| 使われないコード | 未使用 import、未使用 private メソッド、不要コメントは残さない |

### 4.7 フォーマット例

```java
public class SampleService {

    private static final String DEFAULT_ROLE = "ROLE_NORMAL";

    public User normalizeUser(User user) {
        if (user == null) {
            return null;
        }

        String username = InputCheckUtil.trimToNull(user.getUsername());
        if (!InputCheckUtil.hasText(username)) {
            return null;
        }

        user.setUsername(username);
        user.setRole(DEFAULT_ROLE);
        return user;
    }
}
```

## 5. JSP / 画面実装規約

- JSP 名は画面用途が分かる命名とすること。
- 画面に表示する文言は可能な限り統一し、`メッセージ一覧` と不整合を作らないこと。
- 入力項目名は Controller の受取パラメータと対応させること。
- 画面遷移は `画面一覧`、`画面遷移図` と整合すること。

## 6. SQL / DAO 規約

- テーブル名、項目名は定義書準拠とすること。
- 取得条件、更新対象は曖昧にせず、意図が追える形にすること。
- 全件取得メソッドは用途を限定し、将来の件数増加に注意すること。
- 削除系処理は物理削除か論理削除かを設計書と合わせること。

## 7. 例外・ログ規約

- 例外は握りつぶさず、呼出元で判断可能な形にすること。
- 障害調査に必要な情報はログで追えるようにすること。
- 画面利用者には不要な内部情報を直接表示しないこと。
- 例外設計と異なる処理を実装する場合は設計側を更新すること。

## 8. 補足事項

- Checkstyle を導入する場合は、まず本規約に合わせて命名、コメント、未使用 import、空白、波括弧の統一を優先すること。
- 自動整形ツールを使う場合でも、責務分離やコメント品質までは自動では保証されないためレビューで補完すること。
- 既存コードの改善時は、一度に全面改修せず、変更箇所周辺から段階的に規約適用すること。

## 9. レビュー観点

| 観点 | 確認内容 |
|---|---|
| 責務分離 | Controller / Service / DAO の責務逸脱がないか |
| 命名 | 変数、メソッド、クラス名が意図を表しているか |
| コメント | JavaDoc、行コメントが意図説明になっており、実装と一致しているか |
| Java 基本規約 | import、波括弧、空行、定数化、未使用コードに問題がないか |
| 設計一致 | 設計書、URL、画面遷移、テーブル定義と一致しているか |
| 例外処理 | 異常系を考慮した実装になっているか |
| 保守性 | 重複実装、共通化漏れがないか |
